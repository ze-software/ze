# RFC 5549 Implementation Summary

## Meta

| Field | Value |
|-------|-------|
| Title | Advertising IPv4 Network Layer Reachability Information with an IPv6 Next Hop |
| RFC | 5549 |
| Status | Standards Track |
| Date | May 2009 |
| Obsoleted By | RFC 8950 |
| Authors | F. Le Faucheur, E. Rosen (Cisco) |

**Purpose:** Extends MP-BGP AFI/SAFI definitions to allow IPv4 NLRI and VPN-IPv4 NLRI to be advertised with an IPv6 Next Hop address, enabling IPv4 islands to communicate across an IPv6-only backbone.

**Scope:** AFI=1 (IPv4), SAFI=1/2/4/128 with IPv6 Next Hop

**NOTE:** This RFC has been obsoleted by RFC 8950. Implementations SHOULD reference RFC 8950 for current specifications.

## Wire Formats

### Extended Next Hop Encoding Capability

Type code: 5

```
+-----------------------------------------------------+
| NLRI AFI - 1 (2 octets)                             |
+-----------------------------------------------------+
| NLRI SAFI - 1 (2 octets)                            |
+-----------------------------------------------------+
| Nexthop AFI - 1 (2 octets)                          |
+-----------------------------------------------------+
| .....                                               |
+-----------------------------------------------------+
| NLRI AFI - N (2 octets)                             |
+-----------------------------------------------------+
| NLRI SAFI - N (2 octets)                            |
+-----------------------------------------------------+
| Nexthop AFI - N (2 octets)                          |
+-----------------------------------------------------+
```

| Field | Offset | Size | Type | Constraints |
|-------|--------|------|------|-------------|
| NLRI AFI | 0 | 2 | uint16 | 1 (IPv4) only per this RFC |
| NLRI SAFI | 2 | 2 | uint16 | 1, 2, 4, or 128 |
| Nexthop AFI | 4 | 2 | uint16 | 2 (IPv6) only per this RFC |

**Length calculation:** `Capability Length = N * 6` where N is the number of AFI/SAFI/Nexthop-AFI triples.

**Parse order:** Read 6-byte tuples until Capability Length exhausted.

### MP_REACH_NLRI with IPv6 Next Hop for IPv4 NLRI

Uses standard MP_REACH_NLRI format from RFC 4760 with extended Next Hop length.

| Field | Value |
|-------|-------|
| AFI | 1 (IPv4) |
| SAFI | 1, 2, 4, or 128 |
| Length of Next Hop Address | 16 or 32 |
| Next Hop Address | IPv6 address (16 bytes), optionally followed by link-local IPv6 (additional 16 bytes) |
| NLRI | Standard IPv4 or VPN-IPv4 NLRI |

**Next Hop encoding:** Per RFC 2545 Section 3 - global IPv6 address, optionally followed by link-local address.

## Encoding Rules

- Next Hop length of 16: single IPv6 global address
- Next Hop length of 32: IPv6 global address followed by IPv6 link-local address
- For SAFI=128 (VPN-IPv4): Next Hop does NOT include Route Distinguisher (see Errata)
- All multi-byte fields in network byte order

## Decoding Rules

**Determining Next Hop protocol (Section 3):**

"The BGP speaker receiving the advertisement MUST use the Length of Next Hop Address field to determine which network-layer protocol the next hop address belongs to."

| Length | Next Hop Type |
|--------|---------------|
| 4 | IPv4 address (existing behavior) |
| 12 | VPN-IPv4: 8-byte RD + 4-byte IPv4 (SAFI=128 only) |
| 16 | IPv6 global address |
| 32 | IPv6 global + link-local addresses |

**Parse sequence:**
1. Read AFI/SAFI
2. Read Next Hop length
3. Determine Next Hop address family from length
4. Parse Next Hop according to determined family
5. Parse NLRI as per AFI/SAFI definition

## Validation

| Check | Valid | Invalid Action |
|-------|-------|----------------|
| Capability tuple size | Multiple of 6 bytes | Malformed capability |
| NLRI AFI in capability | 1 | Ignore tuple (per RFC scope) |
| NLRI SAFI in capability | 1, 2, 4, or 128 | Ignore tuple (per RFC scope) |
| Nexthop AFI in capability | 2 | Ignore tuple (per RFC scope) |
| Next Hop length for IPv6 NH | 16 or 32 | Attribute error |
| IPv6 Next Hop without capability | N/A | MUST NOT send |

## MUST Requirements

### Tx (Sender)
- "A BGP speaker MUST only advertise to a BGP peer the IPv4 or VPN-IPv4 NLRI with an IPv6 Next Hop if the BGP speaker has first ascertained via BGP Capability Advertisement that the BGP peer supports the Extended Next Hop Encoding capability for the relevant AFI/SAFI pair." (Section 4)

- "A BGP speaker that wishes to advertise to a BGP peer an IPv6 Next Hop for IPv4 NLRI or for VPN-IPv4 NLRI as per this specification MUST use the Capability Advertisement procedures defined in [RFC5492]" (Section 4)

- "The Capability Code field MUST be set to 5" (Section 4)

### Rx (Receiver)
- "The BGP speaker receiving the advertisement MUST use the Length of Next Hop Address field to determine which network-layer protocol the next hop address belongs to." (Section 3)

### Forwarding/Route Reflection
- "When a next hop address needs to be passed along unchanged (e.g., as a Route Reflector (RR) would do), its encoding MUST NOT be changed." (Section 5)

## SHOULD/MAY

- **[SHOULD]** "By default, if a particular BGP session is running over IPvx (where IPvx is IPv4 or IPv6), and if the BGP speaker sending an update is putting its own address in as the next hop, then the next hop address SHOULD be specified as an IPvx address" (Section 5) - Consequence if not: Potential routing issues if peer expects native address family

- **[MAY]** "The Extended Next Hop Encoding capability MAY be dynamically updated through the use of the Dynamic Capability capability" (Section 4) - When to use: Session wants to change supported encodings without restart

- **[MAY]** "This default behavior may be overridden by policy" (Section 5) - When to use: Operator requires specific next hop encoding

## Error Handling

| Condition | Detect How | Response | Code/Subcode |
|-----------|------------|----------|--------------|
| IPv6 NH received without capability | Check negotiated capabilities | Treat as malformed attribute | UPDATE Message Error / Optional Attribute Error |
| Invalid NH length (not 4/12/16/32) | Length field validation | Treat as malformed attribute | UPDATE Message Error / Attribute Length Error |
| Capability tuple not multiple of 6 | Length validation | Ignore capability | N/A (per RFC 5492) |

## Compatibility

**Behavior with non-supporting peers:**
- MUST NOT send IPv6 Next Hop for IPv4 NLRI to peers that have not advertised Extended Next Hop Encoding capability
- Fall back to IPv4 Next Hop or do not advertise route

**Feature negotiation:**
- Both peers must advertise Multiprotocol Extensions capability for the AFI/SAFI (RFC 4760)
- Extended Next Hop Encoding capability is additional to, not replacement for, Multiprotocol Extensions
- "The Extended Next Hop Encoding capability provides information about next hop encoding for a given AFI/SAFI, assuming that AFI/SAFI is allowed. It does not influence whether that AFI/SAFI is indeed allowed." (Section 4)

**Route Reflector considerations:**
- RR clients MUST all support same encodings if routes may be reflected between them
- "For sound routing in certain scenarios, this will require that all the RR clients be able to handle whatever encodings any of them may generate." (Section 5)

## Pitfalls

### Edge Cases
- **IPv4-mapped IPv6 addresses:** "Although not expected to be the typical case, the IPv6 address used as the BGP Next Hop Address could be an IPv4-mapped IPv6 address (as defined in [RFC4291])." (Section 8) - Security policies must account for this.

- **Route Reflector distribution:** If an RR receives a route with IPv6 NH but a client lacks the capability, the route cannot be distributed to that client.

### Interop
- Some implementations may not properly handle the capability tuple validation
- Older implementations may reject unknown capability codes entirely

### Security
- IPv4-mapped IPv6 addresses in Next Hop may bypass IPv4-specific security checks
- "Configuration of the security mechanisms potentially deployed by the network operator (such as security checks on next hop address) need to keep this case in mind also." (Section 8)

## Errata

### EID 5253 (Held for Document Update) - IMPORTANT

**Section 6.2:** The specification states "Length of Next Hop Network Address = 16 (or 32)" for VPN-IPv4 (SAFI=128), but this appears to be an error.

**Impact:** For VPN-IPv4 with standard IPv4 Next Hop, the Next Hop includes an 8-byte Route Distinguisher. The errata suggests the IPv6 case should be 24 (or 48) to maintain consistency, though the main text in Section 3 only specifies 16 or 32.

**Recommendation:** Check RFC 8950 for clarification. In practice, implementations vary:
- Some use 16/32 (literal RFC reading)
- Some use 24/48 (include RD prefix)

### EID 6786 (Rejected)

Proposed clarification on handling malformed NLRI from peers without Extended Next Hop capability. Rejected as RFC 5549 is now obsolete.
