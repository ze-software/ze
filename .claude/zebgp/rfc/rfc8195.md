# RFC 8195: Use of BGP Large Communities

## Meta

| Field | Value |
|-------|-------|
| Title | Use of BGP Large Communities |
| Number | 8195 |
| Status | Informational |
| Date | June 2017 |
| Obsoletes | None |
| Updates | None |
| Obsoleted-by | None |

**Purpose:** Provides operational examples and best practices for using BGP Large Communities, categorizing them into Informational and Action types, with a recommended ASN:Function:Parameter structure.

**Scope:** This is NOT a protocol specification. It provides operational guidance for using the Large Communities attribute defined in RFC 8092. No new wire formats or protocol mechanisms are defined.

**Errata:** None reported.

## Design Convention

RFC 8195 proposes a semantic interpretation of the three Large Community fields:

```
+----------------------+---------------+
|       RFC 8092       | this document |
+----------------------+---------------+
| Global Administrator |      ASN      |
|  Local Data Part 1   |    Function   |
|  Local Data Part 2   |   Parameter   |
+----------------------+---------------+
```

Canonical notation: `ASN:Function:Parameter`

- **ASN (Global Administrator):** The AS that defined the community's meaning
- **Function (Local Data Part 1):** Identifies what the community does
- **Parameter (Local Data Part 2):** Provides a value for the function

## Community Categories

### Informational Communities

Labels describing route attributes:
- Origin of route announcement
- Nature of EBGP neighbor relationship
- Geographic location where route was imported
- Intended propagation audience

**Global Administrator:** Set to ASN of network that tags the route.

**Audience:** Downstream networks, GA itself, any interested AS.

### Action Communities

Requests for specific route treatment within an AS:
- Propagation control (selective NO_EXPORT)
- Path manipulation (AS_PATH prepending)
- Preference adjustment (LOCAL_PREF)
- Route server distribution control

**Global Administrator:** Set to ASN expected to perform the action.

**Audience:** Transit providers, GA itself, any AS willing to act.

## Example Function Assignments

### Informational Functions

| Function | Parameter Type | Purpose |
|----------|----------------|---------|
| 1 | ISO 3166-1 numeric | Country where route was learned |
| 2 | UN M.49 region code | Geographic region where route was learned |
| 3 | Relation code | Relationship type with source peer |

**Relation codes (Function 3):**

| Parameter | Meaning |
|-----------|---------|
| 1 | Route originated internally |
| 2 | Route learned from customer |
| 3 | Route learned from peering partner |
| 4 | Route learned from transit provider |

### Action Functions

| Function | Parameter Type | Action |
|----------|----------------|--------|
| 4 | ASN | Do not export to specified ASN |
| 5 | ISO 3166-1 numeric | Do not export to EBGP neighbors in country |
| 6 | ASN | Prepend own ASN once to specified ASN |
| 7 | ISO 3166-1 numeric | Prepend own ASN once to EBGP neighbors in country |
| 8 | 0 or region | Normal customer route LOCAL_PREF |
| 9 | 0 or region | Backup customer route LOCAL_PREF |
| 10 | 0 or region | Peering route LOCAL_PREF |
| 11 | 0 or region | Upstream transit route LOCAL_PREF |
| 12 | 0 or region | Fallback route LOCAL_PREF |

**LOCAL_PREF Parameter conventions:**
- `0` = Apply globally within AS
- UN M.49 region code = Apply only in that region

### Route Server Functions

For route servers (e.g., AS 64511):

| Community | Action |
|-----------|--------|
| 64511:0:peer-as | Do not announce to peer-as |
| 64511:1:peer-as | Explicitly announce to peer-as |
| 64511:0:0 | Do not announce to any peer by default |
| 64511:1:0 | Announce to all peers by default |

Combinations enable fine-grained control:
- `64511:1:0` + `64511:0:64509` = Announce to all except AS 64509
- `64511:0:0` + `64511:1:64505` + `64511:1:64506` = Announce only to AS 64505 and AS 64506

## Multiple Communities

Routes may carry multiple Large Communities simultaneously. Example:
- `64497:1:528` (learned in Netherlands)
- `64497:2:150` (learned in Europe)
- `64497:3:2` (learned from customer)

## Security Considerations

Per Section 11 of RFC 7454 "BGP Operations and Security":

1. **Any Large Community can be carried in UPDATE** - operators must be aware
2. **Transitive across AS boundaries** - customers and their customers may use communities defined by other neighbors
3. **Scrubbing recommended** - handle communities with own ASN in GA field appropriately
4. **Document and publish** - operators should publish their community meanings

## Recommendations

**For operators:**
- [SHOULD] Publicly publish and maintain documentation on supported communities
- [SHOULD] Publish the relative order in which Action Communities are processed
- [SHOULD] Take care with LOCAL_PREF manipulation that crosses preference class boundaries (see RFC 4264 "BGP Wedgies")

## Pitfalls

### Location-Based NO_EXPORT Limitations

"[T]his might not prevent one of those EBGP neighbors from learning that route in another country and making it available in the country specified by the BGP Large Community." (Section 4.1.2)

Routes filtered geographically can still reach the filtered region via alternate paths.

### LOCAL_PREF Wedgies

Manipulating LOCAL_PREF below another route class can create "BGP Wedgies" per RFC 4264:
- Unintended stable states
- Route oscillation
- Difficult-to-debug routing anomalies

## Implementation Notes

This RFC provides **conventions**, not requirements. Implementations:

1. **Wire format:** Use RFC 8092 - this RFC adds no wire format changes
2. **Function numbers:** Example assignments only - operators define their own
3. **Policy engine:** Must support matching and actions on Large Communities
4. **Configuration:** Should allow flexible function/parameter definitions
5. **Documentation:** Should generate community documentation from config

**UN M.49 (regions):**

- 9 = Oceania
- 142 = Asia
