# RFC 9136: IP Prefix Advertisement in Ethernet VPN (EVPN)

## Wire Formats

```
    +-----------------------------------+
    |    Route Type (1 octet)           |
    +-----------------------------------+

    +-----------------------------------+
```

### IP Prefix Route (Route Type 5) - IPv4

Type code: 5
Length: 34 octets

```
    +---------------------------------------+
    |      RD (8 octets)                    |
    +---------------------------------------+
    |Ethernet Segment Identifier (10 octets)|
    +---------------------------------------+
    |  Ethernet Tag ID (4 octets)           |
    +---------------------------------------+
    |  IP Prefix Length (1 octet, 0 to 32)  |
    +---------------------------------------+
    |  IP Prefix (4 octets)                 |
    +---------------------------------------+
    |  GW IP Address (4 octets)             |
    +---------------------------------------+
    |  MPLS Label (3 octets)                |
    +---------------------------------------+
```

| Field | Offset | Size | Type | Constraints |
|-------|--------|------|------|-------------|
| RD | 0 | 8 | Route Distinguisher | Unique per MAC-VRF or IP-VRF |
| ESI | 8 | 10 | Ethernet Segment ID | Non-zero if ESI overlay index, else all zeros |
| Ethernet Tag ID | 18 | 4 | uint32 | As per RFC 7432/8365 |
| IP Prefix Length | 22 | 1 | uint8 | 0-32 for IPv4 |
| IP Prefix | 23 | 4 | IPv4 address | Full 4 octets always present |
| GW IP Address | 27 | 4 | IPv4 address | Non-zero if GW IP overlay index, else all zeros |
| MPLS Label | 31 | 3 | Label (20-bit) | High-order 20 bits contain label value |

**Total length:** 34 octets

Type code: 5
Length: 58 octets

```
    +---------------------------------------+
    |      RD (8 octets)                    |
    +---------------------------------------+
    |Ethernet Segment Identifier (10 octets)|
    +---------------------------------------+
    |  Ethernet Tag ID (4 octets)           |
    +---------------------------------------+
    |  IP Prefix Length (1 octet, 0 to 128) |
    +---------------------------------------+
    |  IP Prefix (16 octets)                |
    +---------------------------------------+

    +---------------------------------------+
    |  MPLS Label (3 octets)                |
    +---------------------------------------+
```

| Field | Offset | Size | Type | Constraints |
|-------|--------|------|------|-------------|
| RD | 0 | 8 | Route Distinguisher | Unique per MAC-VRF or IP-VRF |
| ESI | 8 | 10 | Ethernet Segment ID | Non-zero if ESI overlay index, else all zeros |
| Ethernet Tag ID | 18 | 4 | uint32 | As per RFC 7432/8365 |
| IP Prefix Length | 22 | 1 | uint8 | 0-128 for IPv6 |
| IP Prefix | 23 | 16 | IPv6 address | Full 16 octets always present |
| GW IP Address | 39 | 16 | IPv6 address | Non-zero if GW IP overlay index, else all zeros |
| MPLS Label | 55 | 3 | Label (20-bit) | High-order 20 bits contain label value |

**Total length:** 58 octets

### Route Key Fields

The following fields form the BGP route key (used for route comparison):
- RD
- Ethernet Tag ID
- IP Prefix Length
- IP Prefix

The following fields are NOT part of the route key:
- ESI
- GW IP Address
- MPLS Label

## Encoding Rules

### Length Field Determination
- IPv4 prefixes: Length field MUST be 34
- IPv6 prefixes: Length field MUST be 58
- IP prefix and GW IP address MUST be from the same IP address family

### MPLS Label Field
- High-order 20 bits contain the label value (as per RFC 7432)
- Label value SHOULD be zero if recursive resolution via Overlay Index is used
- Non-zero label indicates direct forwarding to BGP next hop

### ESI Field
- MUST be non-zero 10-octet identifier if ESI is used as Overlay Index
- MUST be all bytes zero otherwise

### GW IP Address Field
- MUST encode a valid IP address if used as Overlay Index
- MUST be all bytes zero if not used as Overlay Index

### Mutual Exclusion
- ESI and GW IP MUST NOT both be non-zero simultaneously

## Decoding Rules

### Parse Sequence
1. Read Route Type (1 octet) - must be 5
2. Read Length (1 octet) - determines IPv4 (34) or IPv6 (58)
3. Parse fixed-size fields according to address family
4. Check for EVPN Router's MAC Extended Community in path attributes

### Field Boundary Determination
- Length field determines IPv4 vs IPv6 format
- All fields are fixed size within each format
- No variable-length fields within the route type specific data

### Unknown/Unsupported Value Handling
- Per RFC 7606 Section 5.4: node that doesn't recognize RT-5 will ignore it
- Nodes not supporting inter-subnet forwarding can discard by route type value

## Validation

| Check | Valid | Invalid Action |
|-------|-------|----------------|
| Length field | 34 (IPv4) or 58 (IPv6) | Treat as malformed NLRI |
| IP Prefix Length (IPv4) | 0-32 | MUST NOT be greater than 32 |
| IP Prefix Length (IPv6) | 0-128 | MUST NOT be greater than 128 |
| ESI and GW IP both non-zero | Never valid | SHOULD treat as withdraw (Section 3.2) |
| Label=0 without Overlay Index | Never valid | MUST treat as withdraw (Section 3.1) |
| ESI=0, GW IP=0, MAC=0, Label=0 | Never valid | SHOULD treat as withdraw (Section 3.2) |
| MAC in Router's MAC EC | Valid unicast MAC | MUST treat as withdraw if broadcast/multicast |

## MUST Requirements

### Tx (Sender)
- "The Length field...MUST be either 34 (if IPv4 addresses are carried) or 58 (if IPv6 addresses are carried)" (Section 3.1)
- "The IP prefix and gateway IP address MUST be from the same IP address family" (Section 3.1)
- "The Ethernet Segment Identifier MUST be a non-zero 10-octet identifier if the ESI is used as an Overlay Index...It MUST be all bytes zero otherwise" (Section 3.1)
- "The value MUST NOT be greater than 128" (Section 3.1, for IP prefix length)
- "The GW IP field MUST be all bytes zero if it is not used as an Overlay Index" (Section 3.1)

### Rx (Receiver)
- "If the received MPLS label value is zero, the route MUST contain an Overlay Index, and the ingress NVE/PE MUST perform a recursive resolution to find the egress NVE/PE" (Section 3.1)
- "If the received label is zero and the route does not contain an Overlay Index, it MUST be 'treat as withdraw'" (Section 3.1)
- "Irrespective of the recursive resolution, if there is no IGP or BGP route to the BGP next hop of an RT-5, BGP MUST NOT install the RT-5 even if the Overlay Index can be resolved" (Section 3.2)

### Validation
- "In case two or more NVEs are attached to different BDs of the same tenant, they MUST support the RT-5 for the proper inter-subnet forwarding operation of the tenant" (Section 3)
- "The encoding of a MAC address MUST be the 6-octet MAC address specified by [IEEE-802.1Q]" (Table 1 notes)
- "The route MUST be treat as withdraw in case of an invalid MAC address" (Table 1 notes)

### Errors
- Invalid field combinations: "treat as withdraw" per RFC 7606

## SHOULD/MAY

- [SHOULD] "When sending, the label value SHOULD be zero if a recursive resolution based on an Overlay Index is used" (Section 3.1) - non-zero may cause forwarding inconsistency
- [SHOULD] "A route containing a non-zero GW IP and a non-zero ESI (at the same time) SHOULD be treat as withdraw" (Section 3.2) - allows lenient implementations
- [SHOULD] "A route where ESI, GW IP, MAC, and Label are all zero at the same time SHOULD be treat as withdraw" (Section 3.2)
- [SHOULD] "the advertising NVE/PE that sets the Overlay Index SHOULD advertise an RT-2 for the MAC Overlay Index if there are receiving NVE/PEs configured to use the MAC as the Overlay Index" (Table 1 notes)
- [MAY] "An IP Prefix route MAY be sent along with an EVPN Router's MAC Extended Community" (Section 3.1)
- [OPTIONAL] "The support of a MAC Overlay Index in this model is OPTIONAL" (Table 1 notes)

## Error Handling

| Condition | Detect How | Response | Code/Subcode |
|-----------|------------|----------|--------------|
| Length != 34 and != 58 | Parse length field | Treat as malformed NLRI | Per RFC 7606 |
| ESI non-zero AND GW IP non-zero | Check both fields | SHOULD treat as withdraw | Per RFC 7606 |
| Label=0 AND no Overlay Index | Check ESI, GW IP, MAC all zero | MUST treat as withdraw | Per RFC 7606 |
| All of ESI, GW IP, MAC, Label = 0 | Check all four values | SHOULD treat as withdraw | Per RFC 7606 |
| Invalid MAC (broadcast/multicast) | Check Router's MAC EC | MUST treat as withdraw | Per RFC 7606 |

## Overlay Index Resolution

### Overlay Index Types and Priority

| ESI | GW IP | MAC* | Label | Overlay Index |
|-----|-------|------|-------|---------------|
| Non-Zero | Zero | Zero | Don't Care | ESI |
| Non-Zero | Zero | Non-Zero | Don't Care | ESI |
| Zero | Non-Zero | Zero | Don't Care | GW IP |
| Zero | Zero | Non-Zero | Zero | MAC |
| Zero | Zero | Non-Zero | Non-Zero | MAC or None** |
| Zero | Zero | Zero | Non-Zero | None*** |

*MAC = presence of EVPN Router's MAC Extended Community with valid MAC
**Local policy determines whether MAC is used as Overlay Index
***Direct forwarding to BGP next hop using label

### Recursive Resolution Rules

1. **ESI Overlay Index**: Resolve via RT-1 (Ethernet A-D per EVI route) for that ESI
2. **GW IP Overlay Index**: Resolve via RT-2 (MAC/IP Advertisement) containing that IP in IP Address field
3. **MAC Overlay Index**: Resolve via RT-2 (MAC/IP Advertisement) containing that MAC in MAC Address field
4. **None**: No recursive resolution; use BGP next hop and label directly

### Resolution Timing
- RT-1 or RT-2 routes needed for resolution may arrive before or after the RT-5
- RT-5 cannot be used until Overlay Index is resolved
- Resolution applies at IP-VRF installation time, not BGP propagation time

## Use Cases and Overlay Index Selection

| Use Case | Overlay Index | Resolution Route |
|----------|---------------|------------------|
| TS IP Address (Section 4.1) | GW IP | RT-2 for VA IP |
| Floating IP (Section 4.2) | GW IP | RT-2 for floating IP |
| Bump-in-the-Wire (Section 4.3) | ESI or MAC | RT-1 for ESI, RT-2 for MAC |
| Interface-less IP-VRF-to-IP-VRF (Section 4.4.1) | None | Direct via label |
| Interface-ful with SBD IRB (Section 4.4.2) | GW IP | RT-2 for SBD IRB IP |
| Interface-ful Unnumbered SBD (Section 4.4.3) | MAC | RT-2 for SBD IRB MAC |

## Extended Communities

### EVPN Router's MAC Extended Community (RFC 9135)
- Contains MAC address for inner destination header
- Used when:
  - Interface-less IP-VRF-to-IP-VRF with Ethernet NVO tunnels
  - MAC Overlay Index for recursive resolution
  - Bump-in-the-Wire to carry TS MAC
- SHOULD NOT be sent with IP NVO tunnels
- Ignored if ESI or GW IP is the Overlay Index

### BGP Encapsulation Extended Community (RFC 9012)
- Identifies tunnel type (VXLAN, GENEVE, MPLS, etc.)
- Sent with RT-5 when applicable

## Algorithms

### RT-5 Installation Algorithm

```
function install_rt5(route):
    # Step 1: Validate route
    if not validate_rt5(route):
        return TREAT_AS_WITHDRAW

    # Step 2: Check BGP next hop reachability
    if not has_route_to_bgp_next_hop(route.next_hop):
        return DO_NOT_INSTALL  # "MUST NOT install"

    # Step 3: Determine Overlay Index
    overlay_index = determine_overlay_index(route)

    # Step 4: Handle based on overlay index type
    if overlay_index == NONE:
        # Direct forwarding via label
        install_direct(route)
    else:
        # Recursive resolution required
        if can_resolve(overlay_index):
            install_with_resolution(route, overlay_index)
        else:
            # Cannot use until resolution available
            queue_pending_resolution(route, overlay_index)

function determine_overlay_index(route):
    if route.esi != 0:
        return ESI(route.esi)
    elif route.gw_ip != 0:
        return GW_IP(route.gw_ip)
    elif has_router_mac_ec(route):
        if route.label == 0:
            return MAC(route.router_mac)
        else:
            # Local policy decides
            return MAC_OR_NONE(route.router_mac)
    elif route.label != 0:
        return NONE
    else:
        return INVALID  # All zero - treat as withdraw

function validate_rt5(route):
    # Length check
    if route.length != 34 and route.length != 58:
        return false

    # Prefix length bounds
    max_len = 32 if route.length == 34 else 128
    if route.ip_prefix_length > max_len:
        return false

    # Mutual exclusion
    if route.esi != 0 and route.gw_ip != 0:
        return false  # SHOULD treat as withdraw

    # All-zero check
    if route.esi == 0 and route.gw_ip == 0 and
       not has_router_mac_ec(route) and route.label == 0:
        return false  # SHOULD treat as withdraw

    # Label-zero without overlay index
    if route.label == 0 and determine_overlay_index(route) == NONE:
        return false  # MUST treat as withdraw

    # MAC validation if present
    if has_router_mac_ec(route):
        mac = get_router_mac(route)
        if is_broadcast(mac) or is_multicast(mac):
            return false  # MUST treat as withdraw

    return true
```

## Pitfalls

### Edge Cases
- **Label value zero with overlay index**: Valid - resolution uses RT-1/RT-2 label
- **Non-zero label with overlay index**: Overlay index takes precedence; label may be ignored
- **Router's MAC EC with GW IP overlay index**: MAC EC is ignored (GW IP used for resolution)
- **RT-5 arrives before RT-1/RT-2**: Route pending until resolution possible
- **Same prefix from multiple NVEs**: Different Overlay Indexes may point to same or different egress NVEs

### Interop
- **Non-supporting peers**: Per RFC 7606 Section 5.4, nodes ignoring RT-5 can still share BDs with supporting nodes
- **Mixed IP-VRF-to-IP-VRF models**: Receivers may interpret MAC overlay index differently based on local policy

### Security
- **Overlay Index spoofing**: Modifying RT-1/RT-2 routes affects RT-5 resolution
- **Route filtering impact**: "any action that ends up filtering or modifying the RT-2 or RT-1 routes used to convey the Overlay Indexes will modify the resolution of the RT-5" (Section 5)

## Compatibility

### Behavior with Non-Supporting Peers
- Non-supporting NVE ignores RT-5 routes (per RFC 7606)
- Regular RFC 7432 procedures apply for shared BDs
- RT-5 required when NVEs on different tenant BDs need inter-subnet forwarding

### Feature Negotiation
- No explicit capability negotiation defined
- Support implied by sending/receiving RT-5 routes
- Route Target matching determines import/export
