# RFC 4684: Constrained Route Distribution for BGP/MPLS IP VPNs

## Meta

| Field | Value |
|-------|-------|
| RFC | 4684 |
| Title | Constrained Route Distribution for Border Gateway Protocol/MultiProtocol Label Switching (BGP/MPLS) Internet Protocol (IP) Virtual Private Networks (VPNs) |
| Status | Standards Track |
| Date | November 2006 |
| Updates | RFC 4364 |
| Obsoleted-by | None |

**Purpose:** Defines MP-BGP procedures for exchanging Route Target reachability information, enabling construction of a route distribution graph to limit VPN NLRI propagation between autonomous systems or route reflector clusters.

**Scope:** AFI=1, SAFI=132 (Route Target Membership)

## Wire Formats

### Route Target Membership NLRI Format

Advertised via MP_REACH_NLRI and MP_UNREACH_NLRI attributes.

```
+-------------------------------+
| origin as        (4 octets)   |
+-------------------------------+
| route target     (8 octets)   |
+                               +
|                               |
+-------------------------------+
```

| Field | Offset | Size | Type | Constraints |
|-------|--------|------|------|-------------|
| origin-as | 0 | 4 | uint32 | 4-byte AS number |
| route-target | 4 | 0-8 | prefix | Extended community format |

**Length calculation:**
- Total structure: 12 octets (96 bits) maximum
- NLRI encoded as prefix: 0 to 96 bits
- Prefix length 0 = default route target (receive all VPN routes)
- Minimum prefix length = 32 bits (except default)
- Encoded per RFC 2858 Section 4: 1-byte length + variable data

**Next-Hop interpretation:**
- 4 octets = IPv4 address
- 16 octets = IPv6 address

### Route Target Structure (from RFC 4360)

The route-target field follows Extended Community format:

```
+-------------------------------+
| Type (2 octets)               |
+-------------------------------+
| Value (6 octets)              |
+-------------------------------+
```

Common Route Target types:
- Type 0x0002: 2-byte AS + 4-byte Local Admin
- Type 0x0102: 4-byte IPv4 + 2-byte Local Admin
- Type 0x0202: 4-byte AS + 2-byte Local Admin

## Encoding Rules

1. **Prefix encoding:** Per RFC 2858 Section 4 - length in bits followed by minimum octets to contain that many bits
2. **Default RT:** Encoded as zero-length prefix (length byte = 0, no data bytes)
3. **Minimum non-default:** 32 bits (origin-as only, no RT prefix)
4. **AS number:** 4-byte AS encoding (network byte order)

## Decoding Rules

1. Read prefix length (bits) from NLRI
2. If length == 0: default route target
3. If length < 32 and length > 0: INVALID (origin-as cannot be partial)
4. Extract origin-as from first 4 octets (if length >= 32)
5. Remaining bits (length - 32) form route-target prefix
6. Route-target prefix can match multiple extended communities with same high-order bits

## Validation

| Check | Valid | Invalid Action |
|-------|-------|----------------|
| Prefix length | 0, or 32-96 | Treat as malformed |
| Origin-as partial | Never valid | Treat as malformed |
| Next-hop length | 4 or 16 octets | Treat as malformed |

## MUST Requirements

### Tx (Sender)

- Section 3.2(i): "When advertising RT membership NLRI to a route-reflector client, the Originator attribute shall be set to the router-id of the advertiser, and the Next-hop attribute shall be set of the local address for that session."
- Section 3.2(ii): "When advertising an RT membership NLRI to a non-client peer, if the best path as selected by the path selection procedure described in Section 9.1 of the base BGP specification [4] is a route received from a non-client peer, and if there is an alternative path to the same destination from a client, the attributes of the client path are advertised to the peer."

### Rx (Receiver)

- Section 3.2: "When processing RT membership NLRIs received from internal iBGP peers, it is necessary to consider all available iBGP paths for a given RT prefix, for building the outbound route filter, and not just the best path."

### Timers

- Section 6: "If a BGP speaker chooses to delay the advertisement of BGP VPN route updates until it receives this End-of-RIB marker, it MUST limit that delay to an upper bound."

## SHOULD/MAY Requirements

### SHOULD

- Section 6: "[SHOULD] generate an End-of-RIB marker, as defined in [8], for the Route Target membership (afi, safi), regardless of whether graceful-restart is enabled on the BGP session." - Consequence: Receiver cannot know when initial RT membership exchange is complete
- Section 8: "Implementations SHOULD also provide means to filter RT membership information." - Consequence: Cannot control RT advertisements at administrative boundaries
- Section 6: "A BGP speaker should generate the minimum set of BGP VPN route updates (advertisements and/or withdrawls) necessary to transition between the previous and current state of the route distribution graph" - Consequence: Excessive UPDATE churn

### MAY

- Section 5: "[MAY] participate in the distribution of Route Target information without using the learned information for purposes of VPN NLRI output route filtering" - Use: Transitional deployment, though discouraged

## Operation

### VPN Route Filtering Logic

Section 6: "A VPN NLRI route should be advertised to a peer that participates in the exchange of Route Target membership information if that peer has advertised either:
1. the default Route Target membership NLRI, or
2. a Route Target membership NLRI containing any of the targets contained in the extended communities attribute of the VPN route in question."

### RT Membership Update Processing

Section 6: "When a BGP speaker receives a BGP UPDATE that advertises or withdraws a given Route Target membership NLRI, it should examine the RIB-OUTs of VPN NLRIs and re-evaluate the advertisement status of routes that match the Route Target in question."

### Route Distribution Graph

- RT membership flows from VPN consumers toward VPN providers
- VPN NLRI flows in inverse direction of RT membership
- Graph pruning uses standard BGP path selection (AS-PATH length, etc.)

## Algorithms

### Intra-AS RT Advertisement (Route Reflector)

```
on advertise RT-membership NLRI to iBGP peer:
    if peer is RR client:
        set ORIGINATOR_ID = local router-id
        set NEXT_HOP = local session address
    else:  # non-client peer
        best_path = select_best_path(RT_prefix)
        if best_path.source is non-client AND exists client_path to same RT:
            advertise client_path attributes
        else:
            advertise best_path attributes
```

### Outbound Filter Construction (iBGP)

```
on build VPN outbound filter for iBGP peer:
    filter = empty_set
    for each RT-membership path from peer:
        # Consider ALL paths, not just best
        filter.add(RT-membership.route_target)
    return filter
```

### VPN Route Advertisement Decision

```
on consider advertising VPN route to peer:
    if peer not negotiated RT-membership (AFI=1, SAFI=132):
        advertise normally (no filtering)
        return

    peer_rts = get_advertised_rt_memberships(peer)
    if DEFAULT_RT in peer_rts:
        advertise route
        return

    route_rts = route.extended_communities.filter(is_route_target)
    if any(rt matches prefix in peer_rts for rt in route_rts):
        advertise route
    else:
        withdraw route (or don't advertise)
```

## Pitfalls

### Edge Cases

- **Default RT (length=0):** Acts as wildcard - peer wants ALL VPN routes. Common for RR-to-PE sessions.
- **RT prefix matching:** An RT membership with 64-bit prefix (origin-as + partial RT) matches multiple concrete route targets sharing that prefix. Useful for filtering by Global Administrator field.
- **Originator loop detection:** Standard ORIGINATOR_ID check applies. The modified advertisement rule (set ORIGINATOR_ID to advertiser's router-id) ensures RR client doesn't drop reflected RT membership originated by itself.

### Interop Considerations

- **Non-participating peers:** If peer hasn't negotiated AFI=1/SAFI=132, send all VPN routes unfiltered (RFC 4364 behavior)
- **Partial deployment:** Some implementations may advertise RT membership but not filter outbound (permitted by MAY in Section 5)
- **iBGP path selection:** Must track ALL iBGP paths for RT prefixes, not just best - differs from normal BGP behavior

### Security

- Section 8: "output route filters built from RT membership information NLRIs are not intended for security purposes"
- RT membership filtering is optimization, not access control
- Always apply independent NLRI filtering at administrative boundaries

## Compatibility

### Non-Supporting Peers

- If peer doesn't advertise AFI=1/SAFI=132 capability: send all VPN routes without RT-based filtering
- No fallback negotiation; feature simply not used

### Route Reflector Requirements

- Section 3.2: "These procedures assume that the autonomous-system route reflection topology is configured such that IPv4 unicast routing would work correctly. For instance, route reflection clusters must be contiguous."
