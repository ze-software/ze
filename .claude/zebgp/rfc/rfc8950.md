# RFC 8950: Advertising IPv4 NLRI with an IPv6 Next Hop

## Meta

| Field | Value |
|-------|-------|
| RFC | 8950 |
| Title | Advertising IPv4 Network Layer Reachability Information (NLRI) with an IPv6 Next Hop |
| Status | Standards Track |
| Date | November 2020 |
| Obsoletes | RFC 5549 |
| Updates | - |
| Obsoleted-by | - |

**Purpose:** Extends MP-BGP to allow IPv4 and VPN-IPv4 NLRI to be advertised with IPv6 next-hop addresses, enabling IPv4 routing over IPv6-only cores.

**Scope:** AFI=1 (IPv4) with SAFI=1 (unicast), 2 (multicast), 4 (labeled), 128 (VPN unicast), 129 (VPN multicast)

**Errata:** None reported

## Wire Formats

### Extended Next Hop Encoding Capability

Type code: 5

```
+-----------------------------------------------------+
| NLRI AFI - 1 (2 octets)                             |
+-----------------------------------------------------+
| NLRI SAFI - 1 (2 octets)                            |
+-----------------------------------------------------+
| Nexthop AFI - 1 (2 octets)                          |
+-----------------------------------------------------+
| .....                                               |
+-----------------------------------------------------+
| NLRI AFI - N (2 octets)                             |
+-----------------------------------------------------+
| NLRI SAFI - N (2 octets)                            |
+-----------------------------------------------------+
| Nexthop AFI - N (2 octets)                          |
+-----------------------------------------------------+
```

| Field | Offset | Size | Type | Constraints |
|-------|--------|------|------|-------------|
| NLRI AFI | 0 | 2 | uint16 | Network order. Value 1 (IPv4) for this RFC |
| NLRI SAFI | 2 | 2 | uint16 | Network order. Values 1, 2, 4, 128, or 129 |
| Nexthop AFI | 4 | 2 | uint16 | Network order. Value 2 (IPv6) for this RFC |

**Length calculation:** 6 bytes per triple, repeated N times. Capability Length = 6 * N.

**Valid tuples defined by this RFC:**
- `<1, 1, 2>` - IPv4 unicast with IPv6 next hop
- `<1, 2, 2>` - IPv4 multicast with IPv6 next hop
- `<1, 4, 2>` - IPv4 labeled unicast with IPv6 next hop
- `<1, 128, 2>` - VPN-IPv4 unicast with IPv6 next hop
- `<1, 129, 2>` - VPN-IPv4 multicast with IPv6 next hop

### MP_REACH_NLRI Next Hop Encoding

#### IPv4 NLRI (AFI=1, SAFI=1,2,4)

| Next Hop Length | Encoding |
|-----------------|----------|
| 4 | IPv4 address (standard) |
| 16 | IPv6 global address |
| 32 | IPv6 global + IPv6 link-local (per RFC 2545 Section 3) |

#### VPN-IPv4 NLRI (AFI=1, SAFI=128,129)

| Next Hop Length | Encoding |
|-----------------|----------|
| 12 | VPN-IPv4 address (standard) |
| 24 | RD (8 bytes, all zeros) + IPv6 global address (16 bytes) |
| 48 | RD (8 bytes, all zeros) + IPv6 global (16 bytes) + RD (8 bytes, all zeros) + IPv6 link-local (16 bytes) |

**Key difference from RFC 5549:** VPN-IPv4 with IPv6 next hop now uses VPN-IPv6 encoding (24/48 bytes with RD=0) instead of raw IPv6 (16/32 bytes). This addresses Erratum ID 5253 against RFC 5549 and aligns with VPNv4oIPv4 and VPNv6oIPv6 conventions.

## Encoding Rules

1. Next hop length field determines address family:
   - AFI=1, SAFI=1,2,4: Length 16 or 32 = IPv6
   - AFI=1, SAFI=128,129: Length 24 or 48 = VPN-IPv6

2. For VPN-IPv4 NLRI with IPv6 next hop, the Route Distinguisher in the next hop MUST be set to zero (8 zero bytes).

3. Link-local address is optional; include only when needed for on-link next hop resolution.

4. IPv6 next hop construction follows RFC 2545 Section 3.

## Decoding Rules

1. Read Next Hop Length field from MP_REACH_NLRI
2. Determine address family by length:

| AFI | SAFI | Length | Next Hop Type |
|-----|------|--------|---------------|
| 1 | 1,2,4 | 4 | IPv4 |
| 1 | 1,2,4 | 16 | IPv6 global |
| 1 | 1,2,4 | 32 | IPv6 global + link-local |
| 1 | 128,129 | 12 | VPN-IPv4 |
| 1 | 128,129 | 24 | VPN-IPv6 (RD=0 + IPv6 global) |
| 1 | 128,129 | 48 | VPN-IPv6 (RD=0 + IPv6 global + RD=0 + IPv6 link-local) |

3. Parse next hop bytes according to determined type

## Validation

| Check | Valid | Invalid Action |
|-------|-------|----------------|
| Next hop length for AFI=1 SAFI=1,2,4 | 4, 16, or 32 | Treat as malformed UPDATE |
| Next hop length for AFI=1 SAFI=128,129 | 12, 24, or 48 | Treat as malformed UPDATE |
| RD in VPN-IPv6 next hop | All zeros | Implementation-specific (RFC does not specify) |
| Capability tuple values | Only combinations listed above | Ignore unknown tuples |

## MUST Requirements

### Tx (Sender)
- "A BGP speaker MUST only advertise the IPv4 or VPN-IPv4 NLRI with an IPv6 next hop to a BGP peer if the BGP speaker has first ascertained via the BGP Capability Advertisement that the BGP peer supports the Extended Next Hop Encoding capability for the relevant AFI/SAFI pair." (Section 4)

### Rx (Receiver)
- "The BGP speaker receiving the advertisement MUST use the Length of Next Hop Address field to determine which network-layer protocol the next-hop address belongs to." (Section 3)

### Capability
- "MUST use the Capability Advertisement procedures defined in [RFC5492] with the Extended Next Hop Encoding capability to determine whether its peer supports this" (Section 4)
- "The Capability Code field MUST be set to 5" (Section 4)

### Route Reflection
- "When a next-hop address needs to be passed along unchanged (e.g., as a Route Reflector (RR) would do), its encoding MUST NOT be changed." (Section 5)

## SHOULD/MAY

- [SHOULD] "if a particular BGP session is running over IPvx... the next-hop address SHOULD be specified as an IPvx address" - Use session transport address family when setting self as next hop (Section 5)
- [MAY] "This default behavior may be overridden by policy." - Policy can override default next-hop address family selection (Section 5)

## Error Handling

| Condition | Detect How | Response | Code/Subcode |
|-----------|------------|----------|--------------|
| Unexpected next hop length | Length not in valid set for AFI/SAFI | Treat as malformed UPDATE | Per RFC 4760 |
| IPv6 next hop without capability | Received UPDATE with IPv6 NH but capability not negotiated | Implementation-specific | Not specified |
| RR client lacks capability | Check negotiated capabilities | Do not distribute NLRI to that client | N/A (routing decision) |

## Compatibility

### Behavior with Non-Supporting Peers

1. Do not send Extended Next Hop Encoding capability if not supported locally
2. If peer does not advertise the capability for a given AFI/SAFI, do not send IPv6 next hops for that AFI/SAFI
3. Standard IPv4/VPN-IPv4 next hops remain usable regardless of capability negotiation

### Feature Negotiation

- Extended Next Hop Encoding capability is independent of Multiprotocol Extensions capability (RFC 4760)
- Both capabilities may be advertised; MP capability determines if AFI/SAFI is allowed, Extended NH capability determines if IPv6 next hop is allowed
- Capability must be negotiated per-AFI/SAFI tuple

### Route Reflector Considerations

- RR must not modify next-hop encoding when reflecting routes
- If an RR client cannot handle IPv6-encoded next hops (lacks capability), routes with IPv6 next hops cannot be distributed to that client
- All RR clients should support same encodings for consistent routing

## Pitfalls

### Edge Cases
- **IPv4-mapped IPv6:** The IPv6 next hop could be an IPv4-mapped IPv6 address (::ffff:a.b.c.d). Security policies must account for this.
- **Link-local only:** RFC does not prohibit sending only link-local without global; receivers must handle this per RFC 2545 rules.
- **Mixed capability:** Peer may advertise capability for some but not all SAFIs; must track per-SAFI.

### Interop
- **RFC 5549 legacy:** Some implementations may still use 16/32 byte encoding for VPN-IPv4 (old RFC 5549 behavior). RFC 8950 specifies 24/48 bytes with RD=0. Check peer behavior.
- **Capability order:** Multiple tuples may appear in any order within capability value.

### Security
- **Traffic diversion:** Advertising IPv6 next hop allows traffic redirection across IPv6 infrastructure; may enable attacks from distant points.
- **Next hop validation:** Policies checking next-hop validity must handle both IPv4 and IPv6 address families.

## Changes from RFC 5549

1. **VPN-IPv4 next hop encoding:** Changed from 16/32 bytes (raw IPv6) to 24/48 bytes (VPN-IPv6 with RD=0)
2. **SAFI 129 support:** Added IPv4 VPN Multicast (SAFI=129) support
