# RFC 7911 - Advertisement of Multiple Paths in BGP (ADD-PATH)

## Meta

| Field | Value |
|-------|-------|
| RFC | 7911 |
| Title | Advertisement of Multiple Paths in BGP |
| Status | Standards Track |
| Date | July 2016 |
| Obsoletes | - |
| Updates | - |
| Obsoleted by | - |

**Purpose:** Defines a BGP extension allowing advertisement of multiple paths for the same address prefix, where each path is identified by a Path Identifier in addition to the prefix.

**Scope:** Applies to all AFI/SAFI combinations; negotiated per AFI/SAFI via capability exchange.

## Wire Formats

### ADD-PATH Capability

Capability Code: 69

```
+------------------------------------------------+
| Address Family Identifier (2 octets)           |
+------------------------------------------------+
| Subsequent Address Family Identifier (1 octet) |
+------------------------------------------------+
| Send/Receive (1 octet)                         |
+------------------------------------------------+
```

| Field | Offset | Size | Type | Constraints |
|-------|--------|------|------|-------------|
| AFI | 0 | 2 | uint16 | Per RFC 4760 |
| SAFI | 2 | 1 | uint8 | Per RFC 4760 |
| Send/Receive | 3 | 1 | uint8 | 1, 2, or 3 |

**Length calculation:** 4 bytes per AFI/SAFI tuple. Total capability length = 4 * number of tuples.

**Send/Receive values:**
| Value | Meaning |
|-------|---------|
| 1 | Able to receive multiple paths |
| 2 | Able to send multiple paths |
| 3 | Able to send and receive multiple paths |

### Extended NLRI Encoding (IPv4 Unicast)

```
+--------------------------------+
| Path Identifier (4 octets)     |
+--------------------------------+
| Length (1 octet)               |
+--------------------------------+
| Prefix (variable)              |
+--------------------------------+
```

| Field | Offset | Size | Type | Constraints |
|-------|--------|------|------|-------------|
| Path Identifier | 0 | 4 | uint32 | Locally assigned, opaque |
| Length | 4 | 1 | uint8 | Prefix length in bits (0-32 for IPv4) |
| Prefix | 5 | (Length+7)/8 | bytes | Network byte order |

**Length calculation:** 4 + 1 + ceil(prefix_length/8) bytes per NLRI entry.

### Extended MP_REACH_NLRI / MP_UNREACH_NLRI

For AFI/SAFI combinations using MP_REACH_NLRI (RFC 4760), the Path Identifier is prepended to each NLRI in the same manner:

```
+--------------------------------+
| Path Identifier (4 octets)     |
+--------------------------------+
| NLRI (variable, per AFI/SAFI)  |
+--------------------------------+
```

The format of the NLRI portion after the Path Identifier follows the existing encoding for that AFI/SAFI.

## Encoding Rules

- Path Identifier is 4 octets, network byte order
- Path Identifier assignment is purely local matter
- Path Identifier MUST uniquely identify a path for a given (AFI, SAFI, Prefix) tuple when sent to a neighbor
- Re-advertising speaker MUST generate its own Path Identifier (do not pass through received Path Identifier)
- Receiver SHOULD NOT assume Path Identifier carries any semantic meaning

## Decoding Rules

1. Check if ADD-PATH was negotiated for this AFI/SAFI in both directions:
   - Local advertised Send/Receive includes 2 or 3, AND
   - Remote advertised Send/Receive includes 1 or 3
2. If ADD-PATH negotiated: Parse 4-byte Path Identifier before each NLRI
3. If ADD-PATH not negotiated: Parse NLRI without Path Identifier (standard RFC 4271)
4. Use (Prefix, Path Identifier) as key for route table lookups

## Validation

| Check | Valid | Invalid Action |
|-------|-------|----------------|
| Send/Receive value | 1, 2, or 3 | Treat capability as not understood, ignore [RFC 5492] |
| Multiple AFI/SAFI in single capability | Required | N/A (MUST use single instance) |
| Withdraw with unknown Path Identifier | N/A | Silently ignore |

## MUST Requirements

### Tx (Sender)

- "the Path Identifier MUST be assigned in such a way that the BGP speaker is able to use the (Prefix, Path Identifier) to uniquely identify a path advertised to a neighbor" (Section 2)
- "A BGP speaker that re-advertises a route MUST generate its own Path Identifier to be associated with the re-advertised route" (Section 2)
- "the NLRI encoding MUST be extended by prepending the Path Identifier field" (Section 3)
- "A BGP speaker that wishes to indicate support for multiple AFI/SAFIs MUST do so by including the information in a single instance of the ADD-PATH Capability" (Section 4)
- "For a BGP speaker to be able to send multiple paths to its peer, that BGP speaker MUST advertise the ADD-PATH Capability with the Send/Receive field set to either 2 or 3, and MUST receive from its peer the ADD-PATH Capability with the Send/Receive field set to either 1 or 3, for the corresponding <AFI, SAFI>" (Section 5)

### Rx (Receiver)

- "A BGP speaker MUST follow the procedures defined in [RFC4271] when generating an UPDATE message for a particular <AFI, SAFI> to a peer unless the BGP speaker advertises the ADD-PATH Capability to the peer indicating its ability to send multiple paths for the <AFI, SAFI>, and also receives the ADD-PATH Capability from the peer indicating its ability to receive multiple paths for the <AFI, SAFI>" (Section 5)
- "the speaker MUST generate a route update for the <AFI, SAFI> based on the combination of the address prefix and the Path Identifier, and use the extended NLRI encodings specified in this document" (Section 5)

### State

- "The peer SHALL act accordingly in processing an UPDATE message related to a particular <AFI, SAFI>" (Section 5)

## SHOULD/MAY Requirements

- [SHOULD] "If any other value is received [for Send/Receive], then the capability SHOULD be treated as not understood and ignored [RFC5492]" (Section 4) - prevents protocol errors for future extensions
- [SHOULD] "If a BGP speaker receives a message to withdraw a prefix with a Path Identifier not seen before, it SHOULD silently ignore it" (Section 5) - prevents unnecessary processing
- [SHOULD] "A BGP speaker SHOULD include the best route [RFC4271] when more than one path is advertised to a neighbor, unless it is a path received from that neighbor" (Section 5) - ensures optimal routing
- [SHOULD] "an implementation SHOULD take special care so that the underlying forwarding plane of a 'Receiving Speaker' as described in [RFC4724] is not affected during the graceful restart of a BGP session" (Section 5) - graceful restart consideration

## Error Handling

| Condition | Detect How | Response | Code/Subcode |
|-----------|------------|----------|--------------|
| Invalid Send/Receive value (not 1, 2, 3) | Value check | Treat capability as not understood, ignore | N/A (not error) |
| Withdraw with unknown Path Identifier | Route table lookup | Silently ignore | N/A (not error) |
| Malformed NLRI (truncated) | Length check | UPDATE Message Error / Malformed Attribute List | 3/1 |

Note: ADD-PATH itself does not define new error codes. Standard BGP error handling applies.

## Capability Negotiation

ADD-PATH requires bilateral negotiation per AFI/SAFI:

| Local Advertises | Remote Advertises | Result |
|------------------|-------------------|--------|
| Send (2 or 3) | Receive (1 or 3) | Local can send ADD-PATH to remote |
| Receive (1 or 3) | Send (2 or 3) | Local can receive ADD-PATH from remote |
| Neither condition met | - | Standard encoding (no Path Identifier) |

**Negotiation is directional:** ADD-PATH can be enabled in one direction only.

## Algorithms

### Path Identifier Assignment (Local)

```
For each (AFI, SAFI, Prefix) being advertised to a peer:
    If re-advertising a received route:
        Generate new local Path Identifier (do NOT reuse received)
    Else:
        Assign Path Identifier ensuring uniqueness for (Prefix, Path ID) to this peer
    Store mapping: (AFI, SAFI, Prefix, Path ID) -> route
```

### NLRI Encoding Decision

```
When encoding UPDATE for peer P, AFI/SAFI (A, S):
    local_cap = our ADD-PATH capability for (A, S)
    remote_cap = peer's ADD-PATH capability for (A, S)

    If local_cap.send AND remote_cap.receive:
        Use extended NLRI encoding (prepend Path Identifier)
    Else:
        Use standard NLRI encoding (no Path Identifier)
```

### Route Replacement Logic

```
On receiving UPDATE with NLRI (Path ID, Prefix) for negotiated ADD-PATH:
    key = (AFI, SAFI, Prefix, Path ID)
    If key exists in RIB:
        Replace existing route with new attributes
    Else:
        Insert new route

On receiving WITHDRAW with (Path ID, Prefix):
    key = (AFI, SAFI, Prefix, Path ID)
    If key exists in RIB:
        Remove route
    Else:
        Silently ignore (SHOULD)
```

## Pitfalls

### Edge Cases

- **Path Identifier 0:** Valid value; no special meaning. Treat same as any other identifier.
- **Multiple withdraws:** Withdrawing same (Prefix, Path ID) multiple times is valid; subsequent withdraws should be silently ignored.
- **Path ID reuse after withdrawal:** After withdrawing (Prefix, Path ID=X), the same Path ID X can be reused for a different path to the same prefix.
- **Graceful restart:** Path Identifiers may not persist across restart. Receiver must handle stale path cleanup correctly.
- **Partial AFI/SAFI negotiation:** ADD-PATH may be negotiated for some AFI/SAFIs but not others; each UPDATE must check negotiation state.

### Interop Considerations

- **Capability encoding:** All AFI/SAFI tuples MUST be in single capability instance. Some implementations may incorrectly send multiple capability instances.
- **Asymmetric negotiation:** One direction may use ADD-PATH while the other does not. Implementation must track send/receive state independently.
- **Wire analyzer confusion:** Packet captures cannot decode ADD-PATH encoding without knowing negotiation state.

### Security

- **Memory exhaustion:** Multiple paths per prefix can increase memory usage significantly. Consider limiting maximum paths accepted per prefix.
- **Route oscillation:** While ADD-PATH can help prevent some oscillations (RFC 3345), improper deployment may introduce new instability.

## Compatibility

### With Non-Supporting Peers

- If peer does not advertise ADD-PATH capability: Use standard RFC 4271 encoding (single path per prefix)
- Capability negotiation ensures backward compatibility
- No protocol errors from capability mismatch

### Feature Negotiation Fallback

- ADD-PATH requires explicit capability exchange
- Absence of capability = standard BGP behavior
- No implicit ADD-PATH support detection
