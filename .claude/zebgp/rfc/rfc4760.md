# RFC 4760: Multiprotocol Extensions for BGP-4

## Meta

| Field | Value |
|-------|-------|
| RFC | 4760 |
| Title | Multiprotocol Extensions for BGP-4 |
| Status | Standards Track |
| Date | January 2007 |
| Obsoletes | RFC 2858 |
| Authors | T. Bates, R. Chandra, D. Katz, Y. Rekhter |

**Purpose:** Defines extensions to BGP-4 to carry routing information for multiple Network Layer protocols (IPv6, IPX, L3VPN, etc.) by introducing two new path attributes (MP_REACH_NLRI and MP_UNREACH_NLRI) and a capability advertisement.

**Scope:** All AFI/SAFI combinations. Core SAFIs defined: 1 (unicast), 2 (multicast).

## Wire Formats

### MP_REACH_NLRI Attribute (Type Code 14)

Flags: O=1 T=0 P=0 E=? (Optional, Non-transitive)

```
+---------------------------------------------------------+
| Address Family Identifier (2 octets)                    |
+---------------------------------------------------------+
| Subsequent Address Family Identifier (1 octet)          |
+---------------------------------------------------------+
| Length of Next Hop Network Address (1 octet)            |
+---------------------------------------------------------+
| Network Address of Next Hop (variable)                  |
+---------------------------------------------------------+
| Reserved (1 octet)                                      |
+---------------------------------------------------------+
| Network Layer Reachability Information (variable)       |
+---------------------------------------------------------+
```

| Field | Offset | Size | Type | Constraints |
|-------|--------|------|------|-------------|
| AFI | 0 | 2 | uint16 | IANA Address Family Numbers |
| SAFI | 2 | 1 | uint8 | See SAFI values below |
| NH Length | 3 | 1 | uint8 | Length of next hop in octets |
| Next Hop | 4 | variable | bytes | AFI/SAFI-specific encoding |
| Reserved | 4+NH_len | 1 | uint8 | MUST be 0 on Tx, SHOULD ignore on Rx |
| NLRI | 5+NH_len | variable | bytes | Sequence of <length, prefix> tuples |

**Length calculation:** `attr_len = 4 + NH_len + 1 + NLRI_len`

**Parse order:**
1. Read AFI (2 octets)
2. Read SAFI (1 octet)
3. Read NH_len (1 octet)
4. Read Next Hop (NH_len octets)
5. Skip Reserved (1 octet)
6. Remaining bytes are NLRI

### MP_UNREACH_NLRI Attribute (Type Code 15)

Flags: O=1 T=0 P=0 E=? (Optional, Non-transitive)

```
+---------------------------------------------------------+
| Address Family Identifier (2 octets)                    |
+---------------------------------------------------------+
| Subsequent Address Family Identifier (1 octet)          |
+---------------------------------------------------------+
| Withdrawn Routes (variable)                             |
+---------------------------------------------------------+
```

| Field | Offset | Size | Type | Constraints |
|-------|--------|------|------|-------------|
| AFI | 0 | 2 | uint16 | IANA Address Family Numbers |
| SAFI | 2 | 1 | uint8 | See SAFI values below |
| Withdrawn Routes | 3 | variable | bytes | Sequence of <length, prefix> tuples |

**Length calculation:** `attr_len = 3 + withdrawn_len`

**Parse order:**
1. Read AFI (2 octets)
2. Read SAFI (1 octet)
3. Remaining bytes are Withdrawn Routes NLRI

### Multiprotocol Capability (Code 1)

```
                  0       7      15      23      31
                  +-------+-------+-------+-------+
                  |      AFI      | Res.  | SAFI  |
                  +-------+-------+-------+-------+
```

| Field | Offset | Size | Type | Constraints |
|-------|--------|------|------|-------------|
| AFI | 0 | 2 | uint16 | IANA Address Family Numbers |
| Reserved | 2 | 1 | uint8 | SHOULD be 0 on Tx, SHOULD ignore on Rx |
| SAFI | 3 | 1 | uint8 | SAFI value |

**Capability Code:** 1
**Capability Length:** 4 (fixed)

### NLRI Encoding

```
+---------------------------+
|   Length (1 octet)        |
+---------------------------+
|   Prefix (variable)       |
+---------------------------+
```

| Field | Offset | Size | Type | Constraints |
|-------|--------|------|------|-------------|
| Length | 0 | 1 | uint8 | Prefix length in bits |
| Prefix | 1 | ceil(Length/8) | bytes | Address prefix, trailing bits irrelevant |

**Length 0:** Matches all addresses for the address family (default route).

**Prefix byte count:** `ceil(prefix_bits / 8)` or `(prefix_bits + 7) / 8`

## Encoding Rules

- AFI/SAFI: Network byte order (big-endian)
- Next Hop encoding: Determined by AFI/SAFI combination
- If Next Hop can be from multiple protocols, encoding MUST identify protocol (Section 3)
- Reserved field in MP_REACH_NLRI: MUST set to 0 (Section 3)
- Reserved field in Capability: SHOULD set to 0 (Section 8)
- Prefix trailing bits: Value is irrelevant, may be zero-padded

## Decoding Rules

**MP_REACH_NLRI parse sequence:**
1. Validate minimum length (5 bytes: AFI + SAFI + NH_len + Reserved + at least empty NLRI)
2. Read AFI, SAFI
3. Read NH_len, validate NH_len does not exceed remaining bytes
4. Read Next Hop bytes
5. Skip Reserved byte (do not validate)
6. Parse NLRI tuples until end of attribute

**MP_UNREACH_NLRI parse sequence:**
1. Validate minimum length (3 bytes: AFI + SAFI)
2. Read AFI, SAFI
3. Parse Withdrawn Routes NLRI tuples until end of attribute

**NLRI tuple parsing:**
1. Read length byte (prefix bits)
2. Calculate prefix bytes: `(length + 7) / 8`
3. Read prefix bytes
4. Repeat until no bytes remain

**Unknown/unsupported handling:**
- Unknown AFI/SAFI: Attribute is "incorrect" (see Error Handling)
- Reserved field in MP_REACH_NLRI: SHOULD ignore (Section 3)
- Reserved field in Capability: SHOULD ignore (Section 8)

## Validation

| Check | Valid | Invalid Action |
|-------|-------|----------------|
| AFI/SAFI supported | Locally configured | Delete routes for AFI/SAFI, ignore future |
| Attribute length | >= minimum for type | Attribute incorrect |
| NH_len | Does not exceed remaining | Attribute incorrect |
| NLRI prefix length | <= max for AFI (e.g., 32 for IPv4, 128 for IPv6) | Attribute incorrect |
| NLRI bytes available | Prefix fits in remaining | Attribute incorrect |
| Reserved (MP_REACH) | Any value | SHOULD ignore |

## MUST Requirements

### Tx (Sender)
- "Reserved: A 1 octet field that MUST be set to 0" (Section 3)
- "If the Next Hop is allowed to be from more than one Network Layer protocol, the encoding of the Next Hop MUST provide a way to determine its Network Layer protocol." (Section 3)
- "An UPDATE message that carries the MP_REACH_NLRI MUST also carry the ORIGIN and the AS_PATH attributes (both in EBGP and in IBGP exchanges)." (Section 3)
- "Moreover, in IBGP exchanges such a message MUST also carry the LOCAL_PREF attribute." (Section 3)
- "To have a bi-directional exchange of routing information for a particular <AFI, SAFI> between a pair of BGP speakers, each such speaker MUST advertise to the other (via the Capability Advertisement mechanism) the capability to support that particular <AFI, SAFI> route." (Section 8)

### Rx (Receiver)
- "the speaker MUST delete all the BGP routes received from that neighbor whose AFI/SAFI is the same as the one carried in the incorrect MP_REACH_NLRI or MP_UNREACH_NLRI attribute." (Section 7)

### Validation
- MP_REACH_NLRI or MP_UNREACH_NLRI correctness triggers error handling (Section 7)

### Errors
- Delete routes for AFI/SAFI on incorrect attribute (Section 7)

## SHOULD/MAY

### SHOULD
- "Reserved...SHOULD be ignored upon receipt." (Section 3) - Prevents future compatibility issues
- "the speaker then SHOULD ignore all the subsequent routes with that AFI/SAFI received over that session." (Section 7) - Prevents processing more bad data
- "An UPDATE message that carries no NLRI, other than the one encoded in the MP_REACH_NLRI attribute, SHOULD NOT carry the NEXT_HOP attribute." (Section 3) - Avoids confusion
- "If such a message contains the NEXT_HOP attribute, the BGP speaker that receives the message SHOULD ignore this attribute." (Section 3) - Handle legacy/buggy implementations
- "An UPDATE message SHOULD NOT include the same address prefix (of the same <AFI, SAFI>) in more than one of the following fields..." (Section 3) - Processing is undefined otherwise
- "A BGP speaker that uses Multiprotocol Extensions SHOULD use the Capability Advertisement procedures [BGP-CAP] to determine whether the speaker could use Multiprotocol Extensions with a particular peer." (Section 8)
- "The session SHOULD be terminated with the Notification message code/subcode indicating 'UPDATE Message Error'/'Optional Attribute Error'." (Section 7)
- "Res. - Reserved (8 bit) field. SHOULD be set to 0 by the sender and ignored by the receiver." (Section 8)

### MAY
- "the speaker MAY terminate the BGP session over which the UPDATE message was received." (Section 7) - Alternative to continuing with ignored routes
- "An implementation MAY support all, some, or none of the Subsequent Address Family Identifier values defined in this document." (Section 6) - Flexibility in implementation

## Error Handling

| Condition | Detect How | Response | Code/Subcode |
|-----------|------------|----------|--------------|
| Incorrect MP_REACH_NLRI | Parse failure or unsupported AFI/SAFI | MUST delete routes for that AFI/SAFI | UPDATE Message Error / Optional Attribute Error |
| Incorrect MP_UNREACH_NLRI | Parse failure or unsupported AFI/SAFI | MUST delete routes for that AFI/SAFI | UPDATE Message Error / Optional Attribute Error |
| Subsequent bad routes same AFI/SAFI | After initial error | SHOULD ignore | N/A |
| Session termination (optional) | After incorrect attribute | MAY terminate | UPDATE Message Error / Optional Attribute Error |

**Key behavior:** Error handling is per-AFI/SAFI, not session-fatal by default. This allows multi-protocol sessions to continue for other AFI/SAFI combinations.

## Pitfalls

### Edge Cases
- **Zero-length prefix:** Valid, represents default route for the AFI
- **Empty NLRI:** MP_REACH_NLRI with no NLRI tuples after Reserved byte is valid (announces next hop only? - unusual)
- **Empty Withdrawn Routes:** MP_UNREACH_NLRI with just AFI/SAFI is valid (no routes withdrawn)
- **Duplicate prefix in same UPDATE:** "processing is undefined" (Section 3) - implementations vary
- **Multiple MP_REACH_NLRI for same AFI/SAFI:** Not explicitly forbidden but unusual
- **Next Hop in wrong address family:** Parsing depends on AFI/SAFI, not self-describing

### Interop
- **Reserved field non-zero:** Some implementations may reject; SHOULD ignore per RFC
- **Capability Reserved field:** Note in Section 8 warns "not setting the field value to 0 may create issues for a receiver not ignoring the field"
- **NEXT_HOP attribute with MP_REACH_NLRI:** Should be ignored but older implementations may process it

### Security
- **No authentication of Next Hop:** Same as NEXT_HOP attribute - relies on BGP session security
- **Malformed NLRI:** Can cause parsing failures; validate bounds carefully

## Compatibility

### With Non-Supporting Peers
- MP_REACH_NLRI and MP_UNREACH_NLRI are Optional Non-Transitive
- Non-supporting peers silently ignore these attributes
- Non-supporting peers do not propagate the attributes
- Use Capability Advertisement to detect support before sending

### Feature Negotiation Fallback
- If peer does not advertise Multiprotocol Capability for an AFI/SAFI, do not send routes for that AFI/SAFI
- IPv4 Unicast can fall back to traditional NLRI/NEXT_HOP if peer lacks MP capability

## Errata

### Verified Errata

**Errata ID 1573 (Verified)** - Section 4

The AFI and SAFI field descriptions in MP_UNREACH_NLRI incorrectly reference "Next Hop" which does not exist in this attribute. The text was copied from MP_REACH_NLRI without adjustment.

**Affected text:** Section 4 AFI and SAFI descriptions mention "the address carried in the Next Hop field"

**Correction:** Should reference "Withdrawn Routes Network Layer Reachability Information" instead.

**Implementation note:** Parse MP_UNREACH_NLRI without expecting Next Hop field.

### Held for Document Update

**Errata ID 1778, 3848, 3849** - Section 3
- References to "MP_NLRI attribute" should be "MP_REACH_NLRI attribute"
- "Network Reachability Information" should be "Network Layer Reachability Information"

**Errata ID 5120** - Sections 3 and 5
- NLRI encoding section applies specifically to SAFI 1 and 2; other SAFIs may define different encodings
