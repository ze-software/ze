         The ATTR_DISCARD Path Attribute for BGP UPDATE Messages
                   draft-mangin-idr-attr-discard-00

Abstract

   RFC 7606 defines revised error handling for BGP UPDATE messages,
   including an "attribute discard" action where a malformed optional
   attribute is removed and the UPDATE continues to be processed.
   However, RFC 7606 does not specify a wire-level mechanism for this
   operation.  Implementations must either rebuild the path attributes
   section of the UPDATE (expensive) or leave malformed bytes in place
   (risking misinterpretation by downstream consumers).

   This document defines the ATTR_DISCARD path attribute, an optional
   attribute that serves as an in-place tombstone marker.  When a BGP
   speaker applies attribute-discard per RFC 7606, it overwrites the
   malformed attribute's flags, type code, and (when present) first
   value byte, preserving the wire buffer layout without reallocation.
   The first value byte, when present, records the original attribute
   type code for diagnostic purposes.  The Transitive bit is inherited
   from the original attribute, allowing implementations to choose
   whether the marker propagates to peers or remains local.

   ATTR_DISCARD is a wire-level construct.  It is not stored in the
   Adj-RIB-In and MUST NOT affect route selection.

1.  Introduction

   BGP [RFC4271] uses UPDATE messages to convey routing information
   between peers.  Each UPDATE message may contain a set of path
   attributes that describe properties of the advertised routes.

   RFC 7606 [RFC7606] revised the error handling for BGP UPDATE messages
   to reduce unnecessary session resets.  Section 2 of RFC 7606 defines
   four graduated error-handling actions:

     - Session reset: terminate the BGP session (strongest)
     - AFI/SAFI disable: cease to use a particular AFI/SAFI
       with the peer
     - Treat-as-withdraw: treat the UPDATE as a route withdrawal
     - Attribute discard: discard the malformed attribute and continue

   The "attribute discard" action is the mildest response.  It applies
   to optional attributes whose malformation does not affect the
   ability to parse the remainder of the UPDATE.  RFC 7606 Section 2
   states:

     "The attribute MUST be discarded, and the UPDATE message
      continues to be processed."

   However, RFC 7606 does not specify HOW the attribute is discarded
   at the wire level.  This leaves implementations with two choices,
   both unsatisfactory:

     1. Rebuild the path attributes section, removing the malformed
        attribute.  This requires reallocating the buffer and shifting
        all subsequent bytes -- expensive on high-throughput routers.

     2. Leave the malformed bytes in place and rely on per-attribute
        "skip" logic in all downstream consumers.  This is fragile:
        any code path that reads the raw bytes (logging, forwarding,
        caching) may encounter the malformed data.

   This document defines a third option: overwrite the malformed
   attribute's header and first value byte in place with a standardized
   tombstone marker, the ATTR_DISCARD path attribute.

   A key design feature is that ATTR_DISCARD inherits the Transitive
   bit from the original malformed attribute.  This provides two
   behaviors:

     - If the original attribute was transitive (e.g., COMMUNITY,
       AGGREGATOR), the marker propagates along the AS path, providing
       network-wide visibility that a malformation was corrected.

     - If the original attribute was non-transitive (e.g.,
       ORIGINATOR_ID, CLUSTER_LIST), the marker remains local,
       matching the original attribute's propagation scope.

   Additionally, the first byte of the ATTR_DISCARD value records the
   original attribute's type code, enabling diagnostic tools to
   identify which attribute was discarded without interpreting the
   remaining value bytes.

   Implementations MAY override this default behavior through local
   policy, forcing the marker to always propagate or never propagate
   regardless of the original attribute's transitivity.

3.  Problem Statement

   Consider a BGP speaker that receives an UPDATE with the following
   path attributes:

     +--------+--------+--------+----------------------------+
     | Flags  |  Code  | Length |          Value             |
     +--------+--------+--------+----------------------------+
     |  0x40  |   1    |   1    | 0x00 (ORIGIN = IGP)        |
     |  0x40  |   2    |   6    | AS_PATH (valid)            |
     |  0x40  |   3    |   4    | NEXT_HOP (valid)           |
     |  0xC0  |   7    |   5    | AGGREGATOR (MALFORMED)     |
     |  0xC0  |   8    |   8    | COMMUNITY (valid)          |
     +--------+--------+--------+----------------------------+

   Per RFC 7606 Section 7.7, a malformed AGGREGATOR (wrong length for
   the negotiated AS size) calls for attribute-discard.  The
   implementation must "discard" the AGGREGATOR attribute while
   preserving the UPDATE for continued processing.

   The path attributes section is a contiguous byte sequence.  Removing
   the AGGREGATOR (header + 5 value bytes) requires:

     - Shifting COMMUNITY bytes backward
     - Updating the Total Path Attribute Length field
     - Potentially invalidating any cached offsets, iterators, or
       pointers into the buffer

   On a router processing hundreds of thousands of UPDATEs per second,
   this reallocation and byte-shifting is undesirable.  Modern BGP
   implementations use zero-copy techniques where wire bytes are
   referenced directly by iterators and caches.  Buffer rebuilding
   defeats these optimizations.

   The ATTR_DISCARD attribute solves this by overwriting 3 bytes in
   place: the flags byte, the type code byte, and the first value byte
   (which records the original attribute's type code for diagnostic
   purposes).  No bytes are shifted.  No buffers are reallocated.  The
   remaining value bytes are untouched and will never be interpreted
   because no conforming implementation parses the ATTR_DISCARD value
   beyond the first byte.  (The zero-copy benefit applies primarily
   to the receive path and intra-AS forwarding; see Appendix A.1 for
   trade-offs on the outbound path under non-default forwarding
   policies.)

   After applying ATTR_DISCARD to the malformed AGGREGATOR:

     +--------+--------+--------+----------------------------+
     | Flags  |  Code  | Length |          Value             |
     +--------+--------+--------+----------------------------+
     |  0x40  |   1    |   1    | 0x00 (ORIGIN = IGP)        |
     |  0x40  |   2    |   6    | AS_PATH (valid)            |
     |  0x40  |   3    |   4    | NEXT_HOP (valid)           |
     |  0xC0  |  TBD   |   5    | 0x07, <4 bytes>            |
     |  0xC0  |   8    |   8    | COMMUNITY (valid)          |
     +--------+--------+--------+----------------------------+

   The formula produces 0x80 | (0xC0 & 0x50) = 0xC0, which happens to
   match the original flags because AGGREGATOR is already optional and
   transitive.  The type code is overwritten with TBD.  The first
   value byte now contains 0x07 (the original AGGREGATOR type code).
   The marker inherits the Transitive bit, so downstream speakers
   will forward it (with Partial bit set per RFC 4271 Section 5),
   providing network-wide visibility of the discard event.

4.  ATTR_DISCARD Path Attribute

4.1.  Attribute Properties

     Attribute Name:    ATTR_DISCARD
     Attribute Code:    TBD (see Section 8)
     Attribute Flags:   Optional; Transitivity inherited (see 4.2)
     Attribute Length:   Variable (inherited from overwritten attribute)
     Attribute Value:    If value length >= 1, byte 0 is the original
                         attribute type code; remaining bytes are
                         opaque (MUST NOT be interpreted by receivers
                         beyond byte 0).  If value length is 0, no
                         value bytes are present and the original
                         type code is not recorded (see Section 5.1).

4.2.  Transitivity Inheritance

   Unlike most BGP path attributes, ATTR_DISCARD does not have a
   fixed Transitive bit.  Instead, the Transitive bit is inherited
   from the original attribute that was overwritten.

   This produces two propagation behaviors:

     a) Original attribute was transitive:

        The ATTR_DISCARD marker is optional and transitive.  Per
        RFC 4271 Section 5, BGP speakers that do not recognize an
        optional transitive attribute MUST accept it and pass it
        along to other BGP speakers, setting the Partial bit.

        This means the marker propagates along the AS path.
        Downstream speakers (and their operators) can observe that
        an attribute was discarded upstream.

     b) Original attribute was non-transitive:

        The ATTR_DISCARD marker is optional and non-transitive.  Per
        RFC 4271 Section 5, BGP speakers that do not recognize an
        optional non-transitive attribute silently ignore it.

        The marker remains local to the speaker that performed the
        discard, matching the original attribute's scope.

   The flag requirements for ATTR_DISCARD are:

     Optional bit (bit 0):       MUST be 1
     Transitive bit (bit 1):     Inherited from the original attribute
     Partial bit (bit 2):        MUST be 0 at generation
     Extended Length bit (bit 3): Preserved from the original attribute

   The general formula for computing the overwritten flags byte:

     new_flags = 0x80 | (original_flags & 0x50)

   Where:
     - 0x80 sets the Optional bit
     - 0x50 is the mask for Transitive (0x40) and Extended Length
       (0x10) bits; Partial (0x20) is excluded so that the Partial
       bit is cleared (see below)

   The Partial bit is cleared at generation.  When a downstream
   speaker forwards an unrecognized optional transitive attribute,
   it sets the Partial bit per RFC 4271 Section 5.  This allows
   receivers to distinguish between "generated by the discarding
   speaker" (Partial=0) and "forwarded through intermediate
   speakers" (Partial=1).

4.3.  Wire Format

   In all examples, three bytes are modified: the flags byte (if
   the Optional bit must be set), the type code byte, and the first
   value byte (overwritten with the original type code).  The length
   field and remaining value bytes are untouched.

   Example 1: Transitive original (AGGREGATOR, flags 0xC0)

   Before overwrite:

     +------+------+------+------+-----------+
     | 0xC0 | 0x07 | 0x05 | 0xNN | <4 bytes> |
     +------+------+------+------+-----------+
      flags   code   len    val[0]  val[1..4]

   After overwrite:

     +------+------+------+------+-----------+
     | 0xC0 | TBD  | 0x05 | 0x07 | <4 bytes> |
     +------+------+------+------+-----------+
      flags   code   len    orig    (opaque)

   The flags byte is unchanged (0xC0 = optional, transitive);
   the type code is overwritten with TBD; the first value byte is
   overwritten with 0x07 (original AGGREGATOR code).  The marker
   is transitive and will propagate to peers.

   Example 2: Non-transitive original (ORIGINATOR_ID, flags 0x80)

   Before overwrite:

     +------+------+------+------+-----------+
     | 0x80 | 0x09 | 0x04 | 0xNN | <3 bytes> |
     +------+------+------+------+-----------+
      flags   code   len    val[0]  val[1..3]

   After overwrite:

     +------+------+------+------+-----------+
     | 0x80 | TBD  | 0x04 | 0x09 | <3 bytes> |
     +------+------+------+------+-----------+
      flags   code   len    orig    (opaque)

   The flags byte is unchanged (0x80 = optional, non-transitive).
   The marker remains local and will not propagate.

   Example 3: Well-known transitive original (LOCAL_PREF from EBGP,
   flags 0x40)

   Before overwrite:

     +------+------+------+------+-----------+
     | 0x40 | 0x05 | 0x04 | 0xNN | <3 bytes> |
     +------+------+------+------+-----------+
      flags   code   len    val[0]  val[1..3]

   After overwrite:

     +------+------+------+------+-----------+
     | 0xC0 | TBD  | 0x04 | 0x05 | <3 bytes> |
     +------+------+------+------+-----------+
      flags   code   len    orig    (opaque)

   Here the flags change from 0x40 (well-known, transitive) to 0xC0
   (optional, transitive).  The Optional bit is set because
   ATTR_DISCARD is always optional.  The Transitive bit is preserved
   from the original attribute.  The first value byte records 0x05
   (original LOCAL_PREF code).  The marker will propagate to peers,
   signaling that a well-known attribute was discarded.  Note:
   LOCAL_PREF from an EBGP peer is a policy-based discard per
   RFC 7606 Section 7.5 (the attribute is removed regardless of
   whether it is well-formed), not a malformation detection.

   Example 4: Extended length attribute (flags with bit 3 set)

   Before overwrite:

     +------+------+--------+------+-----------+
     | 0xD0 | 0x10 | 0x0100 | 0xNN | <255 B>   |
     +------+------+--------+------+-----------+
      flags   code   len      val[0]  val[1..255]

   After overwrite:

     +------+------+--------+------+-----------+
     | 0xD0 | TBD  | 0x0100 | 0x10 | <255 B>   |
     +------+------+--------+------+-----------+
      flags   code   len      orig    (opaque)

   The Extended Length bit (0x10) is preserved, so the 2-byte
   length field is correctly parsed by all speakers.  The first
   value byte records 0x10 (attribute type code 16, Extended
   Community).  Note that the value 0x10 also appears in the flags
   byte as the Extended Length bit; the two uses are unrelated and
   distinguished by their position in the attribute header.

5.  Processing Rules

5.1.  Generating ATTR_DISCARD (Sender)

   A BGP speaker that determines an attribute requires the
   "attribute discard" action per RFC 7606 MAY use the following
   procedure instead of rebuilding the path attributes section:

     1. Locate the malformed attribute's header in the path attributes
        byte sequence.  The header starts at the flags byte.

     2. Overwrite the flags byte:

          new_flags = 0x80 | (original_flags & 0x50)

        This sets the Optional bit, preserves the Transitive bit
        and Extended Length bit, and clears the Partial bit.

     3. Save the original type code byte (for use in step 5 or local
        logging).

     4. Overwrite the type code byte with the ATTR_DISCARD code (TBD).

     5. If the attribute value length is zero (i.e., the length field
        is 0), skip this step -- there is no value byte to overwrite.
        The original type code is not recorded in the marker; the
        implementation SHOULD log it locally instead.  Otherwise,
        overwrite the first byte of the attribute value with the
        original type code saved in step 3.

     6. Do NOT modify the length field or any value bytes beyond
        the first.

     7. Continue processing the UPDATE message normally.

   Note: Zero-length attribute-discard cases are uncommon but possible.
   RFC 7606 Section 7.5 specifies attribute-discard for LOCAL_PREF
   received from an external neighbor regardless of length; Sections
   7.9 and 7.10 do the same for ORIGINATOR_ID and CLUSTER_LIST.
   Section 7.7 specifies attribute-discard for AGGREGATOR with an
   incorrect length, which includes length zero.  In all these cases,
   the attribute value may be empty, and the overwrite of the first
   value byte MUST be skipped to avoid corrupting adjacent attribute
   data.

   The following table summarizes which RFC 7606 attribute-discard
   triggers may produce zero-length ATTR_DISCARD markers (where the
   original type code cannot be recorded in the first value byte):

     +-----------------+--------+----------------+------------------+
     | Attribute       | RFC    | Trigger        | Zero-length      |
     |                 | 7606   |                | possible?        |
     +-----------------+--------+----------------+------------------+
     | LOCAL_PREF      | 7.5    | EBGP policy    | Yes (any length) |
     | ATOMIC_AGGREGATE| 7.6    | Length != 0    | No (len > 0)     |
     | AGGREGATOR      | 7.7    | Wrong length   | Yes (len = 0)    |
     | ORIGINATOR_ID   | 7.9    | EBGP policy    | Yes (any length) |
     | CLUSTER_LIST    | 7.10   | EBGP policy    | Yes (any length) |
     +-----------------+--------+----------------+------------------+

   Note: Certain "attribute discard" actions in RFC 7606 are policy-
   based rather than error-based: LOCAL_PREF from an external neighbor
   (Section 7.5), ORIGINATOR_ID from an external neighbor (Section
   7.9), and CLUSTER_LIST from an external neighbor (Section 7.10)
   are discarded regardless of whether the attribute value is well-
   formed.  These are routine policy actions, not protocol errors.
   Implementations SHOULD distinguish between policy-based discards
   and malformation-based discards in their logging and counters
   (Section 6.2).  Applying ATTR_DISCARD to policy-based discards
   generates markers on every affected UPDATE (transitive for
   LOCAL_PREF, non-transitive for ORIGINATOR_ID and CLUSTER_LIST),
   which may be frequent in networks where external peers routinely
   include these attributes.  Implementations MAY choose to apply
   ATTR_DISCARD only to genuine malformation cases (e.g., AGGREGATOR
   with incorrect length, ATOMIC_AGGREGATE with non-zero length) and
   use conventional attribute removal for policy-based discards
   that do not indicate a protocol error.

   An implementation MAY override the inherited Transitive bit at
   generation time by clearing it:

     new_flags = 0x80 | (original_flags & 0x10)

   This produces a non-transitive marker regardless of the original
   attribute's transitivity, confining the marker to the local
   speaker.  This is useful when the operator does not want to
   propagate error-handling artifacts to peers.  Conversely, an
   implementation MUST NOT set the Transitive bit if it was not set
   in the original attribute; promoting a non-transitive attribute
   to transitive scope at generation time would violate the original
   attribute's propagation intent.  (Note: the "propagate" forwarding
   policy in Section 5.3c may set the Transitive bit at forwarding
   time on a received marker; this restriction applies only to the
   initial generation of the ATTR_DISCARD overwrite.)

   The choice between inheriting and clearing the Transitive bit
   SHOULD be configurable, with the default being to inherit.

   A BGP speaker MUST NOT generate an ATTR_DISCARD attribute except
   as a result of applying the "attribute discard" action defined in
   RFC 7606.  That is, ATTR_DISCARD is not an attribute that an
   originating speaker includes in a fresh UPDATE; it is strictly an
   error-handling artifact.

   A BGP speaker SHOULD log the original attribute code and the reason
   for discard when generating an ATTR_DISCARD marker.  This aids
   operational debugging without requiring the discarded data to remain
   parseable.

   A BGP speaker MAY apply ATTR_DISCARD to multiple attributes within
   the same UPDATE if multiple attributes independently require the
   "attribute discard" action.  See Section 5.7 for the implications
   of multiple markers sharing the same type code.

5.2.  Receiving ATTR_DISCARD

   The behavior of a receiving BGP speaker depends on whether it
   recognizes the ATTR_DISCARD attribute code.

5.2.1.  Non-Recognizing Speakers

   A BGP speaker that does not recognize the ATTR_DISCARD attribute
   code applies standard RFC 4271 Section 5 processing based on the
   attribute's flags:

     - If the Transitive bit is set (the original attribute was
       transitive): the speaker MUST accept the attribute and pass it
       along to other BGP speakers, setting the Partial bit.

     - If the Transitive bit is clear (the original attribute was
       non-transitive): the speaker silently ignores the attribute.

   No implementation changes are required for non-recognizing speakers.
   The existing RFC 4271 machinery handles both cases correctly.

5.2.2.  Recognizing Speakers

   A BGP speaker that recognizes the ATTR_DISCARD attribute code:

     - MUST NOT attempt to interpret the value bytes beyond the first
       byte.  Bytes 1 through N-1 of the value are opaque and have no
       defined structure.

     - SHOULD read the first value byte, if present (value length
       >= 1), to determine the original attribute type code that was
       discarded.  This enables per-attribute-code counters and more
       specific diagnostic output.  A zero-length ATTR_DISCARD does
       not carry this information (see Section 5.1, step 5).

     - SHOULD increment a counter for discarded attributes received,
       broken down by peer and by original attribute code (from the
       first value byte), for operational monitoring (see Section 6.2).

     - SHOULD log the presence of ATTR_DISCARD attributes for
       diagnostic purposes, including the original attribute code
       (first value byte, if value length >= 1), the inherited
       length (which reveals the original attribute's size), and
       whether the Partial bit is set (indicating the marker was
       forwarded through intermediate speakers).

     - For forwarding behavior, see Section 5.3.

5.2.3.  Malformed ATTR_DISCARD

   Per the guidance of RFC 7606 Section 8, this section defines what
   constitutes a malformed ATTR_DISCARD attribute and how it is handled.

   ATTR_DISCARD has no currently defined malformation conditions.
   Any value length (including zero) is valid: a zero-length
   ATTR_DISCARD is legitimately generated when the overwritten
   attribute had no value bytes (see Section 5.1, step 5).
   Receivers MUST NOT treat any ATTR_DISCARD as malformed based
   solely on its length.

   ATTR_DISCARD has no effect on route selection or installation
   (Section 6.5).  Therefore, per RFC 7606 Section 2, should any
   future revision define malformation conditions for ATTR_DISCARD,
   such a malformed marker SHALL be handled using the approach of
   "attribute discard" (i.e., the marker itself is discarded and
   the UPDATE continues to be processed).

   If an implementation applies the ATTR_DISCARD overwrite mechanism
   to discard a received ATTR_DISCARD marker, the first value byte
   of the resulting marker will contain the ATTR_DISCARD type code
   (TBD) rather than the original attribute's type code.  The
   original diagnostic information is lost from the wire.
   Implementations SHOULD log the first value byte of the received
   ATTR_DISCARD marker before overwriting it.

5.3.  Forwarding Policy

   When a BGP speaker that recognizes ATTR_DISCARD receives an UPDATE
   containing one or more ATTR_DISCARD attributes, the default
   forwarding behavior follows the Transitive bit as with any optional
   attribute:

     - Transitive ATTR_DISCARD: forwarded to peers with Partial bit
       set (RFC 4271 Section 5).

     - Non-transitive ATTR_DISCARD: not forwarded.

   Implementations SHOULD provide a configurable policy to override
   this default, with at least the following options:

     a) "inherit" (default):

        Transitive ATTR_DISCARD markers are forwarded within the AS
        via IBGP, providing diagnostic visibility to all speakers
        (including non-recognizing route reflectors, which forward
        them per RFC 4271 Section 5).

        At the originating AS's EBGP boundary, the sending speaker
        controls propagation.  A recognizing EBGP speaker SHOULD
        clear the Transitive bit before forwarding the marker to
        the EBGP peer.  This prevents the peer from propagating
        the marker further.  However, because the marker becomes
        non-transitive, a non-recognizing peer will silently ignore
        it per RFC 4271 Section 5 -- it will not be visible at the
        BGP layer on that peer.  The marker remains observable via
        BMP [RFC7854] or packet-level inspection at the receiving
        AS.  A recognizing peer will process it normally
        (Section 5.2.2).

        Non-recognizing EBGP speakers in the originating AS will
        forward the marker as any unrecognized optional transitive
        attribute, without containment.  Operators who cannot ensure
        that all their EBGP speakers recognize ATTR_DISCARD SHOULD
        clear the Transitive bit at generation time (Section 5.1)
        to prevent uncontrolled propagation.  This trades intra-AS
        visibility via non-recognizing route reflectors for reliable
        containment at the AS boundary.

        A recognizing speaker that receives a transitive ATTR_DISCARD
        from an EBGP peer SHOULD also clear the Transitive bit after
        processing, as defense in depth.  However, the originating AS
        MUST NOT rely on the receiver's behavior for containment, as
        the receiver may not support this specification.

        Non-transitive ATTR_DISCARD markers are silently ignored by
        non-recognizing speakers per RFC 4271 Section 5 and are not
        forwarded by recognizing speakers under the default "inherit"
        policy.  (The "propagate" policy overrides this; see 5.3c.)

     b) "strip":

        Remove ALL ATTR_DISCARD markers before forwarding, regardless
        of transitivity.  This prevents marker propagation beyond the
        local speaker, useful when operators do not want to expose
        error-handling artifacts to peers.

        When "strip" is active, the implementation MUST exclude
        ATTR_DISCARD attributes from forwarded path attributes.  If
        the implementation uses zero-copy forwarding, it MUST either
        clear the Transitive bit (converting to non-transitive, which
        will then be silently ignored by non-recognizing speakers) or
        rebuild the affected portion of the path attributes.

     c) "propagate":

        Forward ALL ATTR_DISCARD markers as transitive, overriding
        the default containment behavior of "inherit" and setting the
        Transitive bit to 1 if necessary.  This provides maximum
        diagnostic visibility across the network, even across AS
        boundaries and even for attributes that were originally
        non-transitive.

        When "propagate" is active and the ATTR_DISCARD marker has
        the Transitive bit clear, the implementation MUST set the
        Transitive bit before forwarding.  The Partial bit SHOULD
        NOT be set by a recognizing speaker that explicitly forwards
        the attribute, as the Partial bit per RFC 4271 Section 5
        indicates the attribute was not recognized.  Non-recognizing
        speakers that subsequently forward the marker will set the
        Partial bit per standard processing.

   The policy SHOULD be configurable per peer-group or per neighbor,
   allowing operators to strip markers on external sessions while
   propagating them within their own network.

5.4.  Interaction with Route Reflectors

   A route reflector [RFC4456] that applies ATTR_DISCARD to an
   attribute in a received UPDATE reflects the marker to clients and
   non-clients according to the Transitive bit and the configured
   forwarding policy (Section 5.3).

   Under the default "inherit" policy:

     - Transitive ATTR_DISCARD markers are reflected to clients and
       non-clients, preserving the existing Partial bit value.

     - Non-transitive ATTR_DISCARD markers are not reflected.

   A route reflector that validates inbound UPDATEs and applies
   attribute-discard MAY choose to use the "propagate" policy to
   ensure all clients receive visibility of the discard event,
   regardless of the original attribute's transitivity.

   If the route reflector forwards UPDATE messages using zero-copy
   techniques (referencing the original wire bytes) and the configured
   policy is "strip", it MUST ensure that ATTR_DISCARD markers are
   excluded from the forwarded path attributes.

5.5.  Interaction with Confederation Boundaries

   At confederation boundaries [RFC5065], ATTR_DISCARD attributes are
   handled according to their Transitive bit:

     - Transitive ATTR_DISCARD: propagated across the confederation
       boundary, preserving the existing Partial bit value.

     - Non-transitive ATTR_DISCARD: not propagated across the
       confederation boundary.

   This behavior follows naturally from RFC 5065 and requires no
   special implementation.  The forwarding policy (Section 5.3) MAY
   be applied at confederation boundaries to override this default.

5.6.  Interaction with ATTR_SET

   RFC 6368 [RFC6368] defines the ATTR_SET attribute, which contains
   an embedded set of path attributes.  If a malformed attribute is
   found inside an ATTR_SET value, the ATTR_DISCARD mechanism defined
   in this document MUST NOT be applied to the embedded attribute.

   ATTR_DISCARD operates on top-level path attributes only.  It
   relies on overwriting the attribute header at a known offset in
   the path attributes byte sequence; embedded attributes within
   ATTR_SET have a different encapsulation and may be subject to
   additional validation by the ATTR_SET consumer.

   Malformed attributes inside ATTR_SET SHOULD be handled per the
   error handling rules of RFC 6368 and RFC 7606.  If the ATTR_SET
   itself is malformed (e.g., its contained attributes cannot be
   parsed), the error handling action for ATTR_SET as a whole applies.

5.7.  Duplicate Markers in the Same UPDATE

   When multiple attributes in the same UPDATE independently require
   the "attribute discard" action, the sender generates multiple
   ATTR_DISCARD markers, each in the position of its respective
   original attribute.  All markers share the same type code (TBD).

   RFC 7606 Section 3.g states that when the same attribute type
   code appears more than once in an UPDATE (excluding MP_REACH_NLRI
   and MP_UNREACH_NLRI), a receiver SHALL discard all occurrences
   except the first.  This rule applies to ATTR_DISCARD markers:
   a conforming receiver will retain only the first ATTR_DISCARD
   marker and discard the rest.

   The surviving marker is the one that appears at the earliest byte
   position in the path attributes section.  Since the ATTR_DISCARD
   overwrite preserves attribute positions, this is the original
   attribute that appeared earliest in the UPDATE.  The sender has no
   control over which marker survives at the receiver beyond the
   attribute ordering already present in the received UPDATE.

   Note: An UPDATE containing multiple ATTR_DISCARD markers has
   duplicate attribute type codes.  While RFC 4271 Section 6.3
   originally treated duplicate attributes as an error, RFC 7606
   Section 3.g replaces this with "retain first, discard rest" for
   non-MP attributes.  The sender is aware that only the first marker
   survives at the receiver.

   Consequences:

     - Receivers see at most one ATTR_DISCARD marker per UPDATE,
       regardless of how many attributes were discarded by the sender.

     - The first value byte of the surviving marker identifies the
       original attribute code from the positionally earliest
       discarded attribute; information about other discarded
       attributes is lost at the receiver.

     - Counters at the receiver (Section 6.2) reflect the first
       discarded attribute only, not the total count.

   The sender still benefits from overwriting all malformed attributes
   in place (preventing misinterpretation of malformed bytes by local
   consumers), even though remote receivers see only one marker.

   Implementations that generate ATTR_DISCARD markers SHOULD log all
   discard events locally, including the original attribute code and
   reason for each, so that the full diagnostic information is
   available at the point of detection.

   Implementations should be aware that an UPDATE containing multiple
   ATTR_DISCARD markers has duplicate attribute type codes, which some
   BGP speakers may log as anomalies.  In deployments where
   ATTR_DISCARD is used, operators SHOULD configure their monitoring
   systems to recognize duplicate ATTR_DISCARD type codes as expected
   behavior rather than protocol errors.

6.  Operational Considerations

6.1.  Diagnostic Visibility

   The presence of ATTR_DISCARD attributes in received UPDATEs
   provides valuable operational information:

     - It signals that an upstream speaker detected and corrected a
       malformed attribute per RFC 7606.

     - When the marker has non-zero length, the first value byte
       identifies the original attribute type code that was discarded,
       enabling operators to determine which attribute was malformed
       without inspecting the remaining opaque bytes.  A zero-length
       marker (Section 5.1, step 5) does not carry this information.

     - The Partial bit distinguishes locally-generated markers
       (Partial=0) from forwarded markers (Partial=1), indicating
       whether the discard occurred at the immediate peer or further
       upstream.

     - The inherited length field reveals the size of the original
       malformed attribute, which may aid diagnosis.

     - Transitive markers provide network-wide visibility: an operator
       at any point along the AS path can observe that a discard event
       occurred, even if it happened many hops away.

   Implementations SHOULD make ATTR_DISCARD attributes visible in
   diagnostic outputs (route dumps, packet traces, monitoring systems)
   with clear labeling, for example:

     ATTR_DISCARD (original=AGGREGATOR(7), length=5, transitive,
                   partial, remaining data not interpreted)

6.2.  Counting Discarded Attributes

   Implementations SHOULD maintain the following counters:

     - attr-discard-applied: Number of attributes overwritten with
       ATTR_DISCARD on received UPDATEs (per peer, per original
       attribute code).

     - attr-discard-received: Number of ATTR_DISCARD attributes
       observed in received UPDATEs (per peer, per original attribute
       code as read from the first value byte).  A non-zero value
       for this counter indicates that an upstream speaker is
       performing RFC 7606 attribute-discard with this mechanism.

     - attr-discard-received-partial: Subset of attr-discard-received
       where the Partial bit is set, indicating the marker was
       forwarded through at least one intermediate speaker (the
       discard did not occur at the immediate peer).

   These counters enable operators to identify persistent
   malformation sources without requiring packet-level debugging.

   Note: Due to the duplicate-marker deduplication described in
   Section 5.7, the attr-discard-received counter reflects only the
   first discarded attribute per UPDATE.  The attr-discard-applied
   counter (locally generated) is not affected by deduplication.

6.3.  Backward Compatibility

   The ATTR_DISCARD attribute is fully backward compatible:

     - BGP speakers that do not recognize the attribute code apply
       standard RFC 4271 Section 5 processing: transitive markers
       are forwarded with Partial set; non-transitive markers are
       silently ignored.

     - The wire format of the UPDATE message is unchanged: the total
       path attributes length is preserved, and all attribute
       boundaries remain valid.

     - No new capability negotiation is required.  A BGP speaker MAY
       use ATTR_DISCARD regardless of the peer's capabilities.

     - The propagation behavior of transitive markers through
       non-recognizing speakers is identical to that of any other
       unrecognized optional transitive attribute -- a well-established
       BGP mechanism used by many extensions.

   In mixed deployments where some speakers recognize ATTR_DISCARD
   and others do not:

     - Transitive markers propagate through non-recognizing speakers
       unchanged (except for the Partial bit being set), including
       across AS boundaries.  Non-recognizing speakers forward them
       per RFC 4271 Section 5 with no containment.

     - The originating AS controls propagation at its own EBGP
       boundary: recognizing EBGP speakers clear the Transitive bit
       before forwarding ("inherit" policy) or remove markers
       entirely ("strip" policy).  Non-recognizing EBGP speakers in
       the originating AS forward markers without containment.

     - Non-transitive markers are naturally confined to speakers
       that recognize them (or to the generating AS if IBGP route
       reflectors forward them), regardless of peer capabilities.

   In a fully-recognizing deployment within the originating AS,
   transitive markers are contained at the AS boundary: the AS's
   EBGP speakers clear the Transitive bit before forwarding.
   A recognizing peer will process the marker normally
   (Section 5.2.2); a non-recognizing peer will silently ignore
   it (the marker remains observable via BMP or packet capture).

   When the originating AS cannot guarantee that all its EBGP
   speakers recognize ATTR_DISCARD, the generating speaker should
   clear the Transitive bit at generation time (Section 5.1) or
   operators should use the "strip" policy on those EBGP sessions
   that do support it.

   Non-recognizing speakers will also apply the duplicate-attribute
   rule (Section 5.7), retaining only the first ATTR_DISCARD marker
   if multiple are present.

6.4.  RIB Storage

   ATTR_DISCARD is a wire-level construct.  Implementations SHOULD
   NOT store ATTR_DISCARD markers in the Adj-RIB-In or Loc-RIB.

   Most RIB implementations decompose path attributes into typed
   pools (e.g., separate pools for ORIGIN, AS_PATH, COMMUNITY).
   An ATTR_DISCARD marker does not belong to any typed pool and
   would require special-case storage as raw bytes, defeating the
   efficiency of pool-based architectures.

   The diagnostic value of ATTR_DISCARD is captured at receive time
   through logging and counters (Section 6.1, Section 6.2).  Once
   this information is recorded, the marker serves no further purpose
   in the RIB.

   When forwarding a route whose wire bytes contain ATTR_DISCARD
   markers, the forwarding policy (Section 5.3) determines whether
   the marker is included in the outbound UPDATE.  Implementations
   that build outbound UPDATEs from RIB entries (rather than
   forwarding wire bytes) will naturally omit ATTR_DISCARD since
   it is not stored in the RIB.

6.5.  Route Selection

   ATTR_DISCARD MUST NOT affect the BGP best path selection process.
   It is an informational marker, not a route attribute that
   participates in path comparison.

   Implementations MUST NOT use the presence or absence of
   ATTR_DISCARD markers as a tiebreaker or preference criterion.

6.6.  Route Filtering

   Operators MAY configure route filters that match on the presence
   of ATTR_DISCARD markers.  This allows policies such as:

     - Rejecting routes where an upstream speaker detected a
       malformation (preferring routes with clean attribute sets).

     - Accepting routes with ATTR_DISCARD but logging an alert for
       operator review.

     - Filtering based on the original attribute code (first value
       byte), for example rejecting routes where ORIGIN or AS_PATH
       was discarded (which would indicate treat-as-withdraw was
       not applied when it should have been).

   Such filtering is a local policy decision and not mandated by
   this specification.

6.7.  Interaction with BMP

   Implementations that export UPDATE messages via the BGP Monitoring
   Protocol (BMP) [RFC7854] will include ATTR_DISCARD markers in the
   exported Route Monitoring messages, since BMP preserves the wire
   format of received UPDATEs.  Route monitoring stations and
   analytics tools consuming BMP feeds SHOULD recognize the
   ATTR_DISCARD attribute code to provide accurate diagnostics rather
   than reporting it as an unknown attribute.

   The first value byte of ATTR_DISCARD markers in BMP feeds carries
   the original attribute type code, enabling monitoring tools to
   report which attributes were discarded upstream.  A zero-length
   ATTR_DISCARD (see Section 5.1) will appear in BMP feeds without
   the original type code.

6.8.  Interaction with Extended Messages

   RFC 8654 [RFC8654] allows BGP messages up to 65535 bytes when
   the Extended Message capability has been negotiated.  The
   ATTR_DISCARD mechanism operates on individual attribute headers
   within the path attributes section and is independent of the
   overall message size.  No special handling is required for
   extended messages; the overwrite procedure is identical.

6.9.  Interaction with BGP Role

   RFC 9234 [RFC9234] defines the BGP Role capability and the Only
   To Customer (OTC) attribute for route leak prevention.  OTC
   validation operates independently of ATTR_DISCARD: OTC violations
   require route rejection (treat-as-withdraw per RFC 9234 Section
   6), not attribute discard.  The two mechanisms do not conflict.
   If an UPDATE triggers both a malformed attribute (requiring
   attribute discard) and an OTC violation (requiring treat-as-
   withdraw), the strongest action applies per RFC 7606 Section 3.h,
   resulting in treat-as-withdraw for the entire UPDATE.

7.  Security Considerations

   The value bytes of an ATTR_DISCARD attribute (beyond the first
   byte) contain the original data from the overwritten malformed
   attribute.  Implementations MUST NOT interpret this data.  If an
   implementation logs or displays the raw bytes for diagnostic
   purposes, it SHOULD clearly label them as "discarded, not
   interpreted" to prevent operators from treating the values as
   authoritative.

   Transitive ATTR_DISCARD markers carry the original malformed data
   bytes across the network.  While no conforming implementation will
   interpret these bytes (the type code is unrecognized), the data is
   visible to any speaker that inspects raw attribute bytes.
   Operators who are concerned about propagating potentially malformed
   or sensitive data SHOULD use the "strip" forwarding policy
   (Section 5.3) on external sessions.

   Even under the default "inherit" policy, where the originating AS's
   EBGP speaker clears the Transitive bit before forwarding, the
   opaque value bytes are still carried to the direct EBGP peer.  If the
   discarded attribute contained values that the operator considers
   sensitive (e.g., community values encoding internal policy), these
   bytes are visible to the peer's diagnostic tools despite being
   nominally "not interpreted."  Operators who handle sensitive
   attributes SHOULD consider the "strip" policy on EBGP sessions.

   The "propagate" forwarding policy (Section 5.3c) converts
   non-transitive ATTR_DISCARD markers to transitive, causing opaque
   value bytes from originally non-transitive attributes to propagate
   beyond their intended scope.  Non-transitive attributes such as
   ORIGINATOR_ID and CLUSTER_LIST contain internal topology
   information (router-ids, cluster-ids) that was intended to remain
   within the local AS.  Operators deploying the "propagate" policy
   SHOULD be aware that this may expose such information to external
   peers via the opaque value bytes of the forwarded marker.

   An attacker who can inject BGP UPDATE messages could include
   ATTR_DISCARD attributes:

     - Non-transitive injections are silently dropped by the first
       conforming speaker, limiting impact.

     - Transitive injections propagate like any unrecognized optional
       transitive attribute.  This is no worse than injecting any
       other unrecognized optional transitive attribute, which is
       already possible without this specification.

   The ATTR_DISCARD attribute does not weaken any existing BGP
   security guarantees.  It uses the same propagation mechanics
   (optional transitive forwarding per RFC 4271 Section 5) that have
   been in use since the original BGP-4 specification.

   Implementations that compute cryptographic signatures over UPDATE
   messages (e.g., BGPsec [RFC8205]) MUST apply ATTR_DISCARD AFTER
   signature verification but BEFORE any re-signing or forwarding.
   Modifying attribute bytes before verification would invalidate
   the signature.  Note that applying ATTR_DISCARD to a BGPsec-
   signed UPDATE also invalidates the signature for downstream
   verification, requiring the speaker to re-sign the UPDATE.
   In BGPsec deployments, the zero-copy benefit of ATTR_DISCARD
   is limited to local processing; outbound forwarding requires
   re-signing regardless.  The re-signed UPDATE will include the
   ATTR_DISCARD marker and its opaque value bytes as authenticated
   content; downstream BGPsec receivers will see the tombstone as a
   signed attribute whose value they MUST NOT interpret beyond the
   first byte.

Appendix A.  Design Rationale

A.1.  Why Not Strip and Rebuild?

   The straightforward implementation of attribute-discard is to
   remove the malformed attribute from the path attributes byte
   sequence and update the Total Path Attribute Length field.  This
   requires:

     - Allocating a new buffer (or shifting bytes within the existing
       buffer)
     - Updating the length field (2 bytes at a fixed offset)
     - Invalidating any cached attribute positions
     - Breaking zero-copy optimizations where wire bytes are
       referenced by multiple consumers (iterators, caches, logging)

   High-performance BGP implementations process hundreds of thousands
   of UPDATEs per second.  Buffer reallocation on the hot path creates
   garbage collection pressure (in managed-memory languages) or
   fragmentation (in manual-memory languages).

   The ATTR_DISCARD approach modifies exactly 3 bytes per discarded
   attribute (flags, type code, first value byte).  No allocation,
   no shifting, no cache invalidation.

   Note that the zero-copy benefit applies primarily to the receive
   path and intra-AS forwarding.  The "strip" and "propagate"
   forwarding policies (Section 5.3) may require modifying flags or
   rebuilding path attributes on the outbound path.  BGPsec
   deployments require re-signing regardless (Section 7).

A.2.  Why Not Leave the Original Bytes?

   An alternative is to leave the malformed bytes in place and add
   the attribute code to a per-UPDATE "discard set" that is checked
   by every consumer.  This has several drawbacks:

     - Every code path that reads path attributes must check the
       discard set.  Missing a check silently re-introduces the
       malformed data.

     - The discard set must be threaded through all function calls
       that access the path attributes -- a pervasive API change.

     - Forwarding code that copies wire bytes directly would
       inadvertently forward the malformed attribute unless it
       also checks the discard set.

   The ATTR_DISCARD approach modifies the wire bytes once at the
   point of detection.  All downstream consumers see a clean buffer
   with no malformed attributes.

A.3.  Why a New Attribute Code?

   One could argue that overwriting with any unrecognized code (e.g.,
   0 or a random value) achieves the same effect.  However:

     - A standardized code enables diagnostic tools to distinguish
       "attribute was discarded by RFC 7606 enforcement" from
       "unknown attribute received from peer."

     - Counters and logging can specifically track ATTR_DISCARD
       occurrences, enabling operators to identify malformation
       sources.

     - The first value byte carries the original attribute type code,
       providing diagnostic context without requiring value
       interpretation.  This is only meaningful with a standardized
       code; a random code would not signal that the first byte
       has this defined semantics.

A.4.  Why Inherit Transitivity?

   An earlier version of this design forced ATTR_DISCARD to be always
   non-transitive, ensuring the marker never propagated.  Inheriting
   the original attribute's Transitive bit was chosen instead for
   several reasons:

     - It preserves the original attribute's propagation semantics.
       A transitive attribute was intended to reach all speakers along
       the path; replacing it with a non-transitive marker silently
       changes the UPDATE's propagation characteristics.

     - It provides network-wide diagnostic visibility.  When a
       transitive attribute is discarded, operators at any point
       along the AS path can observe the ATTR_DISCARD marker and
       investigate the malformation source.

     - It enables policy flexibility.  Operators who prefer local-
       only markers can use the "strip" forwarding policy
       (Section 5.3) to remove markers before forwarding.  Operators
       who want maximum visibility can use "propagate" to forward
       all markers.  The default "inherit" behavior is a reasonable
       middle ground.

     - It simplifies the overwrite operation.  When the Transitive
       bit is inherited, only the Optional bit needs to be set and
       the type code overwritten -- the flags byte may not need to
       change at all if the original attribute was already optional
       and transitive.

A.5.  Why Overwrite the First Value Byte?

   The first value byte of the ATTR_DISCARD attribute is overwritten
   with the original attribute's type code.  This adds one byte of
   write overhead (3 bytes total instead of 2) but provides
   significant diagnostic value:

     - Receivers can identify which attribute was discarded without
       relying on out-of-band information.

     - The original type code survives the duplicate-marker
       deduplication described in Section 5.7: even though only one
       ATTR_DISCARD marker survives per UPDATE at the receiver, that
       marker identifies the specific attribute that was discarded.

     - It is safe for the vast majority of attribute-discard cases,
       which involve attributes with at least 1 byte of value data.
       Zero-length attributes (ATOMIC_AGGREGATE) trigger attribute-
       discard only when their length is NOT zero (i.e., malformed),
       guaranteeing at least one value byte to overwrite.  However,
       certain attribute-discard triggers can fire on attributes
       with zero value bytes: LOCAL_PREF from an EBGP peer (RFC 7606
       Section 7.5), ORIGINATOR_ID from an EBGP peer (Section 7.9),
       CLUSTER_LIST from an EBGP peer (Section 7.10), and AGGREGATOR
       with a zero length (Section 7.7).  In these cases, the
       first-value-byte overwrite is skipped (Section 5.1, step 5),
       and the ATTR_DISCARD marker has a zero-length value.  The
       original type code is lost from the wire marker but is
       available in the sender's local log.

   The remaining value bytes (bytes 1 through N-1) are left
   untouched.  Zeroing them would add writes proportional to the
   attribute length, whereas the constant 3-byte overwrite preserves
   the performance characteristics that motivate this mechanism.

   Implementations with strict security requirements (e.g., avoiding
   information leakage through diagnostic outputs) MAY choose to zero
   the remaining value bytes as a local policy decision.
