# RFC 7947 -- Internet Exchange BGP Route Server (short)

## Scope

Specifies the operation of a BGP route server (RS): a BGP speaker at an
Internet Exchange Point (IXP) that exchanges NLRI between participating
routers over a multilateral peering fabric, without itself being a transit
router. Complements RFC 7948 (operational requirements).

## Key Requirements

| Area | Behavior |
|------|----------|
| Forwarding model | Re-advertise routes received from each client to every other client, subject to per-client policy. The RS is not in the forwarding path. |
| AS_PATH | MUST NOT prepend its own AS. Client peerings appear as direct eBGP between clients, not transit via the RS. |
| NEXT_HOP | MUST NOT rewrite. Clients receive the original advertised NEXT_HOP (the advertising client's IP). |
| MED | MUST be preserved across the RS. |
| Client policy | Per-client import/export policy applied on each redistribution. A route may be hidden from some clients and sent to others. |
| Path hiding | A naive RS hides alternative paths (single best-path per client-family). Mitigations: multi-RIB per client or add-path (RFC 7911). RFC 7947 flags path hiding as a known issue, not a protocol mandate. |
| Looping | Because AS_PATH is not prepended, loop detection across the RS relies on client AS numbers present elsewhere in the path. |
| Session model | One eBGP session per client pair (client -- RS). No client-to-client session. |

## Implementation Notes (Ze context)

- bgp-rs is the in-engine plugin implementing the forward-all model.
- AS_PATH non-prepending and NEXT_HOP preservation are enforced by default
  forwarding policy (`selectForwardTargets` + downstream filters).
- Path hiding is not yet mitigated in ze; add-path support exists at the
  session layer but RS path-hiding mitigations (per-client RIBs) are out
  of scope for the rs-fastpath umbrella.
- RFC 7947 places no normative requirement on forwarding throughput; the
  umbrella targets are driven by operational need, not by the RFC.
