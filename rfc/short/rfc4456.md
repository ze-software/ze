# RFC 4456: BGP Route Reflection: An Alternative to Full Mesh Internal BGP (IBGP)

## Meta

| Field | Value |
|-------|-------|
| RFC | 4456 |
| Title | BGP Route Reflection: An Alternative to Full Mesh Internal BGP (IBGP) |
| Status | Standards Track |
| Date | April 2006 |
| Obsoletes | RFC 2796 |
| Purpose | Reduce iBGP full-mesh requirement by allowing Route Reflectors to reflect routes between clients |

## Summary

Route reflection allows a designated Route Reflector (RR) to reflect iBGP routes
between peers, eliminating the need for a full mesh of iBGP sessions. The RR
re-advertises routes received from one iBGP peer to other iBGP peers according
to specific rules.

## Key Concepts

### Cluster (Section 7)

A Route Reflector and its clients form a cluster. Each cluster is identified by
a CLUSTER_ID. Typically, the CLUSTER_ID is set to the BGP Identifier (Router ID)
of the RR. When a single RR exists in a cluster, the CLUSTER_ID defaults to the
RR's BGP Identifier.

### ORIGINATOR_ID (Section 8, Type Code 9)

Optional, non-transitive attribute (4 bytes). Created by the RR when reflecting
a route. Set to the BGP Identifier of the originator of the route in the local AS.
Must not be created by a speaker that did not originate the route. The value is
preserved unchanged through the reflection chain.

### CLUSTER_LIST (Section 8, Type Code 10)

Optional, non-transitive attribute. A sequence of CLUSTER_ID values (4 bytes each)
representing the reflection path the route has traversed. When an RR reflects a
route, it prepends its local CLUSTER_ID to the CLUSTER_LIST. If no CLUSTER_LIST
exists, it creates one.

## Loop Detection Rules (Section 8)

### ORIGINATOR_ID Loop Detection

A router that recognizes the ORIGINATOR_ID attribute SHOULD ignore a route
received with its BGP Identifier as the ORIGINATOR_ID. This prevents routes
from being reflected back to their originator.

### CLUSTER_LIST Loop Detection

If the local CLUSTER_ID is found in the CLUSTER_LIST, the advertisement
received SHOULD be ignored. This prevents routing loops within a cluster
when multiple RRs exist.

## Route Reflection Rules (Section 8)

| Source | Destination | Action |
|--------|-------------|--------|
| Non-Client peer | Client peers | Reflect (modify ORIGINATOR_ID, CLUSTER_LIST) |
| Non-Client peer | Non-Client peers | Do NOT reflect |
| Client peer | Non-Client peers | Reflect (modify ORIGINATOR_ID, CLUSTER_LIST) |
| Client peer | Other Client peers | Reflect (modify ORIGINATOR_ID, CLUSTER_LIST) |
| eBGP peer | All iBGP peers | Normal iBGP route distribution |

## Ze Implementation Notes

- Ze uses Router ID as default Cluster ID (RFC 4456 Section 7 default)
- ORIGINATOR_ID (type 9) parsed by `attribute.ParseOriginatorID` (returns netip.Addr)
- CLUSTER_LIST (type 10) parsed by `attribute.ParseClusterList` (returns []uint32)
- Loop detection runs in session validation (session_validation.go), before plugin dispatch
- Treat-as-withdraw on loop detection (route silently withdrawn, session stays up)
