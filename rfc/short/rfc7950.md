# RFC 7950: The YANG 1.1 Data Modeling Language

## Meta

| Field | Value |
|-------|-------|
| RFC | 7950 |
| Title | The YANG 1.1 Data Modeling Language |
| Status | Standards Track (Proposed Standard) |
| Date | August 2016 |
| Author | M. Bjorklund (Tail-f Systems) |
| Obsoletes | - |
| Updates | - |
| Updated-by | RFC 8342, RFC 8526 |

**Purpose:** Defines YANG 1.1, a data modeling language for configuration data, state data, RPCs, and notifications. Used primarily with NETCONF/RESTCONF but applicable to any management protocol.

**Scope:** Data modeling language — defines structure, types, constraints, and extensions for network management data. Ze uses YANG for config schema definition and validation.

**Ze usage:** Config schema (`ze-bgp-conf.yang`, `ze-types.yang`, `ze-extensions.yang`), type validation (range, pattern, enum, length), custom extensions (`ze:syntax`, `ze:key-type`). Ze uses `goyang` library (not `libyang`) for parsing — no XPath evaluation.

| Type | Description | Ze Config Usage |
|------|-------------|-----------------|

| `int8`..`int64` | Signed integers | Not commonly used |
| `uint8`..`uint64` | Unsigned integers | `hold-time` (uint16), `local-as` (uint32) |
| `decimal64` | Fixed-point decimal | Not used |
| `enumeration` | Named string constants | `origin` (igp/egp/incomplete), `connection` (both/passive/active) |
| `bits` | Named bit flags | Not used |
| `binary` | Base64-encoded data | Not used |
| `leafref` | Reference to another leaf | Not used (Ze resolves refs in Go code) |
| `identityref` | Reference to identity | Not used |
| `union` | Multiple type options | `ip-address` (IPv4 or IPv6 union) |
| `empty` | Presence-only, no value | Presence containers (e.g., `atomic-aggregate`) |
| `instance-identifier` | Path to data node | Not used |

### Type Restrictions (Section 9)

These are the validation constraints YANG can express natively on types.

| Restriction | Applies To | Syntax | Ze Enforcement |
|-------------|-----------|--------|----------------|
| `range` | Integer, decimal64 | `range "0 \| 3..65535"` | `yang.Validator` checks via `checkYangRange` |
| `length` | String, binary | `length "1..255"` | `yang.Validator` checks string length |
| `pattern` | String | `pattern '[0-9]+:[0-9]+'` | `yang.Validator` checks via `regexp.MatchString` |
| `pattern` + `modifier` | String (1.1 only) | `modifier invert-match` | Not used in Ze |
| `fraction-digits` | decimal64 | `fraction-digits 2` | Not used |

**Range syntax:** `"min..max"` with `min`/`max` keywords for type limits. Pipe `|` separates disjoint ranges. Example: `range "0 | 3..65535"` means 0 or 3-65535.

**Pattern syntax:** XSD regular expressions (similar to POSIX ERE). Anchored implicitly (full match). Multiple patterns on same type are AND-combined.

**Length syntax:** Same range format as `range` but applied to string byte length.

### Derived Types (typedef)

Typedefs create named types with optional restrictions. Ze uses these extensively in `ze-types.yang`:

| Ze Typedef | Base | Restriction |
|-----------|------|-------------|
| `asn` | uint32 | range "1..4294967295" |
| `asn2` | uint16 | range "1..65535" |
| `port` | uint16 | range "1..65535" |
| `ipv4-address` | string | pattern (dotted decimal) |
| `ipv6-address` | string | pattern (colon hex) |
| `ip-address` | union | ipv4-address or ipv6-address |
| `community` | string | pattern `[0-9]+:[0-9]+` |
| `large-community` | string | pattern `[0-9]+:[0-9]+:[0-9]+` |
| `extended-community` | string | pattern `(target\|origin):[0-9]+:[0-9]+` |
| `route-distinguisher` | string | pattern (Type 0 or Type 1) |

## Constraint Mechanisms

### Leaf/Container Constraints

| Constraint | Section | Purpose | Ze Usage |
|-----------|---------|---------|----------|
| `mandatory` | 7.6.5 | Requires node presence | `peer-as mandatory true`, `local-as mandatory true` |
| `default` | 7.6.4 | Default value when absent | `hold-time default 90` |
| `min-elements` | 7.7.5 | Minimum list/leaf-list entries | Not used |
| `max-elements` | 7.7.6 | Maximum list/leaf-list entries | Not used |
| `unique` | 7.8.3 | Unique leaf combinations in list | Not used |
| `ordered-by` | 7.7.7 | User vs system ordering | Not used |
| `presence` | 7.5.1 | Container carries meaning by existing | `atomic-aggregate`, `withdraw` |

### XPath-Based Constraints

| Constraint | Section | Purpose | Ze Usage |
|-----------|---------|---------|----------|
| `must` | 7.5.3 | XPath boolean condition on data | Not used (goyang does not evaluate XPath) |
| `when` | 7.21.5 | Conditional node applicability | Not used (goyang does not evaluate XPath) |

**must statement:** Defines constraints as XPath 1.0 expressions evaluated against the data tree. When the expression evaluates to false, the data is invalid. Supports `error-message` and `error-app-tag` substatements for error reporting.

**when statement:** Controls whether a node is valid based on other data. If the XPath expression is false, the node must not exist. Used for conditional augmentations and conditional leaves.

**Ze alternative:** Ze uses `ze:validate` extension to reference registered Go functions for constraints that would require `must`/`when` in standard YANG.

### Conditional Features

| Construct | Section | Purpose | Ze Usage |
|-----------|---------|---------|----------|
| `feature` | 7.20.1 | Declares optional capability | Not used |
| `if-feature` | 7.20.2 | Gates nodes on feature support | Not used |

YANG 1.1 extends `if-feature` to boolean expressions with AND, OR, NOT operators. Ze does not use features — plugin YANG is either loaded or not.

## Extension Mechanism (Section 7.19)

Extensions define custom statements that add semantics beyond standard YANG. The `extension` statement declares new keywords with optional arguments.

### Extension Definition

An extension is defined at module level with:
- `argument` — defines the parameter the extension takes
- `description` — human-readable purpose

### Extension Usage

Extensions are referenced via prefix in any YANG module that imports the defining module. Other YANG tools can parse modules containing extensions (they just don't execute them).

### Ze Extensions (ze-extensions.yang)

| Extension | Argument | Purpose |
|-----------|----------|---------|
| `ze:syntax` | mode (flex/freeform/inline-list/etc.) | Controls config parser behavior for special syntax |
| `ze:key-type` | type (string/prefix/ip/uint32) | Specifies key type for inline-list nodes |
| `ze:route-attributes` | (none) | Marks node as accepting standard BGP route attributes |
| `ze:allow-unknown-fields` | (none) | Container accepts arbitrary key-value pairs |
| `ze:validate` | function-name | References registered Go validation function (planned) |

**Key property:** Extensions are valid YANG — standard tools (pyang, libyang) parse Ze's schemas without error. They simply ignore the `ze:` extensions they don't understand.

## Enumeration Type (Section 9.6)

Enumerations restrict a leaf to a fixed set of named string values. Each enum value has an associated integer position (auto-assigned if omitted).

### YANG 1.1 Changes

- Enumerations can be subtyped (restrict enum set in derived type)
- `if-feature` allowed on individual enum values
- Position values preserved across subtyping

### Ze Usage

| Location | Enum Values |
|----------|-------------|
| `origin` attribute | igp, egp, incomplete |
| `connection` mode | both, passive, active |
| `role` type | provider, rs, rs-client, customer, peer |
| `encoding-mode` | text, hex, b64 |
| `wire-format` | hex, base64, parsed, full |
| `ack-mode` | sync, async |
| `afi` | ipv4, ipv6 |
| `safi` | unicast, multicast, mpls, mpls-vpn, flowspec |

## Key Changes from YANG 1.0 (RFC 6020)

### Relevant to Ze

| Change | Impact |
|--------|--------|
| `yang-version 1.1` mandatory | Ze modules must declare version |
| Enum/bits subtyping | Can restrict enum sets in derived types |
| Pattern modifier `invert-match` | Negative patterns available (not currently used) |
| `if-feature` on enum/bit values | Conditional enum values (not currently used) |
| Leaf-list defaults | Default values on leaf-lists now supported |
| `action` statement | Operations on data nodes (not used) |
| `must` in input/output | Constraints on RPC args (not used — no XPath eval) |
| `leafref require-instance` | Can relax reference validation (not used) |
| Multiple identity bases | Identity inheritance (not used) |

### Not Relevant to Ze

| Change | Why Not |
|--------|---------|
| `anydata` statement | Ze uses typed schemas, not arbitrary data |
| `empty` in unions | Ze doesn't use empty type in unions |
| Notification on data nodes | Ze uses IPC events, not YANG notifications |
| Module advertising via yang-library | Ze is not a NETCONF server |

## Error Handling

### YANG Validation Errors

| Error Type | Trigger | YANG Response |
|-----------|---------|---------------|
| Range violation | Value outside `range` bounds | `invalid-value` error-tag |
| Length violation | String outside `length` bounds | `invalid-value` error-tag |
| Pattern violation | String doesn't match `pattern` | `invalid-value` error-tag |
| Enum violation | Value not in enumeration | `invalid-value` error-tag |
| Mandatory missing | Required node absent | `missing-element` error-tag |
| Must constraint fail | XPath evaluates to false | `must-violation` error-tag |
| Unique violation | Duplicate unique key combo | `data-not-unique` error-tag |

### Ze Error Mapping

Ze maps YANG errors to `ValidationError` types in `internal/yang/validator.go`:

| YANG Error | Ze ErrorType |
|-----------|-------------|
| Range violation | `ErrTypeRange` |
| Length violation | `ErrTypeLength` |
| Pattern violation | `ErrTypePattern` |
| Enum violation | `ErrTypeEnum` |
| Mandatory missing | `ErrTypeMissing` |
| Type mismatch | `ErrTypeType` |

## XPath Functions (Section 10)

YANG 1.1 adds these XPath functions for use in `must`/`when` expressions:

| Function | Purpose |
|----------|---------|
| `current()` | Returns context node in must/when evaluation |
| `deref(node)` | Follows leafref/instance-identifier to target |
| `derived-from(identity, base)` | Tests identity inheritance |
| `derived-from-or-self(identity, base)` | Tests identity match or inheritance |
| `enum-value(node)` | Returns numeric position of enum value |
| `bit-is-set(bits, name)` | Tests if named bit is set |

**Ze note:** These functions are not available in Ze because goyang does not include an XPath evaluator. Ze uses `ze:validate` Go functions instead.

## MUST Requirements (Ze-Relevant)

### Type Validation

- "If a leaf data value does not match the type constraints for the leaf, including those defined in the type's range, length, and pattern properties, the server MUST reply with an invalid-value error" (Section 8.3.1)
- "An enumeration value MUST be one of the values specified in the type's enum statements" (Section 9.6)
- "A union value MUST match at least one member type" (Section 9.12)

### Mandatory Fields

- "If a mandatory node does not exist, the server MUST reply with a missing-element error" (Section 7.6.5)
- "If a leaf has a default value and the leaf is not set, the default value is used" (Section 7.6.4)

### Constraints

- "If a must expression evaluates to false, the data is not valid" (Section 7.5.3)
- "All range, length, and pattern restrictions MUST be more restrictive than or equal to the base type's restrictions" (Section 9.2.4)
- "Multiple pattern statements are combined as logical AND" (Section 9.4.5)

## Pitfalls

### For Ze Specifically

- **goyang limitations:** goyang parses YANG schemas and builds entry trees but does not evaluate XPath expressions. `must`/`when` constraints are parsed but not enforced — Ze must implement equivalent checks in Go code.
- **Pattern anchoring:** YANG patterns are implicitly anchored (full match). Ze's validator correctly adds `^` and `$` anchors via `regexp.MatchString("^"+p+"$", str)`.
- **Union type ordering:** Union member types are tried in declaration order. First match wins. Order matters when types overlap.
- **Typedef restriction inheritance:** Restrictions on typedefs are inherited by all uses. A `ze:validate` extension on a typedef should also propagate.
- **Augment from plugins:** Plugin YANG modules augment the base schema (e.g., `ze-graceful-restart.yang` augments `peer/capability`). Augmented nodes are only available after all modules are loaded and resolved.
- **Enum position stability:** Enum integer positions must not change across revisions. Adding new enum values should use explicit `value` statements to prevent position shifts.
