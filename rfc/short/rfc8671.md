# RFC 8671 -- Support for Adj-RIB-Out in BGP Monitoring Protocol (BMP)

## Purpose

Extends BMP (RFC 7854) to support monitoring of Adj-RIB-Out (routes
advertised to peers) in addition to the original Adj-RIB-In (routes
received from peers). Enables collectors to see both directions of
a BGP session.

## Changes to RFC 7854

### O Flag (Per-Peer Header, bit 4)

| Value | Meaning |
|-------|---------|
| 0 | Adj-RIB-In (received routes, original BMP behavior) |
| 1 | Adj-RIB-Out (sent routes, post-policy) |

When O=1, the L flag (bit 6, post-policy) is also set because
Adj-RIB-Out represents routes after export policy.

### Affected Message Types

- **Route Monitoring (type 0)**: carries Adj-RIB-Out UPDATEs when O=1
- **Peer Up (type 3)**: same OPENs for both Adj-RIB-In and Adj-RIB-Out
- **Peer Down (type 2)**: signals loss of Adj-RIB-Out monitoring
- **Statistics Report (type 1)**: O flag identifies which RIB the stats cover

### Peer Up Behavior

A single BGP session may produce two Peer Up messages: one for
Adj-RIB-In (O=0) and one for Adj-RIB-Out (O=1). Both carry the
same sent and received OPEN messages.

## Key Constraints

- O flag uses bit 4 of the Flags field (bits numbered 7 to 0)
- Pre-policy Adj-RIB-Out (L=0, O=1) is valid but uncommon
- Peer Down for Adj-RIB-Out does not imply BGP session loss

## Ze Implementation

- O flag constant: `PeerFlagO` in `internal/component/bgp/plugins/bmp/header.go`
- Sender sets O=1 and L=1 for `direction=sent` events
- Receiver decodes O flag in all per-peer header messages
