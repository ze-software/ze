# RFC 4761 - Virtual Private LAN Service (VPLS) Using BGP for Auto-Discovery and Signaling

## Meta

| Field | Value |
|-------|-------|
| RFC | 4761 |
| Title | Virtual Private LAN Service (VPLS) Using BGP for Auto-Discovery and Signaling |
| Status | Proposed Standard |
| Date | January 2007 |
| Obsoletes | - |
| Updates | - |
| Updated by | RFC 5462, RFC 8395, RFC 8614 |

**Purpose:** Defines BGP-based auto-discovery and signaling mechanisms for Virtual Private LAN Service (VPLS), enabling multipoint Layer 2 VPN connectivity where customers appear connected via a single Ethernet LAN.

**Scope:** AFI = 25 (L2VPN), SAFI = 65 (VPLS)

## Wire Formats

### VPLS BGP NLRI

AFI: 25 (L2VPN)
SAFI: 65 (VPLS)

```
+------------------------------------+
|  Length (2 octets)                 |
+------------------------------------+
|  Route Distinguisher  (8 octets)   |
+------------------------------------+
|  VE ID (2 octets)                  |
+------------------------------------+
|  VE Block Offset (2 octets)        |
+------------------------------------+
|  VE Block Size (2 octets)          |
+------------------------------------+
|  Label Base (3 octets)             |
+------------------------------------+
```

| Field | Offset | Size | Type | Constraints |
|-------|--------|------|------|-------------|
| Length | 0 | 2 | uint16 | Total NLRI length in octets (17 for base format) |
| Route Distinguisher | 2 | 8 | RD | Per RFC 4364 |
| VE ID | 10 | 2 | uint16 | Locally unique within VPLS; same for multi-homed site |
| VE Block Offset | 12 | 2 | uint16 | Starting VE ID for this label block |
| VE Block Size | 14 | 2 | uint16 | Number of VE IDs covered by this label block |
| Label Base | 16 | 3 | MPLS label | 20-bit label in high-order bits |

**Length calculation:** 17 octets for base NLRI format (2+8+2+2+2+3 = 19, but Length field value is 17 as it excludes itself).

**Label Block:** Defines set {LB+VBO, LB+VBO+1, ..., LB+VBO+VBS-1} where:
- LB = Label Base
- VBO = VE Block Offset
- VBS = VE Block Size

**Remote VE Set:** VE IDs from VBO to (VBO + VBS - 1).

**Label correspondence:** VE ID (VBO + n) corresponds to label (LB + n).

### Layer2 Info Extended Community

Extended Community Type: 0x800A

```
+------------------------------------+
| Extended community type (2 octets) |
+------------------------------------+
|  Encaps Type (1 octet)             |
+------------------------------------+
|  Control Flags (1 octet)           |
+------------------------------------+
|  Layer-2 MTU (2 octets)            |
+------------------------------------+
|  Reserved (2 octets)               |
+------------------------------------+
```

| Field | Offset | Size | Type | Constraints |
|-------|--------|------|------|-------------|
| Extended community type | 0 | 2 | uint16 | 0x800A |
| Encaps Type | 2 | 1 | uint8 | 19 for VPLS |
| Control Flags | 3 | 1 | uint8 | See Control Flags format |
| Layer-2 MTU | 4 | 2 | uint16 | MTU for pseudowires |
| Reserved | 6 | 2 | - | MUST be zero |

### Control Flags

```
 0 1 2 3 4 5 6 7
+-+-+-+-+-+-+-+-+
|   MBZ     |C|S|      (MBZ = MUST Be Zero)
+-+-+-+-+-+-+-+-+
```

| Bit | Name | Meaning |
|-----|------|---------|
| 0-5 | MBZ | MUST be zero when sending, ignored when receiving |
| 6 | C | Control word: 1 = MUST use, 0 = MUST NOT use |
| 7 | S | Sequenced delivery: 1 = MUST use, 0 = MUST NOT use |

## Encoding Rules

- All multi-byte fields are network byte order (big-endian)
- Label Base uses standard 3-octet MPLS label encoding (20-bit label in high-order bits)
- Route Target extended community identifies VPLS membership
- Single Route Target suffices for fully-meshed VPLS
- BGP Next Hop is the advertising PE's address (IPv4 = 4 octets, IPv6 = 16 octets)

## Decoding Rules

1. Parse Length field (2 octets)
2. Parse Route Distinguisher (8 octets)
3. Parse VE ID, VE Block Offset, VE Block Size, Label Base
4. Associate with Route Target from extended communities
5. Determine if local VE ID falls within remote VE set [VBO, VBO+VBS-1]
6. If yes, compute demultiplexor label as (LB + local_VE_ID - VBO)

## Validation

| Check | Valid | Invalid Action |
|-------|-------|----------------|
| VE ID in remote VE set | VBO <= local_VE_ID < VBO + VBS | Ignore NLRI for PW setup |
| Route Target match | RT matches configured VPLS | Do not import NLRI |
| MBZ bits in Control Flags | All zero | Ignore on receive (MUST be zero on send) |
| Same VE ID multi-homing | Same RD for both PEs | Path selection error if RD differs |
| Encaps Type for VPLS | 19 | - |

## MUST Requirements

### Tx (Sender)

- "an implementation MUST maintain a separate routing storage for each service" (Section 2.3)
- "PE-a MUST withdraw all its announcements for VPLS foo that contain VE ID V" when V is removed from configuration (Section 3.3)
- "MBZ... MUST be set to zero when sending" Control Flags (Section 3.2.4)
- Inter-AS Method (b): "the Length, the Route Distinguisher, the VE ID, the VE Block Offset, and the VE Block Size MUST be the same" when ASBR re-advertises (Section 3.4.2)

### Rx (Receiver)

- "MBZ... MUST be ignored when receiving" Control Flags (Section 3.2.4)
- "If two PEs are assigned the same VE ID in a given VPLS, they MUST use the same Route Distinguisher" (Section 3.5)
- "it MUST pick only one" when receiving equivalent NLRIs (Section 3.5)

### State

- "PE-b MUST make a new announcement" if announcing PE's VE ID not in any remote VE set PE-b announced (Section 3.2.3)
- "X MUST tear down its ends of the pseudowire between X and Y" when Y withdraws NLRI (Section 3.2.3)
- "the VE MUST update its FIB to reflect the new port P'" when MAC address moves (Section 4.2.1)
- "S MUST be flushed from the FIB" if age exceeds aging time T (Section 4.2.2)

### Forwarding

- "PEx MUST NOT send the frame to other PEs" for split horizon (Section 4.2.5)

### Security

- "Any implementation of VPLS that allows VPLS packets to be tunneled as described in that document MUST contain an implementation of IPsec" for MPLS-in-IP/GRE tunnels (Section 6)
- "MUST allow the egress PE to validate... the IP source address of any tunneled packet" if not using IPsec (Section 6)

## SHOULD/MAY Requirements

- [SHOULD] "PE-a SHOULD either withdraw all its NLRIs for VPLS foo or let other PEs in the VPLS foo know in some way that PE-a is no longer connected to its CEs" when CE links go down (Section 3.3)
- [SHOULD] "they SHOULD announce the same VE Block Size for a given VE Offset" for multi-homed VE IDs (Section 3.5)
- [SHOULD] "VPLS PEs SHOULD have an aging mechanism to remove a MAC address" (Section 4.2.2)
- [SHOULD] "An implementation SHOULD provide a configurable knob to set the aging time T on a per-VPLS basis" (Section 4.2.2)
- [SHOULD] "To be useful, an implementation SHOULD allow this mapping function [802.1p to EXP] to be different for each VPLS" (Section 4.2.7)
- [MAY] "A VE MAY implement a mechanism to damp flapping of source ports for a given MAC address" (Section 4.2.1)
- [MAY] "an implementation MAY accelerate aging of all MAC addresses in a VPLS if it detects certain situations, such as a Spanning Tree topology change" (Section 4.2.2)
- [MAY] "a VE MAY also use certain techniques to restrict transmission of multicast frames to a smaller set of receivers" (Section 4.2.4)
- [MAY] "an implementation MAY choose to map 802.1p bits in a customer Ethernet frame with a VLAN tag to an appropriate setting of EXP bits" (Section 4.2.7)

## Error Handling

| Condition | Detect How | Response | Code/Subcode |
|-----------|------------|----------|--------------|
| Local VE ID not in remote VE set | VBO > local_VE_ID OR local_VE_ID >= VBO + VBS | Ignore NLRI, skip PW setup | N/A |
| Route Target mismatch | RT not in import policy | Do not import NLRI | N/A |
| NLRI withdrawal received | BGP WITHDRAW message | Tear down pseudowire ends | N/A |
| Malformed NLRI | Length check | UPDATE Message Error | 3/1 |
| Equivalent NLRIs from multiple PEs | Same (RD, VE ID, VBO) | BGP path selection, pick one | N/A |

## Algorithms

### Pseudowire Setup (Section 3.2.3)

```
On receiving VPLS NLRI from PE-a with (VE_ID=V, VBO, VBS, LB):
    local_VE_ID = W (configured for this VPLS)

    # Step 1: Check if local VE ID in remote VE set
    if NOT (VBO <= W < VBO + VBS):
        return  # Ignore, W not covered

    # Step 2: Set up PW to PE-a
    demux_label_to_PE_a = LB + W - VBO

    # Step 3: Check if PE-a's VE ID covered by our announcements
    for each (VBO', VBS', LB') announced by us:
        if VBO' <= V < VBO' + VBS':
            # V is covered, PE-a can set up PW to us
            demux_label_from_PE_a = LB' + V - VBO'
            return

    # Step 4: V not covered, MUST announce new label block
    announce_new_label_block_covering(V)
```

### Label Block Allocation

```
To cover remote VE IDs {V1, V2, ..., Vn}:
    Option A: Single large block
        VBO = min(V1..Vn)
        VBS = max(V1..Vn) - min(V1..Vn) + 1
        LB = allocate_labels(VBS)
        announce(VE_ID, VBO, VBS, LB)

    Option B: Multiple smaller blocks
        for each contiguous range [start, end]:
            VBO = start
            VBS = end - start + 1
            LB = allocate_labels(VBS)
            announce(VE_ID, VBO, VBS, LB)
```

### Multi-homed Path Selection (Section 3.5)

```
NLRIs are equivalent if:
    (NLRI_A.RD == NLRI_B.RD) AND
    (NLRI_A.VE_ID == NLRI_B.VE_ID) AND
    (NLRI_A.VBO == NLRI_B.VBO)

On receiving equivalent NLRIs:
    Apply standard BGP path selection:
        1. Highest Local Preference
        2. Shortest AS Path
        3. ... (standard BGP tie-breakers)
    Select exactly ONE NLRI
    Use selected NLRI for PW setup
```

### Split Horizon Forwarding (Section 4.2.5)

```
On receiving frame at PE:
    if frame.source == attached_CE:
        # Frame from customer
        flood_to_all_other_CEs()
        flood_to_all_other_PEs()
    else:
        # Frame from another PE (via pseudowire)
        flood_to_attached_CEs_only()
        # DO NOT flood to other PEs (split horizon)
```

### MAC Learning and Aging (Sections 4.2.1, 4.2.2)

```
On receiving frame with src_MAC on logical_port:
    if FIB[src_MAC] exists:
        if FIB[src_MAC].port != logical_port:
            FIB[src_MAC].port = logical_port  # MAC moved
    else:
        FIB[src_MAC] = {port: logical_port, age: 0}
    FIB[src_MAC].last_seen = now()

Periodically:
    for each (MAC, entry) in FIB:
        if (now() - entry.last_seen) > aging_time:
            delete FIB[MAC]
```

## Multi-AS VPLS Methods

### Method (a): VPLS-to-VPLS at ASBRs (Section 3.4.1)

- ASBRs act as PEs for inter-AS VPLSs
- Neighboring ASBR treated as CE
- Requires Ethernet interconnect with separate VLAN per VPLS
- Full PE operations (discovery, signaling, MAC learning, flooding) on ASBRs
- Requires STP or loop detection per VPLS

### Method (b): EBGP Redistribution between ASBRs (Section 3.4.2)

- I-BGP: PE1 <-> ASBR1, E-BGP: ASBR1 <-> ASBR2, I-BGP: ASBR2 <-> PE2
- ASBRs re-advertise NLRI with new labels and next-hop self
- Three tunnels: T1 (PE1->ASBR1), T2 (ASBR1->ASBR2), T3 (ASBR2->PE2)
- ASBRs perform label swap operations
- Requires MPLS on inter-AS link (not necessarily Ethernet)
- Loop prevention via BGP path selection

**ASBR Label Swap (Section 3.4.2):**
```
Given:
    PE1 sends Label Base L1, block size N
    ASBR1 sends Label Base L2 to ASBR2
    Tunnel label from ASBR1 to PE1 = T

ASBR1 installs:
    swap L2 with L1 and push T
    swap L2+1 with L1+1 and push T
    ...
    swap L2+N-1 with L1+N-1 and push T
```

### Method (c): Multi-Hop EBGP (Section 3.4.3)

- Multi-hop E-BGP between PEs or Route Reflectors across ASes
- BGP Next Hop unchanged (PE address)
- Requires tunnel LSP from PE1 to PE2 (e.g., via labeled IPv4 routes)
- No VPLS state on ASBRs (control or data plane)
- Most scalable method

## Pitfalls

### Edge Cases

- **VE ID = 0:** Valid value; no special semantics.
- **Empty label block:** VBS = 0 is degenerate; no VE IDs covered.
- **Overlapping label blocks:** A PE may advertise multiple label blocks with different offsets covering different VE ID ranges.
- **VE ID outside all blocks:** Remote PE cannot set up PW; announcing PE must add new label block.
- **BGP startup timing:** PE may wait for several advertisements before allocating label blocks for efficiency.
- **CE link flap:** Should withdraw NLRIs or signal unavailability to prevent blackholing.

### Interop Considerations

- **LDP VPLS (RFC 4762):** Distinct and incompatible; cannot interoperate directly. IESG Note explicitly warns about this.
- **Route Reflectors:** Purely control plane; no data plane state. Multiple RR hierarchies possible for scaling.
- **Route Target Filtering (RFC 4684):** Recommended for limiting VPLS message propagation to interested PEs.
- **Control Word:** Both endpoints must agree (via C flag). Mismatch causes PW to not form correctly.
- **MTU mismatch:** Layer-2 MTU should match across VPLS; mismatch may cause fragmentation or drops.

### Security

- **No confidentiality:** VPLS packets sent in clear; use IPsec tunnels if encryption needed.
- **Control plane attacks:** BGP session spoofing could redirect traffic; use TCP MD5 (RFC 2385).
- **Data plane attacks:** Accept VPLS labels only from valid interfaces (P routers, authorized ASBRs).
- **Misconfiguration:** Wrong Route Target can connect unintended CEs; careful RT assignment required.
- **MAC flooding:** Malicious MAC address flooding can exhaust FIB; consider MAC learning limits.

## Compatibility

### With Non-Supporting Peers

- BGP peers not supporting AFI 25/SAFI 65 will ignore VPLS NLRIs
- Standard capability negotiation (RFC 4760) applies
- No protocol errors from AFI/SAFI mismatch

### Feature Negotiation Fallback

- Control Word (C flag): If endpoints disagree, PW does not form correctly
- Sequenced delivery (S flag): Optional; can be negotiated per endpoint
- MTU: Should match; RFC 4448 defines encapsulation behavior
