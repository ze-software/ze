# RFC 5883 - BFD for Multihop Paths

## Meta

| Field | Value |
|-------|-------|
| RFC | 5883 |
| Title | Bidirectional Forwarding Detection (BFD) for Multihop Paths |
| Status | Standards Track |
| Date | June 2010 |
| Authors | D. Katz, D. Ward (Juniper Networks) |
| Base | RFC 5880, RFC 5881 |

**Purpose:** Specifies how BFD is carried across arbitrary, multi-hop IP paths. Addresses the three differences from single-hop: GTSM no longer applies, session demultiplexing cannot rely on the ingress interface, and the Echo function is unusable.

**Scope:** Any IPv4 or IPv6 path that crosses more than one routing hop, including iBGP loopback peerings, VPN overlays, and routed tunnels.

## Wire Format

### Encapsulation

| Role | UDP Destination Port | Note |
|------|---------------------:|------|
| Control | 4784 | Distinct from single-hop 3784 to allow coexistence |
| Echo | -- | Echo MUST NOT be used over multihop paths |

All other encapsulation details inherit from RFC 5881: Control packets are UDP over IPv4 or IPv6, source ports are chosen from the ephemeral range, the BFD payload is the RFC 5880 mandatory section with optional authentication.

### TTL / Hop Limit

GTSM (TTL=255 on transmit and receive) does NOT apply. Implementations MAY enforce a configurable minimum TTL on receive as a weaker spoofing defence but the base spec does not mandate a specific value.

## States and Transitions

RFC 5883 adds no states. All FSM rules are inherited from RFC 5880.

## Demultiplexing

The core multihop problem: `Your Discriminator == 0` packets cannot be demultiplexed by ingress interface (the interface is not part of the session identity). RFC 5883 Section 4 lists three approaches.

### Approach 1: Single session per address pair (Section 4.1)

Limit BFD to at most one session per `(local address, peer address)` tuple. Multiple sessions between the same system pair MUST use distinct address pairs (e.g., different loopbacks).

Initial packets with Your Discriminator = 0 are demultiplexed by the source/destination address pair alone. After discriminators are exchanged, all packets demultiplex by Your Discriminator as usual.

Suitable for: general multihop liveness, OSPF virtual links, iBGP peerings between distinct loopbacks.

### Approach 2: Out-of-band discriminator signalling (Section 4.2)

Discriminators are exchanged through a side channel (control plane protocol, configuration) before the first BFD packet. The first packet therefore already carries the correct Your Discriminator and no zero-discriminator ambiguity exists.

Used by: BFD for MPLS LSPs (RFC 5884).

Trade-off: requires BFD-aware logic outside the BFD implementation itself.

### Approach 3: Unidirectional links (Section 4.3)

A unidirectional link is modelled as multihop because the return path may be arbitrary. Roles are enforced:

- Unidirectional Sender: MUST use the Active role (transmits from session creation).
- Unidirectional Receiver: MUST use the Passive role (transmits nothing until it learns the remote discriminator from the first received Control packet).

The receiver demultiplexes the first received packet via the physical/logical link over which it arrived, then echoes the learned discriminator back over the arbitrary return path. After that, both sides demultiplex by Your Discriminator alone.

### Echo Function (Prohibited)

Echo packets MUST NOT be used over multihop paths. An intermediate router would simply forward the Echo packet back toward its source via normal routing, so the returning packet does not prove end-to-end liveness.

## Algorithms

### Initial Packet Demultiplexing (Single-Session-Per-Pair)

```
on receive Control packet with Your Discriminator == 0:
    session = lookup by (dst_addr, src_addr) in this VRF
    if session == nil:
        discard (or create, per local policy)
    otherwise:
        associate packet with session
        apply RFC 5880 reception procedure
```

### Active/Passive Selection

Single-hop: both sides Active.
Multihop unidirectional: Sender Active, Receiver Passive.
Multihop bidirectional with OOB signalling: either arrangement works because discriminators are pre-exchanged.

## Pitfalls

### Edge Cases

- **No GTSM:** a multihop session can be spoofed from anywhere with IP reachability to the receiver. Cryptographic authentication becomes the primary defence.
- **Route changes:** the source address and/or outgoing interface MAY change if the routing table does. Sessions keyed purely on Your Discriminator survive this; sessions keyed on the address tuple may break. Subscribe to next-hop tracking on the consumer side if aggressive timers are in use.
- **Multiple paths between the same loopback pair:** RFC 5883 does not address ECMP. Only one path is tracked; a different path may fail without detection.
- **Coexistence with single-hop:** distinct UDP ports (3784 vs 4784) prevent accidental session blending. Implementations MUST bind RX sockets to the correct port per session type.
- **Echo is unusable:** any multihop implementation MUST reject an attempt to enable echo on a multihop session.

### Interop Considerations

- **Congestion control (BCP):** RFC 5880 Operational Considerations require congestion control for multihop deployments. Implementations MUST reduce TX rate when congestion is detected; exact mechanism is out of scope.
- **Authentication strength:** RFC 5883 Section 6 SHOULD cryptographic authentication for all multihop sessions. This is the standard interoperability expectation.

### Security

- **Spoofing surface:** increases with hop count. Any system between the endpoints can forge packets.
- **Authentication:** SHOULD be used. Keyed SHA1 or stronger is recommended. The sequence-number persistence concern from RFC 5880 applies unchanged.
- **DoS:** a malicious peer can flood fake zero-discriminator packets to exhaust session-creation slots. Rate-limit session creation per source address pair.

## Compatibility

### With Single-Hop

Distinct UDP ports (3784 vs 4784) allow single-hop and multihop sessions to coexist on the same interface. Single-hop sessions retain GTSM enforcement; multihop sessions rely on authentication instead.
