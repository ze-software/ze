# RFC 1877 - PPP Internet Protocol Control Protocol Extensions for Name Server Addresses

## Meta

| Field | Value |
|-------|-------|
| RFC | 1877 |
| Title | PPP Internet Protocol Control Protocol Extensions for Name Server Addresses |
| Status | Informational |
| Date | December 1995 |
| Author | S. Cobb |
| Obsoletes | - |
| Updates | - (informationally extends RFC 1332; does not formally update it) |
| Updated by | - |

**Purpose:** Adds four optional IPCP Configuration Options that let the
peer learn addresses for the primary and secondary DNS and NBNS
(NetBIOS Name Server) servers it should use while the PPP link is up.

**Scope:** Only four option types are defined. They do not alter the
IPCP packet format, do not change the state machine, and are
informational in the sense that either side may ignore them by
Configure-Reject; the link MUST still be usable for IPv4 traffic with
or without DNS assignment.

## Relationship to IPCP

RFC 1877 rides entirely on top of RFC 1332's IPCP. It reuses the LCP
packet shape (Code, Identifier, Length, Data), the IPCP state machine,
and the IPCP Protocol field `0x8021`. Only the Data field (the option
list inside a Configure-Request / Ack / Nak / Reject) carries new type
values.

## Configuration Options

Each option has the standard TLV shape: one-byte Type, one-byte
Length (including the TLV header), four-byte IPv4 address.

| Type | Length | Name |
|------|--------|------|
| 129 | 6 | Primary-DNS-Address |
| 130 | 6 | Primary-NBNS-Address |
| 131 | 6 | Secondary-DNS-Address |
| 132 | 6 | Secondary-NBNS-Address |

A length value other than 6 is a malformed option; the receiver MUST
Configure-Reject it.

## Negotiation Semantics

The negotiation pattern mirrors the IP-Address option from RFC 1332:

| Request value | Meaning |
|---------------|---------|
| `0.0.0.0` | "I do not know this server; please tell me" (client request) |
| Non-zero | "I believe this is the server address" |

The peer responds:

| Condition | Response |
|-----------|----------|
| Requested value is `0.0.0.0` and we have a value to suggest | Configure-Nak carrying our suggestion |
| Requested value is non-zero and we accept it | Configure-Ack (echo the Data verbatim) |
| Requested value is non-zero and we disagree | Configure-Nak carrying our counter-proposal |
| We do not support the option | Configure-Reject (echo the offending option) |

## How Ze Uses These Options

Ze runs as an LNS (Section 1.1 of RFC 5072, server role). On session
start the NCP coordinator emits `EventIPRequest{Family: ipv4}` on the
IPEventsOut channel; the external IP handler (l2tp-pool plugin in
production) returns `Local`, `Peer`, `DNSPrimary`, `DNSSecondary` in
its `IPResponse`. Ze then builds its initial IPCP Configure-Request
with:

- IP-Address = `Local` (ze's PPP-link address, RFC 1332 Section 3.3)
- Primary-DNS-Address = `DNSPrimary` if non-zero
- Secondary-DNS-Address = `DNSSecondary` if non-zero

Ze does NOT implement options 130 / 132 (NBNS). NBNS is a
Microsoft-specific service that modern clients ignore; the handoff
from the peer to a DHCP-based NBNS discovery or a dedicated `nmbd`
lookup is the operator's responsibility.

If the peer Configure-Rejects Primary-DNS-Address or Secondary-DNS-
Address, ze clears the corresponding field and omits the option from
its next Configure-Request. Configure-Reject of IP-Address (RFC 1332
type 3) is fatal because it removes the only option ze needs to
negotiate (spec-l2tp-6c-ncp AC-16).

## Security Considerations

RFC 1877 Section 3 observes that because IPCP is not authenticated,
the DNS addresses carried in these options are as trustworthy as the
PPP peer. Ze assumes the peer is a subscriber CPE that will forward
user DNS queries, so a malicious peer can only harm its own user; a
malicious LAC could push a hostile resolver on ze, which is why ze's
`l2tp-pool` plugin is the single source of DNS values and the reactor
ignores any peer-originated Primary/Secondary-DNS Naks that conflict
with policy.
