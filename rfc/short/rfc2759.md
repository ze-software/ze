# RFC 2759 - Microsoft PPP CHAP Extensions, Version 2

## Meta

| Field | Value |
|-------|-------|
| RFC | 2759 |
| Title | Microsoft PPP CHAP Extensions, Version 2 |
| Status | Informational |
| Date | January 2000 |
| Author | G. Zorn (Microsoft) |
| Obsoletes | - |
| Updates | - |
| Updated by | - |

**Purpose:** Defines the PPP Microsoft CHAP (MS-CHAP) authentication protocol
Version 2. MS-CHAPv2 layers mutual authentication, integrated change-password
support, and a new cryptographic authenticator-response onto the CHAP packet
format defined by RFC 1994. Algorithm code is `0x81`.

**Scope:** MS-CHAPv2 runs during the PPP Authentication phase (between LCP
Opened and the first NCP), driven by the authenticator. Unlike RFC 1994 CHAP
where Success is an advisory string, MS-CHAPv2 Success carries a
cryptographically-bound Authenticator-Response that the peer verifies so
authentication is mutual.

## Relationship to PPP

MS-CHAPv2 is selected via the PPP LCP Authentication-Protocol Configuration
Option (RFC 1661 option type 3) with Authentication-Protocol value `0xC223`
(shared with RFC 1994 CHAP) and Algorithm value `0x81`. PPP frames carrying
MS-CHAPv2 packets use protocol number `0xC223` in the PPP Information field
and share the CHAP packet framing -- Code, Identifier, Length, Value-Size,
Value -- from RFC 1994 Section 4. The Value field and Success/Failure Message
field formats differ.

## Wire Formats

### MS-CHAPv2 Packet (shared CHAP framing)

```
+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
|     Code      |  Identifier   |            Length             |
+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
|    Data ...
+-+-+-+-+
```

| Field | Size | Meaning |
|-------|------|---------|
| Code | 1 octet | MS-CHAPv2 packet type (shared with CHAP: 1/2/3/4) |
| Identifier | 1 octet | Matches Challenges, Responses, replies |
| Length | 2 octets | Total packet length including Code/Identifier/Length/Data |
| Data | 0+ octets | Format depends on Code |

### Code values

| Code | Meaning |
|------|---------|
| 1 | Challenge |
| 2 | Response |
| 3 | Success |
| 4 | Failure |
| 7 | Change-Password (out of scope for ze Phase 6) |

### Challenge Packet (Section 4)

```
+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
|     Code      |  Identifier   |            Length             |
+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
|  Value-Size   |  Challenge Value (16 octets) ...
+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
|  Name ...
+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
```

| Field | Size | Meaning |
|-------|------|---------|
| Code | 1 octet | 1 (Challenge) |
| Identifier | 1 octet | Changed on every new Challenge |
| Length | 2 octets | Packet length including header |
| Value-Size | 1 octet | MUST be 16 (Authenticator Challenge fixed size) |
| Challenge Value | 16 octets | Authenticator Challenge: random bytes from a cryptographic source |
| Name | Length - 5 - 16 octets | Authenticator identification; runs to end of packet |

### Response Packet (Section 4, Value breakdown Section 5)

```
+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
|     Code      |  Identifier   |            Length             |
+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
|  Value-Size   |  Peer-Challenge (16 octets) ...
+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
|  Reserved (8 octets, MUST be zero) ...
+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
|  NT-Response (24 octets) ...
+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
|  Flags (1 octet, MUST be zero) |  Name ...
+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
```

| Field | Size | Meaning |
|-------|------|---------|
| Code | 1 octet | 2 (Response) |
| Identifier | 1 octet | Copied from the Challenge that elicited this Response |
| Length | 2 octets | Packet length including header |
| Value-Size | 1 octet | MUST be exactly 49 (16 + 8 + 24 + 1) |
| Peer-Challenge | 16 octets | Peer-generated random value bound into the hash inputs |
| Reserved | 8 octets | MUST be zero on transmit; MUST be rejected if non-zero on receive |
| NT-Response | 24 octets | DES-encrypted output of ChallengeHash + NtPasswordHash |
| Flags | 1 octet | Reserved; MUST be zero in this version |
| Name | Length - 5 - 49 octets | Peer identity (ASCII, optionally `DOMAIN\user`); runs to end of packet |

### Success Packet (Section 5)

```
+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
|     Code      |  Identifier   |            Length             |
+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
|  Message  ...
+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-
```

| Field | Size | Meaning |
|-------|------|---------|
| Code | 1 octet | 3 (Success) |
| Identifier | 1 octet | Copied from the Response that elicited this Success |
| Length | 2 octets | Packet length including header |
| Message | Length - 4 octets | Formatted string `S=<auth_string> M=<human-readable>` |

Success Message format (exactly as emitted by the authenticator):

```
S=<40-hex-uppercase> M=<free-form ASCII>
```

| Element | Size | Meaning |
|---------|------|---------|
| Literal `S=` | 2 octets | Field introducer |
| auth_string | 40 octets | ASCII hex (uppercase A-F) of the 20-octet Authenticator-Response |
| Literal ` M=` | 3 octets | Separator + message introducer |
| message text | 0+ octets | Free-form ASCII; informational only |

### Failure Packet (Section 5)

Same CHAP framing as Success; Code is 4. Message format:

```
E=<decimal-error> R=<0|1> C=<32-hex-uppercase> V=<decimal-version> M=<free-form ASCII>
```

| Element | Size | Meaning |
|---------|------|---------|
| Literal `E=` | 2 octets | Error-code introducer |
| error code | 3-10 octets | ASCII decimal; see Error Codes below |
| Literal ` R=` | 3 octets | Retry-flag introducer |
| retry | 1 octet | `0` (no retry) or `1` (retry allowed) |
| Literal ` C=` | 3 octets | Challenge introducer; MUST be present |
| new challenge | 32 octets | ASCII hex (uppercase) of a fresh 16-octet challenge |
| Literal ` V=` | 3 octets | Version introducer |
| version | 1-3 octets | ASCII decimal; SHOULD always be `3` for MS-CHAPv2 |
| Literal ` M=` | 3 octets | Message introducer |
| message text | 0+ octets | Free-form ASCII; informational only |

### Algorithm value (Section 2)

| Value | Meaning |
|-------|---------|
| 0x80 | MS-CHAPv1 (RFC 2433, ze does not implement) |
| 0x81 | MS-CHAPv2 (this RFC) |

## Encoding Rules

| Aspect | Rule |
|--------|------|
| Byte order | Network (big-endian) for Length; raw bytes for Challenge / Peer-Challenge / NT-Response |
| Length | Two octets, covers Code + Identifier + Length + Data |
| Identifier | One octet. MUST change on every new Challenge; retransmissions reuse the same Identifier |
| Value-Size | One octet. 16 in Challenge; 49 in Response |
| Reserved | 8 octets of zero in Response Value; MUST be rejected if non-zero on receive |
| Flags | 1 octet of zero in Response Value; MUST be zero in this version |
| Message ASCII | Success and Failure Message bodies are printable ASCII; hex digits A-F MUST be uppercase |
| Padding | Data beyond the Length field is Data Link Layer padding and ignored |

## Validation

| Check | Action on failure |
|-------|-------------------|
| Length < 4 or beyond frame | Silently discard (RFC 1994 inheritance) |
| Unknown Code | Silently discard |
| Response Value-Size != 49 | Reject as malformed -- protocol violation |
| Response Reserved octets non-zero | Reject as malformed (Section 4) |
| Response Flags octet non-zero | Reject -- Flags MUST be zero in this version |
| Response Identifier does not match outstanding Challenge | Silently discard or fail (RFC 1994 Implementation Notes apply) |
| NT-Response hash mismatch | Send Failure with error-code `E=691`; SHOULD terminate the link |

## MUST Requirements

**Tx (authenticator):**
- Send a Challenge with a fresh Identifier and a random 16-octet Authenticator Challenge.
- On correct NT-Response, send a Success whose Message carries `S=<40-hex>` encoding the 20-octet Authenticator-Response.
- On mismatch, send a Failure whose Message carries the fully formatted `E=.../R=.../C=.../V=.../M=...` string. The `C=` field MUST be exactly 32 hex octets and MUST be present.
- The Identifier in Success / Failure MUST be copied from the Response that caused this reply.

**Tx (peer):**
- Whenever a Challenge is received, compute `NT-Response = GenerateNTResponse(AuthenticatorChallenge, PeerChallenge, UserName, Password)` and transmit a Response.
- The Reserved octets in Response MUST be transmitted as zero.
- The Flags octet in Response MUST be transmitted as zero.

**Mutual authentication:**
- The peer MUST verify the Authenticator-Response delivered in Success by recomputing `GenerateAuthenticatorResponse(Password, NT-Response, PeerChallenge, AuthenticatorChallenge, UserName)` and comparing to the 20-octet value decoded from the `S=` field.
- If the Authenticator-Response is missing or incorrect, the peer MUST end the session.

**Randomness:**
- Authenticator Challenge MUST be drawn from a cryptographic random source.
- Peer-Challenge MUST be drawn from a cryptographic random source.

**Character casing:**
- Hex digits A-F in the `S=`, `C=`, and (implicitly) any other hex field MUST be uppercase.

## SHOULD / MAY

- The authenticator SHOULD limit the number of password retries to make brute-force attacks harder (Security Considerations).
- The `V=` (version) field of Failure SHOULD always be `3`.
- The `R=` field of Failure MAY be `1` (retry allowed) or `0` (retry forbidden); the peer MAY honor or ignore this advisory flag.
- The `M=` field of Success / Failure is advisory and MAY be logged by the peer; its contents MUST NOT affect protocol state.
- The authenticator MAY repeat the Challenge/Response exchange at any time during the Network-Layer Protocol phase (inherits from RFC 1994 Section 2.2).

## Three-Way Handshake / Mutual Authentication

1. **Authenticator -> Peer:** Challenge (Code 1) with a fresh Identifier, a fresh 16-octet Authenticator Challenge, and the authenticator's Name.
2. **Peer -> Authenticator:** Response (Code 2) with:
   - `Value-Size = 49`
   - 16-octet Peer-Challenge
   - 8 zero octets (Reserved)
   - 24-octet NT-Response computed via `GenerateNTResponse`
   - 1 zero octet (Flags)
   - peer's Name
3. **Authenticator -> Peer:** verifies NT-Response by recomputing the expected value from the peer's stored NT password hash. On match: emits Success (Code 3) with Message = `"S=<auth_string> M=<msg>"`. On mismatch: emits Failure (Code 4) with Message = `"E=... R=... C=... V=... M=..."` and SHOULD terminate the link.
4. **Peer:** verifies the 20-octet Authenticator-Response decoded from the `S=` field. If missing or incorrect, the peer MUST end the session. Otherwise the peer treats authentication as complete and PPP proceeds to the Network-Layer Protocol phase.

## Algorithms (input recipes only; ze does not implement)

ze never computes MS-CHAPv2 hashes. The RADIUS / l2tp-auth plugin validates the
NT-Response and produces the 20-octet Authenticator-Response. ze only forwards
bytes. The input recipes are recorded here so ze's codec tests can build
well-formed wire inputs for handler tests without implementing the crypto.

### `ChallengeHash(PeerChallenge, AuthenticatorChallenge, UserName) -> 8-octet Challenge`

- SHA-1 over the concatenation `PeerChallenge || AuthenticatorChallenge || UserName`.
- Output: first 8 octets of the SHA-1 digest.
- Only the bare user name (excluding any `DOMAIN\` prefix) is fed to SHA-1.

### `GenerateNTResponse(AuthenticatorChallenge, PeerChallenge, UserName, Password) -> 24-octet NT-Response`

- `Challenge = ChallengeHash(PeerChallenge, AuthenticatorChallenge, UserName)`.
- `PasswordHash = MD4(UnicodeLE(Password))` (the NT password hash, 16 octets).
- `NT-Response = ChallengeResponse(Challenge, PasswordHash)` where `ChallengeResponse` runs three DES encryptions, each keyed by a 7-octet slice of the 21-octet padded `PasswordHash`, over the 8-octet `Challenge`. The three 8-octet ciphertexts concatenate to 24 octets.

### `GenerateAuthenticatorResponse(Password, NT-Response, PeerChallenge, AuthenticatorChallenge, UserName) -> 20-octet AuthenticatorResponse (binary, before hex encode)`

- `PasswordHash = MD4(UnicodeLE(Password))`.
- `PasswordHashHash = MD4(PasswordHash)`.
- `Magic1 = "Magic server to client signing constant"` (39 octets).
- `Digest1 = SHA-1(PasswordHashHash || NT-Response || Magic1)`.
- `Challenge = ChallengeHash(PeerChallenge, AuthenticatorChallenge, UserName)`.
- `Magic2 = "Pad to make it do more than one iteration"` (41 octets).
- `AuthenticatorResponse = SHA-1(Digest1 || Challenge || Magic2)` (20 octets).
- The 20-octet value is encoded as 40 uppercase ASCII hex digits for the `S=` field.

## Error Codes (Section 6)

| Code | Symbol | Meaning |
|------|--------|---------|
| 646 | ERROR_RESTRICTED_LOGON_HOURS | Account disallowed at this time |
| 647 | ERROR_ACCT_DISABLED | Account disabled |
| 648 | ERROR_PASSWD_EXPIRED | Password expired; peer MAY initiate Change-Password (out of ze scope) |
| 649 | ERROR_NO_DIALIN_PERMISSION | Account has no dial-in permission |
| 691 | ERROR_AUTHENTICATION_FAILURE | Credentials incorrect (default reject) |
| 709 | ERROR_CHANGING_PASSWORD | Change-Password exchange failed |

ze forwards the `E=` string verbatim; it does not interpret the code.

## Change-Password (Section 8) -- OUT OF SCOPE for ze Phase 6

Change-Password uses a Code 7 packet that carries an encrypted new password, an
encrypted old hash, a Peer-Challenge, and an NT-Response computed on the new
password. Change-Password is triggered only when the authenticator returns
`E=648` (ERROR_PASSWD_EXPIRED) in a Failure packet. ze does not implement
Change-Password; the `C=` field in ze-emitted Failures is the auth handler's
string, and ze does not examine or honor Code 7 traffic from the peer.

## Pitfalls

- Value-Size is **fixed at 49** on Response. Unlike RFC 1994 CHAP where Value-Size varies with the hash algorithm, any other value on the wire is a protocol violation and MUST be rejected.
- The 8 Reserved octets and 1 Flags octet in Response are NOT length-variable. They are fixed-size, MUST-be-zero bytes. A parser that folds them into the NT-Response by miscounting offsets will produce a valid-looking but wrong NT-Response.
- The `S=` / `E=/R=/C=/V=/M=` formatted strings are bodies of the Message field, NOT a separate wire structure. There is NO Msg-Length octet between the 4-byte header and the Message (inherited from RFC 1994 Section 4.2, same shape as CHAP Success/Failure and unlike PAP Ack/Nak).
- ze does not verify the 32-hex `C=` challenge in Failure. That is a Change-Password challenge; ze never initiates Change-Password and passes the field through.
- Only the bare user name (without `DOMAIN\`) is hashed into `ChallengeHash`. A validator that feeds the full `DOMAIN\user` to SHA-1 will never match.
- The NT password hash is `MD4(UnicodeLE(Password))` -- UTF-16 little-endian, NOT UTF-8. This is a common interop break between Windows and Linux RADIUS backends.
- The authenticator Identifier in Challenge is shared with RFC 1994 CHAP-MD5 via the same LCP Auth-Protocol `0xC223`; LCP negotiates exactly ONE Algorithm per session, so a single per-session identifier counter covers both methods.

## Security Considerations

Selected text from Section 9:

- "The authentication protocol described here ... is to a large extent the result of a rather ad hoc process ... the protocol is best employed in combination with another mechanism providing data encryption." MS-CHAPv2 alone does not provide confidentiality or integrity for the session; IPsec or MPPE is expected for that.
- "The authenticator SHOULD limit the number of password retries allowed to make brute-force password guessing attacks more difficult."
- The mutual authentication does not protect the password from off-line dictionary attack by an attacker who observes a Challenge / Response pair. Combining MS-CHAPv2 with MPPE or IPsec is the expected operational mitigation.
- NT password hash weaknesses (MD4, no salt) mean MS-CHAPv2 offers at best the security of the underlying NT-hash plus the challenge. Modern deployments prefer EAP-TLS or equivalent.

For ze specifically: ze never sees plaintext passwords for MS-CHAPv2. ze acts
as a conduit for the challenge / response / authenticator-response bytes to
the l2tp-auth plugin, which forwards to RADIUS. The cryptographic weaknesses
above are outside ze's control.
