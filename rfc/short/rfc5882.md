# RFC 5882 - Generic Application of BFD

## Meta

| Field | Value |
|-------|-------|
| RFC | 5882 |
| Title | Generic Application of Bidirectional Forwarding Detection (BFD) |
| Status | Standards Track |
| Date | June 2010 |
| Authors | D. Katz, D. Ward (Juniper Networks) |
| Base | RFC 5880, RFC 5881, RFC 5883 |

**Purpose:** Specifies how "BFD clients" -- routing protocols, static routes, management functions -- should request, react to, and coexist with BFD sessions. Covers the service interface, session sharing, state hysteresis, AdminDown semantics, adjacency establishment, graceful restart interactions, and per-protocol guidance for OSPF, IS-IS, BGP, and RIP.

**Scope:** Advisory. RFC 5882 carries no wire format of its own; everything here describes how consumers of BFD should behave.

## Wire Format

None. RFC 5882 does not define any packet structure; it governs the client contract around RFC 5880 sessions.

## Service Interface

The BFD implementation exposes a simple contract:

| Direction | Data |
|-----------|------|
| Client -> BFD | Session parameters: neighbor address, timers, protocol options |
| BFD -> Client | Session state transitions (most importantly to/from Up) |

Responsibilities:

- BFD has no discovery mechanism. Clients bootstrap sessions from their own neighbor discovery.
- BFD is "advisory." Clients react to state transitions using their own existing mechanisms; BFD never carries client data.
- A client can establish or destroy a session at any time. BFD does not require the peer to agree that a session exists.

## Session Sharing

**One session per data protocol path (Section 2).** Only one BFD session SHOULD exist between a given pair of systems for a particular data protocol path, regardless of how many clients are interested. All interested clients subscribe to the same session.

Parameter conflicts (two clients want different timers) are a local matter. BFD chooses one set of parameters; the strictest (fastest) is typically preferred.

A session per client is wasteful and has no benefit.

## Session State Hysteresis (Section 3.1)

A BFD implementation MAY hide rapid Up -> Down -> Up transitions from its clients if the session recovers within a "reasonable" period. This supports control-plane process restarts without triggering full reroute. The exact suppression window is out of scope.

If a session does NOT recover within that window, the client MUST be notified.

## AdminDown Semantics (Section 3.2)

AdminDown signals "BFD is administratively disabled"; it does NOT assert connectivity failure.

| Client independent liveness | Notify on AdminDown? |
|-----------------------------|----------------------|
| Yes (e.g. has its own hello protocol) | No -- ignore the AdminDown |
| No | Yes -- treat as Down |

Rationale: AdminDown may mean BFD is being reconfigured on a live path; a client with another liveness signal can rely on its own detection, whereas a client without one has no way to determine whether the data plane is still forwarding.

## Adjacency Establishment (Section 4.1)

A control protocol adjacency SHOULD be blocked when:

- Both systems are willing to establish a BFD session, AND
- The BFD session cannot be established.

A control protocol adjacency SHOULD NOT be blocked when the peer is believed not to support BFD. Otherwise mixed-support deployments break.

Mechanisms for determining mutual BFD willingness are out of scope; explicit signalling in the control protocol is the typical answer.

On multi-access media where some systems run BFD and others do not, blocking adjacency on absence of BFD can violate the protocol's assumption of full connectivity. Implementations SHOULD be aware of this.

## Reaction to State Changes (Section 4.2)

### Suppressed Transitions

- Up -> AdminDown: client takes NO action (RFC 5880 Section 4.2 semantics).
- Up -> Down because the peer reported AdminDown: client takes NO action.

### Observable Transitions

For other Up -> Down transitions, the client's reaction depends on its topology model.

| Client Type | Action |
|-------------|--------|
| Single data protocol | Signal path unavailable in the control protocol. If the client has an explicit path-state mechanism, use it; otherwise emulate a hello timeout for the neighbor. |
| Multi-protocol, shared topology | Path failure SHOULD remove the neighbor from ALL topologies (otherwise one data protocol is black-holed). |
| Multi-protocol, independent topologies | Only the topology corresponding to the failed BFD session is affected. |

For control protocols that operate out-of-band from the failing data protocol, changing the control protocol's own state is a poor reaction -- use an explicit path-failure signal if one exists.

## Graceful Restart Interaction (Section 4.3)

The C (Control Plane Independent) bit in BFD packets signals whether BFD shares fate with the control plane. It determines how BFD should interact with control-protocol graceful restart.

### C Bit Set in Both Directions (Section 4.3.1)

BFD is independent of the control plane on both ends. Control-plane restarts do NOT perturb BFD.

Rule: if a BFD session fails while a graceful restart is in progress, the graceful restart SHOULD be aborted. A BFD failure means the forwarding plane is broken, so preserving route state would create black holes.

### C Bit Clear in Either Direction (Section 4.3.2)

BFD shares fate with the control plane. Restart-induced BFD failure is indistinguishable from real failure.

Rule: BFD session failures during a restart SHOULD NOT abort the restart. Otherwise BFD and graceful restart cannot coexist.

Sub-case: control protocols with *planned* restart signalling (e.g. BGP graceful restart capability exchanged before the restart) SHOULD suppress the BFD-triggered failure during the planned window.

Sub-case: control protocols without planned restart signalling usually see the BFD timeout before the restart is announced. The restarting system MAY pre-configure longer BFD detection times across the restart period, and SHOULD NOT send BFD Control packets during the restart window (the first packet would cause the peer to declare the session Down).

## Multiple Control Protocols (Section 4.4)

If multiple control protocols want a BFD session to the same remote system for the same data protocol, all MUST share ONE BFD session.

For hierarchical or dependent protocols (e.g. OSPF underneath iBGP), it is often best to run BFD only at the lowest layer. The dependent protocol inherits faster failure reaction via the control-plane interaction between the layers; running BFD at both layers is redundant and can be harmful (BGP peer flaps are expensive).

## Non-Protocol Uses (Section 5)

BFD may drive non-protocol functions such as static route withdrawal.

| Peer BFD Capable | Session Up | Action |
|------------------|-----------|--------|
| Yes | No | Withdraw static route |
| No | N/A | Leave static route in place |

Bootstrap for the non-protocol case is via configuration. Discriminators and endpoints come from configuration; there is no discovery.

## Data Protocol Demultiplexing (Section 6)

BFD protects ONE data protocol at a time. A pair of systems running IPv4 and IPv6 over the same link MUST use two sessions, one per protocol.

Finer-grained paths -- for example, different DiffServ service levels on the same IPv4 link -- each deserve an independent BFD session. Control packets for such sessions MUST be marked the same way the data packets are, both for fate-sharing and for demultiplexing the initial zero-discriminator packet.

## Multi-Link Subnetworks (Section 7)

When multiple parallel links are aggregated at layer N-1, BFD may be applied:

| Approach | Works When | Behavior |
|----------|-----------|----------|
| Complete decoupling | Always | BFD runs only at layer N; detects failure only on timeout |
| Layer N-1 hints | Always | Layer N-1 tells layer N when the aggregation fails; faster reaction |
| Aggregate per-link BFD sessions | Only when layer N neighbor == layer N-1 neighbor | Run BFD per member link; collapse state for the client |
| Combinations | Various | e.g. BFD per link + BFD at higher layer |

## Hitless Bring-Up (Section 3.3)

Establishing a BFD session between a pair of systems SHOULD NOT perturb existing client state. Similarly, restarting BFD SHOULD NOT appear as a connectivity failure to clients that already have their own liveness detection. Clients MAY establish their state before BFD exists.

## Protocol-Specific Guidance (Section 10, Non-Normative)

### OSPFv2, OSPFv3, IS-IS

- Session MUST be bound to the interface carrying OSPF/IS-IS adjacency traffic.
- Baseline hello granularity is OSPF 2 seconds, IS-IS 1 second; BFD supplements these to achieve faster detection.
- Session bring-up: when both sides advertise BFD willingness and both sides run BFD, the adjacency is gated on the BFD session (per Section 4.1).
- OSPF virtual links are multi-hop: use RFC 5883, not RFC 5881.

### BGP

- Most useful for eBGP where alternative paths are available.
- iBGP sessions in the same IGP SHOULD NOT use BFD if the IGP already does; the IGP reconvergence provides the failure reaction and BGP peer flaps are expensive.
- BFD session is bound to the TCP peer's address, not the BGP peer's routing identity.

### RIP

- RIP has no adjacency state. On BFD session failure, simulate route timeouts for the affected neighbor (withdraw routes learned from that peer).

## Interop and Other Application Issues (Section 8)

- BFD on tunnels: prefer running BFD inside the tunnel so it exercises the tunnel's full path. Tunnel endpoints supply the BFD addresses if they are routable.
- Planned outages: a system should transition the session into AdminDown BEFORE the outage, and remain in AdminDown for at least one Detection Time so the peer sees the transition.
- Deconfiguring BFD: same discipline. Do not simply stop sending; go through AdminDown first.

## Pitfalls

- **Running BFD at the wrong layer** (e.g., on iBGP over an IGP that already runs BFD) doubles cost and amplifies flap damage.
- **Not honoring the C bit during graceful restart** causes BFD and GR to interact incorrectly: aborting every restart on a shared-fate system, or preserving state across a real forwarding-plane failure on an independent-fate system.
- **Blocking control protocol adjacency when peer does not support BFD** silently breaks mixed deployments.
- **Ignoring AdminDown on a client with no independent liveness** black-holes traffic through a path whose real state is unknown.
- **Treating a single BFD session as protecting multiple data protocols** violates Section 6; each protocol needs its own session.

## Compatibility

### With Base Protocol

RFC 5882 defines the client contract on top of RFC 5880 state transitions. It does not alter wire format or state machine.
