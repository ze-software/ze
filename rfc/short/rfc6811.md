# RFC 6811: BGP Prefix Origin Validation

## Meta

| Field | Value |
|-------|-------|
| RFC | 6811 |
| Title | BGP Prefix Origin Validation |
| Status | Standards Track |
| Date | January 2013 |
| Obsoletes | - |
| Updates | - |
| Obsoleted-by | - |

**Purpose:** Defines a mechanism for BGP speakers to validate the origin AS of a route against a database of prefix-to-AS mappings (Validated ROA Payloads from RPKI), assigning a validation state of Valid, Invalid, or NotFound.

**Scope:** All address families. Operates on BGP routes using external prefix-to-AS mapping data. No new wire formats, capabilities, or message types.

## Core Concepts

### Validated ROA Payload (VRP)

A VRP is a locally stored object derived from the RPKI (or any prefix-to-AS mapping database):

| Field | Type | Description |
|-------|------|-------------|
| IP address | prefix | Network address |
| Prefix length | uint8 | Minimum prefix length |
| Maximum length | uint8 | Maximum prefix length covered by this authorization |
| Origin AS number | uint32 | AS authorized to originate prefixes in range |

### Route Origin ASN Derivation

The origin AS of a BGP route is derived from AS_PATH as follows:

| AS_PATH final segment type | Route Origin ASN |
|----------------------------|-----------------|
| AS_SEQUENCE | Rightmost AS in final segment |
| AS_CONFED_SEQUENCE | Local BGP speaker's own ASN |
| AS_CONFED_SET | Local BGP speaker's own ASN |
| Empty AS_PATH | Local BGP speaker's own ASN |
| Any other type | Distinguished value "NONE" |

A Route Origin ASN of "NONE" can never match any VRP. An origin ASN of zero can never match (per AS0 processing rules).

### Matching Definitions

| Term | Definition |
|------|------------|
| Covered | VRP prefix length <= Route prefix length, AND VRP prefix address matches Route prefix for all VRP prefix length bits (Route is identical to or more specific than VRP prefix) |
| Matched | Route is Covered by VRP, AND Route prefix length <= VRP maximum length, AND Route Origin ASN == VRP ASN |

### Validation States

| State | Condition | Meaning |
|-------|-----------|---------|
| NotFound | No VRP Covers the Route Prefix | No RPKI data exists for this prefix |
| Valid | At least one VRP Matches the Route Prefix | Origin AS is authorized for this prefix |
| Invalid | At least one VRP Covers the Route Prefix, but no VRP Matches it | Prefix is covered by RPKI but origin AS is not authorized (wrong AS, or prefix too specific) |

## Algorithm

### Validation Procedure (Section 2.1)

```
Input:  route_prefix, route_prefix_length, route_origin_as
Output: validation_state (NotFound, Valid, Invalid)

result = NotFound
prefix_exists = false

for each VRP covering route_prefix:
    prefix_exists = true
    if route_prefix_length <= VRP.max_length:
        if route_origin_as != NONE
           AND VRP.origin_as != 0
           AND route_origin_as == VRP.origin_as:
            return Valid

if prefix_exists:
    return Invalid

return NotFound
```

The order in which VRPs are visited does not matter -- the output is fully determined regardless of evaluation order.

A route may be Covered or Matched by more than one VRP. A single Match is sufficient for Valid.

### Invalidity Causes

A route is Invalid when VRPs Cover it but none Match. This happens when:

| Cause | Example |
|-------|---------|
| Wrong origin AS | VRP authorizes AS 64500 for 10.0.0.0/16, route originated by AS 64501 |
| Prefix too specific | VRP authorizes 10.0.0.0/16 max /20, route is 10.0.0.0/24 |
| Both | Wrong AS and prefix too specific |

## Policy Control (Section 3)

Validation state is a local property of the route. Policy actions based on validation state include:

| Action | Description |
|--------|-------------|
| Filter | Reject Invalid routes (most common deployment) |
| Prefer | Adjust LOCAL_PREF based on validation state (e.g., Valid > NotFound > Invalid) |
| Tag | Set community/extended community for propagation to IBGP peers |
| No action | Carry validation state for logging/monitoring only |

The validation state is not a BGP path attribute -- it is not carried in UPDATE messages. To propagate validation state within IBGP, use communities.

## Deployment Considerations (Section 5)

- Considering Invalid routes in the decision process is a local policy matter requiring caution
- When selection algorithm uses non-propagated properties (like LOCAL_PREF adjustment), validation state may need propagation to IBGP peers via communities
- Incremental deployment: NotFound routes (no RPKI coverage) should generally be accepted

## Pitfalls

### Edge Cases

- **Multiple covering VRPs:** A prefix may be covered by many VRPs with different ASNs and max-lengths. A single Match is sufficient for Valid.
- **AS0 in VRP:** A VRP with ASN 0 means "no AS is authorized to originate" -- any route matching that prefix is Invalid (the route_origin_as == entry->origin_as check fails since no valid route has ASN 0).
- **NONE origin:** Routes with AS_PATH final segment not AS_SEQUENCE (e.g., AS_SET) get origin "NONE" which can never match any VRP -- such routes are either NotFound or Invalid depending on VRP coverage.
- **Max length vs prefix length:** A VRP for 10.0.0.0/16 with max-length /24 authorizes all prefixes from /16 through /24. A /25 announcement is Invalid even if the AS matches.
- **Cache inconsistency:** Different routers may have different RPKI cache contents, leading to different validation states for the same route. This is inherent to distributed caching.

### Interop

- **RTR protocol (RFC 6810):** The mechanism for delivering VRPs from RPKI cache to BGP speakers. Implementation of this RFC typically requires RTR support.
- **No wire format:** This RFC adds no BGP messages, capabilities, or attributes. It is purely a local validation procedure.
- **IBGP propagation:** Validation state does not travel in BGP UPDATEs. Use communities (RFC 8097 standardizes extended community for this).

### Security

- **Limited protection:** Attacker can defeat validation by prepending the authorized origin AS to a forged route.
- **No path validation:** Only verifies origin AS, not the full AS_PATH.
- **Database attacks:** Compromised RPKI data can cause valid routes to be marked Invalid (denial of service) or invalid routes to be marked Valid.
- **Protection against misconfiguration:** Primary value is catching accidental mis-origination (fat-finger errors, route leaks), not determined attackers.
