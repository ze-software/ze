# RFC 1994 - PPP Challenge Handshake Authentication Protocol (CHAP)

## Meta

| Field | Value |
|-------|-------|
| RFC | 1994 |
| Title | PPP Challenge Handshake Authentication Protocol (CHAP) |
| Status | Draft Standard |
| Date | August 1996 |
| Author | W. Simpson (DayDreamer) |
| Obsoletes | RFC 1334 |
| Updates | - |
| Updated by | RFC 2484 (PPP LCP Internationalization) |

**Purpose:** Defines a PPP authentication protocol that uses a random
Challenge with a cryptographically hashed Response depending upon the
Challenge and a shared secret. Only one algorithm is required to be
implemented: CHAP with MD5.

**Scope:** CHAP runs during the PPP Authentication phase (between LCP
Opened and the first NCP). It is a periodic three-way handshake driven
by the authenticator; the peer never initiates. A Challenge MAY also
be sent at any time during the Network-Layer Protocol phase.

## Relationship to PPP

CHAP is selected via the PPP LCP Authentication-Protocol Configuration
Option (RFC 1661 option type 3). The Authentication-Protocol value is
`0xC223` followed by a one-octet Algorithm field. The default (and only
required) algorithm is MD5 (value 5). LCP must reach Opened before CHAP
begins. PPP frames carrying CHAP packets use protocol number `0xC223`
in the PPP Information field.

## Wire Formats

### CHAP Packet (Section 4)

```
+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
|     Code      |  Identifier   |            Length             |
+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
|    Data ...
+-+-+-+-+
```

| Field | Size | Meaning |
|-------|------|---------|
| Code | 1 octet | CHAP packet type (see table below) |
| Identifier | 1 octet | Matches challenges, responses, replies |
| Length | 2 octets | Total packet length including Code/Identifier/Length/Data |
| Data | 0+ octets | Format depends on Code |

Octets outside the range of the Length field are Data Link Layer padding
and are ignored on reception.

### Code values (Section 4)

| Code | Meaning |
|------|---------|
| 1 | Challenge |
| 2 | Response |
| 3 | Success |
| 4 | Failure |

### Challenge / Response Packet (Section 4.1)

```
+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
|     Code      |  Identifier   |            Length             |
+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
|  Value-Size   |  Value ...
+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
|  Name ...
+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
```

| Field | Size | Meaning |
|-------|------|---------|
| Code | 1 octet | 1 Challenge, 2 Response |
| Identifier | 1 octet | Challenge: changed each send. Response: copied from the Challenge |
| Length | 2 octets | Packet length including header |
| Value-Size | 1 octet | Length of Value in octets |
| Value | Value-Size octets | Challenge: random stream (MSB first). Response: hash output (16 octets for MD5) |
| Name | Length - (5 + Value-Size) octets | Identification of the transmitting system; runs to end of packet |

The Name field has no content restriction; it MAY be ASCII or ASN.1
identifiers. It is not NUL or CR/LF terminated; its size is derived
from Length.

### Success / Failure Packet (Section 4.2)

```
+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
|     Code      |  Identifier   |            Length             |
+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
|  Message  ...
+-+-+-+-+-+-+-+-+-+-+-+-+-
```

| Field | Size | Meaning |
|-------|------|---------|
| Code | 1 octet | 3 Success, 4 Failure |
| Identifier | 1 octet | Copied from the Response that caused this reply |
| Length | 2 octets | Packet length including header |
| Message | 0+ octets | Implementation-defined, human-readable; size from Length |

Message is advisory; it MUST NOT affect operation of the protocol.
Recommended character range is ASCII 32-126.

### Algorithm values (Section 3)

| Value | Meaning |
|-------|---------|
| 0-4 | Reserved |
| 5 | CHAP with MD5 (required) |
| 6+ | Reserved / assigned outside this RFC (e.g. RFC 2759 defines MS-CHAPv1 and MS-CHAPv2 under separate Algorithm values) |

## Encoding Rules

| Aspect | Rule |
|--------|------|
| Byte order | Network (big-endian). "Most significant octet is transmitted first" for Value |
| Length | Two octets, covers Code + Identifier + Length + Data |
| Identifier | One octet. MUST change on every new Challenge; retransmissions of the same Challenge reuse the same Identifier |
| Value-Size | One octet, length of Value in octets. For MD5 responses this is 16 |
| Padding | Data beyond the Length field is Data Link Layer padding and ignored |

## Validation

| Check | Action on failure |
|-------|-------------------|
| Length < 4 or beyond frame | Silently discard |
| Unknown Code | Silently discard |
| Response Identifier does not match an outstanding Challenge | Silently discard unless it matches the current Challenge Identifier during Network-Layer phase (then resend prior reply) |
| Response received outside Authentication or Network-Layer phase | Silently discard |
| Value mismatch | Send Failure, SHOULD terminate the link |

## MUST Requirements

**Tx (authenticator):**
- "The authenticator MUST transmit a CHAP packet with the Code field set to 1 (Challenge)." (Section 4.1)
- "Additional Challenge packets MUST be sent until a valid Response packet is received, or an optional retry counter expires." (Section 4.1)
- "Based on this comparison, the authenticator MUST send a Success or Failure packet". (Section 4.1)
- "If the Value received in a Response is equal to the expected value, then the implementation MUST transmit a CHAP packet with the Code field set to 3 (Success)." (Section 4.2)
- "If the Value received in a Response is not equal to the expected value, then the implementation MUST transmit a CHAP packet with the Code field set to 4 (Failure), and SHOULD take action to terminate the link." (Section 4.2)

**Tx (peer):**
- "Whenever a Challenge packet is received, the peer MUST transmit a CHAP packet with the Code field set to 2 (Response)." (Section 4.1)

**Identifier / Value:**
- "The Identifier field MUST be changed each time a Challenge is sent." (Section 4.1)
- "The Response Identifier MUST be copied from the Identifier field of the Challenge which caused the Response." (Section 4.1)
- "The Challenge Value MUST be changed each time a Challenge is sent." (Section 4.1)
- Success/Failure "Identifier field MUST be copied from the Identifier field of the Response which caused this reply." (Section 4.2)

**Rx / replay handling:**
- "The authenticator MUST allow repeated Response packets during the Network-Layer Protocol phase after completing the Authentication phase." (Section 4.1, Implementation Notes)
- "any Response packets received having the current Challenge Identifier MUST return the same reply Code previously returned for that specific Challenge (the message portion MAY be different)." (Section 4.1)
- "Any Response packets received during any other phase MUST be silently discarded." (Section 4.1)

**Config / secret:**
- "If authentication of the link is desired, an implementation MUST specify the Authentication-Protocol Configuration Option during Link Establishment phase." (Section 1)
- "The CHAP algorithm requires that the length of the secret MUST be at least 1 octet." (Section 2.3)
- Message field "MUST NOT affect operation of the protocol." (Section 4.2)

**Interop:**
- "An implementation which does not include this option MUST be prepared to interoperate with another implementation which does include the option." (Section 1.1, on MAY semantics)

## SHOULD / MAY

- "At random intervals, the authenticator sends a new challenge to the peer, and repeats steps 1 to 3." (Section 2) -- the re-challenge MAY happen any time after link establishment.
- "This is done upon initial link establishment, and MAY be repeated anytime after the link has been established." (Section 2)
- "A Challenge packet MAY also be transmitted at any time during the Network-Layer Protocol phase to ensure that the connection has not been altered." (Section 4.1)
- "the connection SHOULD be terminated" on authentication mismatch (Section 2).
- "The peer SHOULD expect Challenge packets during the Authentication phase and the Network-Layer Protocol phase." (Section 4.1)
- "The secret SHOULD be at least as large and unguessable as a well-chosen password." (Section 2.3)
- "Each challenge value SHOULD be unique" and "SHOULD exhibit global and temporal uniqueness" and "SHOULD also be unpredictable". (Section 2.3)
- "The implementation SHOULD provide the capability of logging the error ... and SHOULD record the event in a statistics counter." (Section 1.2, silently discard)
- Name MAY contain ASCII strings or ASN.1 identifiers. (Section 4.1)
- Success/Failure Message MAY differ between replies for the same Identifier. (Section 4.1, Implementation Notes)
- "The secret SHOULD NOT be the same in both directions." (Security Considerations)

## Three-Way Handshake

1. The authenticator sends a Challenge (Code=1) with a fresh Identifier, a random Value, and its Name.
2. The peer computes `Response = MD5(Identifier || Secret || Challenge-Value)` and sends a Response (Code=2) with Value=MD5 output (16 octets) and Name=peer name. The Response Identifier is copied from the Challenge.
3. The authenticator recomputes the expected value from its stored secret and the Challenge it sent. If they match it sends Success (Code=3); otherwise it sends Failure (Code=4) and SHOULD terminate the link.

`Identifier` above is the 1-octet Identifier field of the Challenge packet; it is used as the first octet of the hash input.

## Algorithms

Response computation (Section 4.1, Algorithm value 5 = MD5):

1. Take the 1-octet Identifier field from the received Challenge packet.
2. Concatenate the shared Secret bytes (opaque octet string) after the Identifier.
3. Concatenate the Challenge Value bytes (exactly Value-Size octets) after the Secret.
4. Compute MD5 (RFC 1321) over this concatenation.
5. The 16-octet MD5 digest is placed in the Response Value field; Value-Size is 16.

The length of the Challenge Value is independent of the hash algorithm. The length of the Response Value depends on the hash algorithm (16 octets for MD5).

## Periodic Re-Authentication (Section 2.2)

The authenticator MAY send additional Challenges after the initial Success, at random intervals during the Network-Layer Protocol phase. Each re-challenge follows the full three-way handshake with a new Identifier and a new Challenge Value. Failure of a re-challenge SHOULD terminate the link. The authenticator is in sole control of frequency and timing of re-challenges; the peer never initiates.

## Error Handling

| Situation | Action |
|-----------|--------|
| No Response within the authenticator's timeout | Retransmit the Challenge reusing the same Identifier and Value until the retry counter expires (Section 4.1) |
| Retry counter expires | Send Failure and SHOULD terminate the link; in practice tear down via LCP Terminate-Request |
| Response arrives after Success | If the Identifier matches the current Challenge, replay the previous reply Code (Section 4.1) |
| Response arrives outside Authentication / Network-Layer phase | Silently discard (Section 4.1) |
| Failure packet lost | LCP Terminate-Request / Terminate-Ack indicates failure as a fallback (Section 4.1) |
| Malformed packet (bad Length, unknown Code) | Silently discard; SHOULD log (Section 1.2) |

## Timers and Counters

| Name | Value | Source |
|------|-------|--------|
| Challenge retransmit timer | Implementation-defined (commonly 3 seconds) | RFC 1994 does not mandate a value; see PPP LCP Restart timer for precedent |
| Max-Challenges (retry counter) | Implementation-defined (optional per Section 4.1) | Stops retransmission and triggers Failure |
| Re-challenge interval | Implementation-defined, at random intervals | Section 2, Section 2.2 |
| Authentication-failure counter | SHOULD NOT be reset until after successful authentication or link termination | Security Considerations |

## Pitfalls

- The authenticator controls the exchange; the peer MUST NOT spontaneously authenticate. In ze, the LNS is the authenticator.
- The Identifier from the Challenge goes **into** the hash input as the first octet. Using the PPP frame ID, a session ID, or the Response Identifier is a common interop break.
- The shared secret is an opaque octet string, not necessarily a printable string. Treat it as bytes; do not NUL-terminate and do not trim whitespace.
- CHAP cannot begin unless LCP negotiated Authentication-Protocol = `0xC223` with Algorithm = 5. A CHAP frame arriving before that is a Protocol-Reject at LCP, not a CHAP Failure.
- The Failure Message is advisory only. Log it but tear the link down regardless; do not branch on its content.
- The L2TP tunnel authentication in RFC 2661 Section 4.2 reuses the `MD5(ID || secret || challenge)` construction but with the L2TP Message Type (2 for SCCRP, 3 for SCCCN) as the ID byte -- not the CHAP Identifier. The two uses must not be confused: distinct ID-byte source, distinct secret scope (tunnel vs. PPP), distinct packet format (AVPs vs. CHAP packet).
- CHAP is mutual only if both ends negotiate CHAP in their respective Authentication-Protocol options. Unilateral CHAP (LNS authenticates peer) is the common case.
- Retransmitted Challenges reuse the **same** Identifier and Value. Every new Challenge (not a retransmission) must use a different Identifier and a different Value.

## Security Considerations

Quoted verbatim from Section "Security Considerations":

- "The interaction of the authentication protocols within PPP are highly implementation dependent."
- "upon failure of authentication, some implementations do not terminate the link. Instead, the implementation limits the kind of traffic in the Network-Layer Protocols to a filtered subset".
- "There is no provision for re-tries of failed authentication. However, the LCP state machine can renegotiate the authentication protocol at any time".
- "any counters used for authentication failure not be reset until after successful authentication, or subsequent termination of the failed link."
- "There is no requirement that authentication be full duplex or that the same protocol be used in both directions."
- "The secret SHOULD NOT be the same in both directions. This allows an attacker to replay the peer's challenge, accept the computed response, and use that response to authenticate."
- "It is not anticipated that a particular named user would be authenticated by multiple methods. This would make the user vulnerable to attacks which negotiate the least secure method from among a set (such as PAP rather than CHAP). If the same secret was used, PAP would reveal the secret to be used later with CHAP."
- "Passwords and other secrets should be stored at the respective ends such that access to them is as limited as possible. Ideally, the secrets should only be accessible to the process requiring access in order to perform the authentication."

From Section 2.3 (design requirements carrying security weight):
- "Each challenge value SHOULD be unique" and "SHOULD exhibit global and temporal uniqueness."
- "Each challenge value SHOULD also be unpredictable, least an attacker trick a peer into responding to a predicted future challenge".
- "It is preferred that the secret be at least the length of the hash value for the hashing algorithm chosen (16 octets for MD5)."
