# RFC 9319: The Use of maxLength in the RPKI

## Meta

| Field | Value |
|-------|-------|
| RFC | 9319 |
| Title | The Use of maxLength in the Resource Public Key Infrastructure (RPKI) |
| Status | Best Current Practice (BCP 185) |
| Date | October 2022 |
| Obsoletes | None |
| Updates | None |
| Obsoleted-by | None |

**Purpose:** Recommends ways to reduce the forged-origin hijack attack surface by limiting the set of IP prefixes included in Route Origin Authorizations (ROAs). Primary recommendation is to avoid using the maxLength attribute in ROAs except in specific cases.

**Scope:** This is NOT a protocol specification. It provides operational guidance for RPKI ROA creation, complementing and extending RFC 7115. No new wire formats or protocol mechanisms are defined.

**Errata:** None reported.

## Core Problem: Forged-Origin Sub-Prefix Hijack

When a ROA uses maxLength, it authorizes an AS to originate sub-prefixes that may not actually be announced in BGP. An attacker can exploit this by announcing an unauthorized sub-prefix with the legitimate AS forged into the AS_PATH as origin. Because RPKI-ROV validates the origin AS (not the full path), the forged announcement appears valid.

**Example:**

| ROA Type | ROA Content | Vulnerability |
|----------|-------------|---------------|
| Loose (bad) | `(192.168.0.0/16-24, AS 64496)` | All /17 through /24 sub-prefixes not announced by AS 64496 are hijackable |
| Minimal (good) | `(192.168.0.0/16, 192.168.225.0/24, AS 64496)` | Only forged-origin prefix hijack possible (less damaging) |

**Why sub-prefix hijack is worse than prefix hijack:**
- Sub-prefix hijack: attacker's route is the only route to the sub-prefix (longest-prefix match wins unconditionally)
- Prefix hijack: legitimate AS also originates the prefix, and attacker's path is longer (less propagation)

## Measurements (June 2017)

| Metric | Value |
|--------|-------|
| ROAs using maxLength longer than prefix length | 12% of all ROA prefixes |
| Of those, vulnerable to forged-origin sub-prefix hijack | 84% |

## MUST Requirements

- Operators adopting minimal ROAs MUST review existing ROAs, especially those using maxLength, to ensure the set of included prefixes is minimal with respect to current BGP origination and routing policies (Section 5)
- Published ROAs MUST be replaced as necessary (Section 5)
- Review MUST be repeated whenever the operator makes changes to origination or routing policy (Section 5)
- Operators providing RTDR filtering MUST NOT make the creation of non-minimal ROAs a pre-requisite for its use (Section 6)

## SHOULD/MAY Requirements

### SHOULD

- [SHOULD] Operators SHOULD use minimal ROAs whenever possible — containing only IP prefixes actually originated in BGP (Section 5)
- [SHOULD] Operators SHOULD avoid using the maxLength attribute in ROAs (Section 5)
- [SHOULD NOT] Operators SHOULD NOT create non-minimal ROAs for the purpose of advertising RTDR routes (Section 6)
- [SHOULD] ROAs for DDoS mitigation SHOULD only include prefixes that are always originated plus those sometimes originated; SHOULD NOT include prefixes known to never be originated (Section 5.1)
- [SHOULD] ROAs for DDoS mitigation SHOULD NOT use maxLength unless it has no impact on the set of included prefixes (Section 5.1)
- [SHOULD] User interface designers SHOULD provide warnings about risks of non-minimal ROAs and maxLength usage (Section 7)

### Exceptions Where maxLength May Be Justified

| Exception | Condition |
|-----------|-----------|
| All sub-prefixes announced | Every more-specific prefix permitted by maxLength is actually announced in BGP |
| Large range, many announcements | maxLength is substantially larger than prefix length AND a large number of more-specific prefixes are announced (rare in practice) |
| DDoS mitigation | Service provider needs to originate variable-length sub-prefixes during attacks (accepted risk) |

## DDoS Mitigation Scenario

When using DDoS scrubbing services that originate more-specific prefixes to attract traffic:

| Approach | ROAs | Risk |
|----------|------|------|
| Fixed scrubbing prefix | `ROA:(prefix, AS-victim)` + `ROA:(scrub-prefix, AS-scrubber)` | Scrub prefix vulnerable to forged-origin hijack when not originated |
| Variable scrubbing prefix | `ROA:(prefix, AS-victim)` + `ROA:(scrub-prefix-maxlen, AS-scrubber)` | All sub-prefixes in maxLength range vulnerable |

Key constraint: RPKI propagation latency is too slow for on-the-fly ROA creation during DDoS attacks, so ROAs must be pre-created (making them non-minimal by definition).

## RTDR (Remotely Triggered Black Hole) Filtering

RTDR uses highly specific prefixes (>/24 IPv4, >/48 IPv6, up to /32 or /128) tagged with the BLACKHOLE community (RFC 7999). RPKI-ROV is a poor fit for validating RTDR routes because:

- Creating ROAs with such specific prefixes would require very large maxLength values
- This would massively expand the forged-origin hijack attack surface
- No new RPKI procedures for RTDR are defined (out of scope)

## Defensive De-aggregation Trade-off

When responding to a prefix hijack, a victim may want to originate more-specific prefixes to compete with hijack routes. Minimal ROAs may prevent these defensive announcements from being validated. Despite this risk, minimal ROAs are still recommended because:

- They offer the best protection against the initial attack
- Growing RPKI-ROV adoption reduces neighborhoods where this risk exists
- Direct EBGP adjacencies can work around the limitation

## Operational Considerations

| Concern | Impact |
|---------|--------|
| Routing agility | Minimal ROAs require ROA updates before BGP origination changes |
| Propagation delay | RPKI changes propagate slower than BGP; plan ROA changes in advance |
| Large address space | More onerous for operators with many more-specific announcements; automation recommended |
| Self-inflicted DoS | Failure to update ROAs before routing changes can invalidate legitimate announcements |

## Pitfalls

### For BGP Daemon Implementers

- **RPKI-ROV validation (RFC 6811) is unchanged** — this RFC affects ROA creation practices, not the validation algorithm itself
- **maxLength semantics unchanged** — the wire format and validation logic in RFC 6482/6811 are not modified
- **Route validity states remain the same** — Valid, Invalid, NotFound per RFC 6811
- **No new BGP wire format** — all changes are operational/procedural for ROA issuers

### For Operators Using Ze

- **ROA review is mandatory** — operators adopting these practices MUST audit existing ROAs
- **Pre-create ROAs for DDoS mitigation** — RPKI propagation latency prevents on-the-fly creation
- **RTDR and RPKI-ROV are incompatible** — do not expand ROAs to cover RTDR prefixes
- **Automation recommended** — large networks with many sub-prefix announcements should automate ROA management

## Implementation Notes

This RFC is primarily relevant to operators creating ROAs and to RPKI portal/UI implementers, not to BGP daemon implementations directly. For a BGP daemon like Ze:

1. **No wire format changes** — RPKI-ROV validation per RFC 6811 is unaffected
2. **No protocol changes** — BGP OPEN/UPDATE processing is unchanged
3. **Relevance to Ze** — informational context for operators configuring RPKI-ROV policies; Ze's RPKI-ROV integration (if any) validates routes per RFC 6811 regardless of how ROAs were created
4. **UI guidance** — if Ze provides ROA management tooling, warn users about maxLength risks per Section 7
