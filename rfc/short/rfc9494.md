# RFC 9494: Long-Lived Graceful Restart for BGP

**Status:** Standards Track
**Updates:** RFC 6368
**Date:** November 2023

## Overview

Extends BGP Graceful Restart (RFC 4724) to allow stale routes to be retained for much longer periods (hours/days vs minutes). Introduces the LLGR Capability and two well-known communities.

## Key Concepts

| Term | Meaning |
|------|---------|
| LLGR | Long-Lived Graceful Restart |
| LLST | Long-Lived Stale Time (24-bit, seconds) |
| Helper | Router that detects session failure and retains routes |
| Depreference | Reduce route preference to "least preferred" |

## LLGR Capability (Code 71)

**Wire Format:**

```
+--------------------------------------------------+
| Address Family Identifier (16 bits)              |
+--------------------------------------------------+
| Subsequent Address Family Identifier (8 bits)    |
+--------------------------------------------------+
| Flags for Address Family (8 bits)                |
+--------------------------------------------------+
| Long-Lived Stale Time (24 bits)                  |
+--------------------------------------------------+
| ... (repeated for each AFI/SAFI)                 |
+--------------------------------------------------+
```

**Flags:**

| Bit | Name | Description |
|-----|------|-------------|
| 0 | F | Forwarding state preserved during restart |
| 1-7 | Reserved | MUST be zero |

**Requirements:**
- MUST also advertise GR Capability (RFC 4724)
- If GR not present, LLGR MUST be ignored
- LLST can be up to 16,777,215 seconds (~194 days)

## Well-Known Communities

| Community | Value | Purpose |
|-----------|-------|---------|
| LLGR_STALE | 0xFFFF0006 | Marks routes retained beyond GR timer |
| NO_LLGR | 0xFFFF0007 | Routes should NOT be retained if stale |

## Operation Timeline

```
Session fails
    │
    ▼
┌─────────────────────────┐
│   GR Restart Time       │  Routes retained, preference unchanged
│   (0-4095 seconds)      │
└─────────────────────────┘
    │
    ▼
┌─────────────────────────┐
│   LLGR Period           │  Routes get LLGR_STALE, become least-preferred
│   (0-16,777,215 sec)    │  Routes withdrawn from non-LLGR peers
└─────────────────────────┘
    │
    ▼
Routes deleted (if session not re-established)
```

## Key Behaviors

### When LLGR Period Begins

1. **Start LLST timer** for each AFI/SAFI with nonzero LLST
2. **Attach LLGR_STALE** community to stale routes (requires readvertisement)
3. **Delete NO_LLGR routes** immediately
4. **Depreference** routes (treat as least preferred)

### LLGR_STALE Route Processing

- Treat as **least preferred** in route selection
- **SHOULD NOT advertise** to peers without LLGR capability
- **MUST NOT remove** LLGR_STALE when readvertising
- Only used as last resort (no better route available)

### Session Re-establishment

- RFC 4724 procedures apply for stale routes
- LLST timer continues running until EoR received
- If F bit clear or AFI/SAFI missing in new LLGR cap → delete stale routes
- If both GR and LLGR caps missing → delete all stale routes

## Partial Deployment (Section 4.6)

When advertising to IBGP peers without LLGR:

1. MUST be internal (IBGP/Confederation) neighbors
2. MUST attach NO_EXPORT community
3. MUST set LOCAL_PREF to zero

## VPN Considerations (Section 4.7)

### EBGP PE-CE

- MAY advertise stale routes to CE without LLGR (explicit config required)
- SHOULD attach NO_EXPORT by default
- All other LLGR rules still apply

### IBGP PE-CE (RFC 6368 Update)

When using ATTR_SET:
- LLGR_STALE from imported VPN route MUST be included in CE advertisement
- Even if not in ATTR_SET attribute

## Deployment Guidance

**MUST NOT enable by default** - requires explicit per-AFI/SAFI configuration.

### Safe to Use

- MPLS/tunneled forwarding networks
- Non-forwarding data (VPLS auto-discovery, FlowSpec)
- Route reflectors (control-plane only)

### Use with Caution

- Hop-by-hop forwarded networks (risk of forwarding loops)
- Conventional reachability information

### Security: Label Reuse

For MPLS VPNs, ensure:
- **Label reuse lower bound** > **LLST upper bound**
- Prevents VPN traffic isolation breach

## Implementation Notes

### Timer Interaction

| GR Restart Time | LLST | Behavior |
|-----------------|------|----------|
| 0 | nonzero | Skip GR, go straight to LLGR |
| nonzero | 0 | GR only, no LLGR |
| 0 | 0 | Neither GR nor LLGR (base BGP only) |
| nonzero | nonzero | GR then LLGR (serial) |

### F Bit Coordination

- If LLGR F bit set → GR F bit should also be set
- Otherwise GR will clear state before LLGR kicks in

### Route Selection

```
Normal routes (any preference)
    ↓ more preferred than
LLGR_STALE routes (least preferred)
```

Between two LLGR_STALE routes: normal tiebreaking applies.

## ZeBGP Implementation Considerations

### Capability

- Code: 71
- Variable length (7 bytes per AFI/SAFI tuple)
- Requires GR capability (code 64) or ignore

### Communities

```go
const (
    CommunityLLGRStale = 0xFFFF0006
    CommunityNoLLGR    = 0xFFFF0007
)
```

### State Machine Impact

- Add LLST timer per AFI/SAFI
- Track LLGR state separately from GR state
- Handle F bit in both GR and LLGR capabilities
