# RFC 2918: Route Refresh Capability for BGP-4

## Meta

| Field | Value |
|-------|-------|
| RFC | 2918 |
| Title | Route Refresh Capability for BGP-4 |
| Status | Standards Track |
| Date | September 2000 |
| Author | E. Chen (Redback Networks) |
| Obsoletes | - |
| Updates | - |
| Obsoleted-by | - |

**Purpose:** Defines a BGP capability and message type that allows dynamic exchange of route refresh requests between BGP speakers, triggering re-advertisement of Adj-RIB-Out without session reset.

**Scope:** All AFI/SAFI combinations (address family agnostic).

## Wire Formats

### Route Refresh Capability

Capability code: 2
Capability length: 0

```
No capability data - zero-length capability
```

| Field | Offset | Size | Type | Constraints |
|-------|--------|------|------|-------------|
| Code | 0 | 1 | uint8 | Must be 2 |
| Length | 1 | 1 | uint8 | Must be 0 |

### ROUTE-REFRESH Message

Type code: 5

```
         0       7      15      23      31
         +-------+-------+-------+-------+
         |      AFI      | Res.  | SAFI  |
         +-------+-------+-------+-------+
```

| Field | Offset | Size | Type | Constraints |
|-------|--------|------|------|-------------|
| AFI | 0 | 2 | uint16 | Address Family Identifier |
| Reserved | 2 | 1 | uint8 | Set to 0 by sender, ignored by receiver |
| SAFI | 3 | 1 | uint8 | Subsequent Address Family Identifier |

**Length calculation:** Fixed 4 bytes (no BGP header included in calculation).

**Total message size:** 23 bytes (19-byte BGP header + 4-byte payload).

## Encoding Rules

- AFI: 16-bit, network byte order (big-endian)
- Reserved: Set to 0 by sender
- SAFI: 8-bit value
- AFI/SAFI encoding follows RFC 2858 Section 7

## Decoding Rules

1. Verify message length is exactly 23 bytes (header + 4 bytes payload)
2. Parse AFI (bytes 0-1 of payload)
3. Skip Reserved byte (byte 2)
4. Parse SAFI (byte 3)

## Validation

| Check | Valid | Invalid Action |
|-------|-------|----------------|
| Message length | Exactly 4 bytes payload | Send NOTIFICATION (Message Header Error) |
| AFI/SAFI advertised by peer | Yes | Ignore message silently |

## MUST Requirements

### Tx (Sender)

- "A BGP speaker may send a ROUTE-REFRESH message to its peer only if it has received the Route Refresh Capability from its peer." (Section 4)
- Reserved field "Should be set to 0 by the sender" (Section 3)

### Rx (Receiver)

- "If a BGP speaker receives from its peer a ROUTE-REFRESH message with the <AFI, SAFI> that the speaker didn't advertise to the peer at the session establishment time via capability advertisement, the speaker shall ignore such a message." (Section 4)
- "Otherwise, the BGP speaker shall re-advertise to that peer the Adj-RIB-Out of the <AFI, SAFI> carried in the message, based on its outbound route filtering policy." (Section 4)

### Validation

- Reserved field "ignored by the receiver" (Section 3)

## SHOULD/MAY

- [SHOULD] "A BGP speaker that is willing to receive the ROUTE-REFRESH message from its peer should advertise the Route Refresh Capability to the peer" (Section 4) - If not advertised, peer cannot send ROUTE-REFRESH
- [SHOULD] "<AFI, SAFI> carried in such a message should be one of the <AFI, SAFI> that the peer has advertised to the speaker at the session establishment time via capability advertisement" (Section 4) - Request for unadvertised AFI/SAFI will be ignored
- [MAY] "A BGP speaker may send a ROUTE-REFRESH message to its peer" (Section 4) - Sending is optional, typically triggered by policy change

## Error Handling

| Condition | Detect How | Response | Code/Subcode |
|-----------|------------|----------|--------------|
| Unknown AFI/SAFI | Lookup in peer's advertised capabilities | Ignore silently | N/A |
| Capability not advertised by peer | Check peer's capability set | Do not send ROUTE-REFRESH | N/A |
| Malformed message | Length != 4 bytes | NOTIFICATION | Per RFC 4271 |

**Note:** This RFC does not define new NOTIFICATION error codes. Malformed messages use standard BGP-4 error handling.

## Algorithms

### Sending ROUTE-REFRESH

```
1. Check peer advertised Route Refresh Capability
2. If not: abort (cannot send)
3. Check peer advertised target AFI/SAFI
4. If not: abort (will be ignored anyway)
5. Build message: AFI(2) + Reserved(0x00) + SAFI(1)
6. Send with BGP header (type=5)
```

### Receiving ROUTE-REFRESH

```
1. Parse AFI, SAFI from message
2. Lookup AFI/SAFI in locally advertised capabilities
3. If not found: ignore message, done
4. Schedule Adj-RIB-Out re-advertisement for AFI/SAFI
5. Apply outbound route filtering policy during re-advertisement
```

## Pitfalls

### Edge Cases

- **Multiple AFI/SAFI:** Each ROUTE-REFRESH is for a single AFI/SAFI. To refresh all address families, send multiple messages.
- **Reserved field:** Must be ignored on receive even if non-zero (forward compatibility).
- **Timing:** RFC does not specify when re-advertisement must complete. Implementation may batch or delay.

### Interop

- **Cisco proprietary (Type 128):** Some older implementations use message type 128 instead of 5. May need to support both.
- **Draft vs RFC:** Capability code 2 is standardized; some ancient implementations may use draft values.

### Security

- **Resource exhaustion:** Malicious peer could spam ROUTE-REFRESH causing CPU load from repeated Adj-RIB-Out scans. Consider rate limiting.
- **No authentication:** RFC notes "This extension to BGP does not change the underlying security issues" (Section 5).

## Compatibility

### Non-Supporting Peers

- If peer does not advertise Route Refresh Capability: do not send ROUTE-REFRESH messages
- Graceful fallback: use soft-reconfiguration (store all routes) or session reset for policy changes

### Feature Negotiation

- Capability is unilateral: advertising means "I can receive ROUTE-REFRESH"
- Both peers may independently advertise the capability
- Capability advertisement during OPEN exchange
