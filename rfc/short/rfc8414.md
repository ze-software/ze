# RFC 8414 - OAuth 2.0 Authorization Server Metadata

## Meta

| Field | Value |
|-------|-------|
| RFC | 8414 |
| Title | OAuth 2.0 Authorization Server Metadata |
| Status | Proposed Standard |
| Date | June 2018 |
| Authors | M. Jones, N. Sakimura, J. Bradley |
| Related | RFC 9728 (resource metadata), RFC 7517 (JWKS), OpenID Connect Discovery 1.0 |

**Purpose:** Standardises a discovery document the authorization server publishes so clients and resource servers can learn endpoint URLs, supported algorithms, and the `jwks_uri` where signing keys live. Ze uses RFC 8414 as a resource server: it reads `issuer` (for JWT `iss` matching) and `jwks_uri` (for token verification keys). No other field is consumed.

## Well-Known Endpoint (Section 3)

| Item | Value |
|------|-------|
| Path | `/.well-known/oauth-authorization-server` |
| Method | GET |
| Auth | MUST be reachable without authentication |
| Content-Type | `application/json` |

Note: OpenID Connect Discovery uses `/.well-known/openid-configuration` for the same purpose. Ze queries the RFC 8414 path explicitly; if the operator's AS is OIDC-only, they must configure its concrete metadata URL.

## Required Fields (Section 2)

Ze trusts these fields from the document:

| Field | Required? | Ze use |
|-------|-----------|--------|
| `issuer` | MUST | Compared exact-match against JWT `iss` claim on every token |
| `jwks_uri` | SHOULD | Fetched once at startup + refreshed on cache TTL to obtain verification keys |
| `authorization_endpoint` | MUST (for clients) | Not used by resource server |
| `token_endpoint` | MUST (for clients) | Not used by resource server |
| `scopes_supported` | RECOMMENDED | Not used; operator declares required scopes separately |
| `response_types_supported` | MUST | Not used by resource server |
| `grant_types_supported` | OPTIONAL | Not used |
| `token_endpoint_auth_methods_supported` | OPTIONAL | Not used |
| `id_token_signing_alg_values_supported` | OPTIONAL | Not used (Ze's JWT verifier has its own allow-list) |

## Security Properties (Section 3.3)

| Requirement | Detail |
|-------------|--------|
| TLS | Fetching metadata MUST use TLS in production. Ze's config verifier rejects `auth-mode=oauth` on non-loopback without TLS. |
| Issuer validation | The `issuer` field in the document MUST match the base URL the client used to fetch it. Ze enforces this at startup. |
| Content integrity | Metadata changes only when the operator updates AS configuration; document is cacheable. Ze refreshes JWKS on TTL, not the metadata document itself. |

## Ze Implementation

- Fetcher: `internal/component/mcp/as_metadata.go` `fetchASMetadata` -- single HTTP GET, bounded body size, JSON decode via `map[string]any` (avoids snake_case struct tags).
- Consumption: `internal/component/mcp/streamable.go` `buildAuthForMode` fetches metadata at startup, feeds `issuer` into the JWT verifier's `ExpectedIssuer`, constructs a `jwksCache` around `jwks_uri`.
- Failure mode: startup error if metadata fetch fails -- the daemon does not start with a misconfigured AS URL; surfaces the problem to the operator immediately rather than 401-ing every request.
