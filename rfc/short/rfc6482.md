# RFC 6482: A Profile for Route Origin Authorizations (ROAs)

## Meta

| Field | Value |
|-------|-------|
| RFC | 6482 |
| Title | A Profile for Route Origin Authorizations (ROAs) |
| Status | Standards Track |
| Date | February 2012 |
| Obsoletes | - |
| Updates | - |
| Obsoleted-by | - |

**Purpose:** Defines the profile for Route Origin Authorizations (ROAs), digitally signed RPKI objects that authorize a specific AS to originate routes to one or more IP address prefixes. ROAs are the mechanism by which address holders express routing intent in the RPKI.

**Scope:** IPv4 (AFI 0001) and IPv6 (AFI 0002) address prefixes. One ROA per AS; multiple ROAs needed to authorize multiple ASes for the same prefixes.

## ROA Structure

### Content-Type

OID: `1.2.840.113549.1.9.16.1.24` (routeOriginAuthz)

This OID appears in both the `eContentType` in the `encapContentInfo` object and the `content-type` signed attribute in the `signerInfo` object.

### ASN.1 Definition

```
RPKI-ROA { iso(1) member-body(2) us(840) rsadsi(113549)
   pkcs(1) pkcs9(9) smime(16) mod(0) 61 }

DEFINITIONS EXPLICIT TAGS ::= BEGIN

RouteOriginAttestation ::= SEQUENCE {
   version [0] INTEGER DEFAULT 0,
   asID  ASID,
   ipAddrBlocks SEQUENCE (SIZE(1..MAX)) OF ROAIPAddressFamily }

ASID ::= INTEGER

ROAIPAddressFamily ::= SEQUENCE {
   addressFamily OCTET STRING (SIZE (2..3)),
   addresses SEQUENCE (SIZE (1..MAX)) OF ROAIPAddress }

ROAIPAddress ::= SEQUENCE {
   address IPAddress,
   maxLength INTEGER OPTIONAL }

IPAddress ::= BIT STRING

END
```

### Field Descriptions

| Field | Type | Constraints | Description |
|-------|------|-------------|-------------|
| version | INTEGER | MUST be 0 | ROA version number |
| asID | INTEGER | Valid AS number | AS authorized to originate routes |
| ipAddrBlocks | SEQUENCE OF ROAIPAddressFamily | At least 1 entry | IP prefixes the AS may originate |

### ROAIPAddressFamily

| Field | Type | Constraints | Description |
|-------|------|-------------|-------------|
| addressFamily | OCTET STRING | 2-3 bytes; 0001 (IPv4) or 0002 (IPv6) | AFI per IANA registry |
| addresses | SEQUENCE OF ROAIPAddress | At least 1 entry | Prefixes within this address family |

### ROAIPAddress

| Field | Type | Constraints | Description |
|-------|------|-------------|-------------|
| address | BIT STRING | IP prefix per RFC 3779 encoding | The authorized prefix |
| maxLength | INTEGER | OPTIONAL; >= prefix length, <= address family max (32 for IPv4, 128 for IPv6) | Maximum prefix length the AS may advertise |

## maxLength Semantics

The `maxLength` field controls how specific the authorized advertisements may be.

| Scenario | maxLength | Authorized Advertisements |
|----------|-----------|--------------------------|
| Exact prefix only | absent | Only the exact prefix in the ROA |
| With maxLength | present | The prefix itself and any more-specific up to maxLength |

**Example:** prefix 203.0.113.0/24 with maxLength 26 authorizes:
- 203.0.113.0/24 (the prefix itself)
- 203.0.113.0/25, 203.0.113.128/25 (length 25)
- 203.0.113.0/26, 203.0.113.64/26, 203.0.113.128/26, 203.0.113.192/26 (length 26)
- NOT 203.0.113.0/27 (exceeds maxLength)

## Validation

ROA validation requires two layers:

| Step | Source | Description |
|------|--------|-------------|
| 1. Generic RPKI validation | RFC 6488 | CMS signature, certificate chain, validity period, all standard RPKI signed object checks |
| 2. ROA-specific validation | RFC 6482 Section 4 | Each IP prefix in the ROA MUST be contained within the EE certificate's IP address delegation extension (RFC 3779) |

The EE certificate is embedded within the ROA's CMS wrapper. Its IP address delegation extension defines the address space the certificate holder controls. Every prefix in the ROA must fall within that delegated space.

## MUST Requirements

### Structure
- "The version number of the RouteOriginAttestation MUST be 0" (Section 3.1)
- "addressFamily MUST be either 0001 or 0002" (Section 3.3)
- "the maxLength MUST be an integer greater than or equal to the length of the accompanying prefix, and less than or equal to the length (in bits) of an IP address in the address family (32 for IPv4 and 128 for IPv6)" (Section 3.3)

### Content-Type
- "This OID MUST appear both within the eContentType in the encapContentInfo object as well as the content-type signed attribute in the signerInfo object" (Section 2)

### Validation
- "the relying party MUST first validate the ROA" before using it (Section 4)
- "the relying party MUST perform all the validation checks specified in [RFC6488] as well as the following additional ROA-specific validation step" (Section 4)
- "each IP address prefix(es) in the ROA is contained within the set of IP addresses specified by the EE certificate's IP address delegation extension" (Section 4)
- "the integrity of a ROA MUST be established" (Section 5)
- "one MUST verify the signature on the ROA using an X.509 certificate issued under this PKI, and check that the prefix(es) in the ROA match those in the certificate's address space extension" (Section 5)

## SHOULD/MAY

- [NOT RECOMMENDED] "a ROA MAY contain two ROAIPAddress elements, where the IP address prefix is identical in both cases. However, this is NOT RECOMMENDED as [...] the ROAIPAddress with the shorter maxLength grants no additional privileges" (Section 3.3)

## Relationship to RPKI Certificates

| Component | Role |
|-----------|------|
| RPKI Resource Certificate (RFC 6487) | X.509 certificate binding public key to IP address blocks and AS numbers |
| End-Entity (EE) Certificate | Embedded in ROA; its IP address delegation extension constrains which prefixes the ROA may authorize |
| CMS Signed Object (RFC 6488) | Wrapper providing digital signature and certificate chain for the ROA |
| ROA eContent | The payload: AS number + authorized prefixes |

The trust chain: RPKI Trust Anchor -> Intermediate CA(s) -> EE Certificate -> ROA signature. The EE certificate's RFC 3779 IP address extension MUST encompass all prefixes listed in the ROA.

## BGP Implementer Notes

A BGP daemon does not create or sign ROAs. The implementer-relevant aspects are:

| Aspect | Relevance |
|--------|-----------|
| ROA consumption | A BGP daemon may use validated ROA data (from an RPKI validator/RTR cache) to perform Route Origin Validation (RFC 6811) |
| Route Origin Validation | Compare received BGP announcements (origin AS + prefix) against ROA data to determine validity state: Valid, Invalid, or NotFound |
| maxLength | A route is Valid only if the prefix length is <= the ROA's maxLength (or equals the prefix length if maxLength absent) |
| Multiple ROAs | A prefix may be covered by multiple ROAs; route is Valid if ANY matching ROA authorizes it |
| RTR Protocol | RFC 6810/8210 defines the protocol between RPKI validators and BGP routers for delivering validated ROA payload |

## Pitfalls

### Edge Cases
- **Overlapping prefixes in a single ROA:** Valid; a ROA may contain 203.0.113.0/24 (maxLength 26) and 203.0.113.0/28 (maxLength 28) simultaneously
- **Duplicate prefixes with different maxLength:** Allowed but NOT RECOMMENDED; shorter maxLength is redundant
- **Missing maxLength:** AS authorized for exact prefix only, not more-specifics
- **One AS per ROA:** Multiple ASes for same prefix require separate ROAs

### Security
- **No confidentiality:** ROAs are public objects stored in publicly accessible repositories
- **Authorization not authentication:** RPKI validates the right to originate, not identity
- **No non-repudiation:** ROA signatures do not convey non-repudiation
