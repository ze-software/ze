# RFC 5881 - BFD for IPv4 and IPv6 (Single Hop)

## Meta

| Field | Value |
|-------|-------|
| RFC | 5881 |
| Title | Bidirectional Forwarding Detection (BFD) for IPv4 and IPv6 (Single Hop) |
| Status | Standards Track |
| Date | June 2010 |
| Authors | D. Katz, D. Ward (Juniper Networks) |
| Base | RFC 5880 |
| Related | RFC 5082 (GTSM), RFC 5882 (Generic Application) |

**Purpose:** Specifies how BFD is carried and demultiplexed for liveness detection across a single IP hop between directly-connected systems. Covers UDP encapsulation, port assignments, interface binding, and TTL/Hop Limit security.

**Scope:** IPv4 and IPv6, single-hop only. Tunnels that do not decrement the TTL may reuse this mechanism.

## Wire Format

### Encapsulation

BFD Control packets are UDP over IPv4 or IPv6. The BFD payload is the mandatory section (and optional authentication section) defined in RFC 5880.

| Role | UDP Destination Port |
|------|---------------------:|
| Control | 3784 |
| Echo | 3785 |

| Role | UDP Source Port |
|------|----------------:|
| Control | 49152-65535 (ephemeral), same across all packets of a session |
| Echo | Implementation choice (not specified) |

The same UDP source port MUST be reused for all Control packets in a session and SHOULD be unique across sessions on the system. With more than 16384 concurrent sessions, source ports MAY be reused, but the number of distinct sessions per source port SHOULD be minimised.

### TTL / Hop Limit Rules (Section 5)

| Condition | Transmit | Receive |
|-----------|----------|---------|
| Authentication NOT in use | MUST be 255 | MUST be discarded if != 255 |
| Authentication in use | MUST be 255 | MAY be discarded if != 255 (check MAY be done before crypto to save CPU) |

This is the Generalized TTL Security Mechanism (GTSM, RFC 5082): any packet received with TTL != 255 has been routed through a remote system and cannot have originated from the direct neighbour.

"Authentication in use" means the system is transmitting with A=1 and discarding unauthenticated packets demultiplexed to the session.

## States and Transitions

RFC 5881 adds no states. All state handling is inherited from RFC 5880.

## Demultiplexing and Session Identity (Section 3, 6)

### Active/Active Roles

Both endpoints MUST take the Active role (both send initial Control packets with Your Discriminator = 0). This is because single-hop sessions are fully described by the interface and the address pair; no passive discovery is needed.

### First-Packet Rule

A received packet with Your Discriminator = 0 MUST be associated with the session bound to:

- The remote system's IP address
- The ingress interface
- The data protocol (IPv4 or IPv6)

Once the peer's My Discriminator has been learned, Your Discriminator alone demultiplexes all subsequent packets (source address and even interface MAY change without breaking the session, subject to the TTL=255 rule).

### Per-Protocol Sessions

A separate BFD session MUST be established for each data protocol over the same link. A dual-stack link therefore requires two sessions: one IPv4, one IPv6.

### Addressing Rules

| Topology | Source Address | Destination Address |
|----------|----------------|---------------------|
| Multi-access | From the local interface's subnet | On the peer's interface in that subnet |
| Point-to-point | MAY vary across packets (any source accepted initially; subsequent packets demultiplex via Your Discriminator) | Address configured at session creation |

On a point-to-point link, the initial source address does NOT identify the session. The local system MUST NOT change its outgoing destination to track a changed peer source; it continues using the destination configured at session creation. Applications MAY be notified that the peer's source address changed.

## Echo Function

### Encapsulation

- UDP destination port 3785
- UDP source port is unspecified
- Source IP address MUST be chosen so the peer does not send ICMP/ND Redirects (typically NOT from the subnet bound to the egress interface, NOT an IPv6 link-local)
- Destination IP address MUST be chosen so the peer forwards the packet back to the local system (typically the local system's own address)

On multi-access media, the L2 destination address MUST be the remote system's MAC so the packet physically reaches the remote before the IP layer reflects it.

### Constraints

- Ingress filtering (BCP 38) is incompatible with Echo: implementations MUST either exempt Echo packets from ingress filtering or not enable Echo on filtered interfaces.
- The echo originator typically bypasses normal forwarding lookup to send a packet addressed to itself. This may require platform APIs that are not universally available.

## Algorithms

### TTL/Hop Limit Check

```
on receive Control packet:
    if bfd_auth_in_use:
        if ttl != 255:
            MAY discard (check MAY run before auth to save CPU)
    else:
        if ttl != 255:
            MUST discard
    continue to RFC 5880 reception procedure
```

### UDP Source Port Allocation

```
on session create:
    pick src_port in [49152, 65535] unique across local sessions if possible
    bind all Control packets in this session to that src_port
```

## Pitfalls

### Edge Cases

- **One session per data protocol:** dual-stack requires two sessions (one IPv4, one IPv6), not a single session carrying both.
- **Identical sessions on the same link:** MUST traverse distinct network-layer paths in both directions; the spec forbids multiple sessions sharing a single L3 path.
- **Tunnels that decrement TTL:** cannot use this mechanism. Only tunnels that preserve the inner TTL qualify.
- **Echo source address in the local subnet:** may trigger ICMP/ND Redirects from the peer, breaking echo.
- **Echo on a multi-access network:** the L2 destination MUST match the peer; L3 alone is not enough.
- **Changing peer source address on point-to-point:** the local system continues using the original destination; apps MAY be notified but BFD does not switch.

### Interop Considerations

- **Source port uniqueness** aids the peer's demultiplexing on the initial packet but is advisory. Ultimately Your Discriminator is authoritative.
- **TTL=255 is the default security perimeter** for single-hop BFD. Any firewall or overlay that rewrites TTL breaks this.
- **Same-interface constraint:** the session is bound to one interface. If routing moves traffic to a different interface, the session breaks by design.

### Security

- TTL=255 on both transmit and receive provides equivalent anti-spoofing to other infrastructure protocols (OSPF, ISIS) without cryptographic cost.
- Authentication remains optional; when combined with TTL=255, the check may be done before the cryptographic computation to reduce DoS exposure.
- Echo function does not need authentication per this RFC because the peer never parses the packet (it is looped at IP); the originator is responsible for matching returning packets to sessions.

## Compatibility

### With the Base Protocol

RFC 5881 is a companion to RFC 5880 and reuses all state machine, timer, discriminator, and FSM rules. It adds only encapsulation, port assignment, and TTL semantics.
