# RFC 8210: The Resource Public Key Infrastructure (RPKI) to Router Protocol, Version 1

## Meta

| Field | Value |
|-------|-------|
| RFC | 8210 |
| Title | The Resource Public Key Infrastructure (RPKI) to Router Protocol, Version 1 |
| Status | Standards Track |
| Date | September 2017 |
| Authors | R. Bush (Internet Initiative Japan), R. Austein (Dragon Research Labs) |
| Updates | RFC 6810 |
| Obsoletes | - |
| Obsoleted-by | - |

**Purpose:** Defines version 1 of the protocol by which routers receive validated prefix origin data (ROAs) and router keys from trusted RPKI caches, enabling BGP origin validation (RPKI-ROV).

**Scope:** Cache-to-router communication. Not BGP wire format itself, but the protocol that feeds RPKI validation data to BGP speakers.

## Changes from RFC 6810 (Version 0)

| Change | Section | Impact |
|--------|---------|--------|
| New Router Key PDU (Type 9) | 5.10 | Supports BGPsec router key distribution |
| Explicit timing parameters in End of Data PDU | 5.8, 6 | Cache controls Refresh, Retry, Expire intervals |
| Protocol version incremented from 0 to 1 | 5.1 | All PDUs carry version field = 1 |
| Protocol version negotiation added | 7 | Router and cache agree on version at connection start |
| New error code 8: Unexpected Protocol Version | 12 | Fatal error for version mismatch after negotiation |

## Wire Formats

### Common PDU Header

All PDUs share an 8-byte header:

```
 0          8          16         24        31
.-------------------------------------------.
| Protocol |   PDU    |   Session ID /      |
| Version  |   Type   |   Flags / zero      |
+-------------------------------------------+
|                                           |
|                  Length                    |
`-------------------------------------------'
```

| Field | Offset | Size | Type | Constraints |
|-------|--------|------|------|-------------|
| Protocol Version | 0 | 1 | uint8 | 1 (for this version) |
| PDU Type | 1 | 1 | uint8 | 0-10 (see PDU Types table) |
| Session ID / Flags / zero | 2 | 2 | uint16 | Depends on PDU type |
| Length | 4 | 4 | uint32 | Total PDU length including header |

### PDU Types

| Type | Name | Length | Direction | Bytes 2-3 Usage |
|------|------|--------|-----------|-----------------|
| 0 | Serial Notify | 12 | Cache->Router | Session ID |
| 1 | Serial Query | 12 | Router->Cache | Session ID |
| 2 | Reset Query | 8 | Router->Cache | zero |
| 3 | Cache Response | 8 | Cache->Router | Session ID |
| 4 | IPv4 Prefix | 20 | Cache->Router | zero |
| 6 | IPv6 Prefix | 32 | Cache->Router | zero |
| 7 | End of Data | 24 | Cache->Router | Session ID |
| 8 | Cache Reset | 8 | Cache->Router | zero |
| 9 | Router Key | variable | Cache->Router | Flags (byte 2), zero (byte 3) |
| 10 | Error Report | variable | Either | Error Code |

Note: PDU Type 5 is intentionally skipped (not assigned).

### Serial Notify (Type 0)

```
 0          8          16         24        31
.-------------------------------------------.
| Version  |  Type=0  |     Session ID      |
+-------------------------------------------+
|                 Length=12                  |
+-------------------------------------------+
|               Serial Number               |
`-------------------------------------------'
```

| Field | Offset | Size | Type |
|-------|--------|------|------|
| Session ID | 2 | 2 | uint16 |
| Length | 4 | 4 | uint32 (fixed: 12) |
| Serial Number | 8 | 4 | uint32 |

### Serial Query (Type 1)

Same format as Serial Notify. Router sends to request incremental updates since specified serial.

### Reset Query (Type 2)

```
 0          8          16         24        31
.-------------------------------------------.
| Version  |  Type=2  |        zero         |
+-------------------------------------------+
|                 Length=8                   |
`-------------------------------------------'
```

No payload beyond header. Router requests full data set.

### Cache Response (Type 3)

```
 0          8          16         24        31
.-------------------------------------------.
| Version  |  Type=3  |     Session ID      |
+-------------------------------------------+
|                 Length=8                   |
`-------------------------------------------'
```

Begins a sequence of zero or more payload PDUs followed by End of Data.

### IPv4 Prefix (Type 4)

```
 0          8          16         24        31
.-------------------------------------------.
| Version  |  Type=4  |        zero         |
+-------------------------------------------+
|                 Length=20                  |
+-------------------------------------------+
|  Flags   |  Prefix  |   Max    |   zero   |
|          |  Length   |  Length  |          |
+-------------------------------------------+
|              IPv4 Prefix (4 bytes)        |
+-------------------------------------------+
|         Autonomous System Number          |
`-------------------------------------------'
```

| Field | Offset | Size | Type | Constraints |
|-------|--------|------|------|-------------|
| Flags | 8 | 1 | uint8 | Bit 0: 1=announce, 0=withdraw |
| Prefix Length | 9 | 1 | uint8 | 0-32 |
| Max Length | 10 | 1 | uint8 | >= Prefix Length, 0-32 |
| zero | 11 | 1 | uint8 | Must be 0 |
| IPv4 Prefix | 12 | 4 | [4]byte | Network byte order |
| ASN | 16 | 4 | uint32 | 32-bit AS number |

### IPv6 Prefix (Type 6)

Same structure as IPv4 Prefix but with 16-byte prefix. Fixed length: 32 bytes.

| Field | Offset | Size | Type | Constraints |
|-------|--------|------|------|-------------|
| Flags | 8 | 1 | uint8 | Bit 0: 1=announce, 0=withdraw |
| Prefix Length | 9 | 1 | uint8 | 0-128 |
| Max Length | 10 | 1 | uint8 | >= Prefix Length, 0-128 |
| zero | 11 | 1 | uint8 | Must be 0 |
| IPv6 Prefix | 12 | 16 | [16]byte | Network byte order |
| ASN | 28 | 4 | uint32 | 32-bit AS number |

### End of Data (Type 7)

```
 0          8          16         24        31
.-------------------------------------------.
| Version  |  Type=7  |     Session ID      |
+-------------------------------------------+
|                 Length=24                  |
+-------------------------------------------+
|               Serial Number               |
+-------------------------------------------+
|              Refresh Interval             |
+-------------------------------------------+
|               Retry Interval              |
+-------------------------------------------+
|              Expire Interval              |
`-------------------------------------------'
```

| Field | Offset | Size | Type | Constraints |
|-------|--------|------|------|-------------|
| Session ID | 2 | 2 | uint16 | Must match Cache Response |
| Length | 4 | 4 | uint32 (fixed: 24) | |
| Serial Number | 8 | 4 | uint32 | Current cache serial |
| Refresh Interval | 12 | 4 | uint32 | 1-86400 seconds |
| Retry Interval | 16 | 4 | uint32 | 1-7200 seconds |
| Expire Interval | 20 | 4 | uint32 | 600-172800 seconds |

**Note:** In version 0 (RFC 6810), End of Data was only 12 bytes (no timing parameters). The three interval fields are new in version 1.

### Cache Reset (Type 8)

```
 0          8          16         24        31
.-------------------------------------------.
| Version  |  Type=8  |        zero         |
+-------------------------------------------+
|                 Length=8                   |
`-------------------------------------------'
```

Sent by cache when it cannot provide incremental update from the requested serial.

### Router Key (Type 9) -- NEW in Version 1

```
 0          8          16         24        31
.-------------------------------------------.
| Version  |  Type=9  |  Flags   |   zero   |
+-------------------------------------------+
|                  Length                    |
+-------------------------------------------+
|                                           |
|        Subject Key Identifier (20 bytes)  |
|                                           |
+-------------------------------------------+
|              AS Number (4 bytes)          |
+-------------------------------------------+
|                                           |
|       Subject Public Key Info (variable)  |
|                                           |
`-------------------------------------------'
```

| Field | Offset | Size | Type | Constraints |
|-------|--------|------|------|-------------|
| Flags | 2 | 1 | uint8 | Bit 0: 1=announce, 0=withdraw |

| Version  |  Type=10 |     Error Code      |

## Protocol Timing Parameters

| Parameter | Default | Min | Max | Purpose |
|-----------|---------|-----|-----|---------|
| Refresh Interval | 3600s (1 hour) | 1s | 86400s (1 day) | Time between normal cache polls |
| Retry Interval | 600s (10 min) | 1s | 7200s (2 hours) | Time between retries after failed poll |
| Expire Interval | 7200s (2 hours) | 600s (10 min) | 172800s (2 days) | Data validity without successful poll |

Constraint: Expire Interval MUST be larger than both Refresh Interval and Retry Interval.

## Protocol Version Negotiation

```
1. Router starts connection by sending Reset Query or Serial Query with version 1
2. If cache supports version 1: responds with version 1 PDUs
3. If cache only supports version 0:
   a. Cache may terminate (possibly with v0 Error Report) -> router MAY retry with v0
   b. Cache may reply with v0 response -> router MUST downgrade to v0 or terminate
4. If cache supports v1 but receives v0 query:
   cache MUST downgrade to v0 OR send v1 Error Code 4 and terminate
5. Once negotiated, version is stable for session lifetime
6. Version mismatch after negotiation -> Error Code 8 (Unexpected Protocol Version)
```

**During negotiation:** Router MUST ignore any Serial Notify PDUs received from cache. Caches SHOULD NOT send Serial Notify before negotiation completes.

## Protocol Sequences

### Start or Restart

```
Router -> Cache:  Reset Query (or Serial Query with previous Session ID)
Cache  -> Router: Cache Response (with Session ID)
Cache  -> Router: Payload PDUs (zero or more IPv4/IPv6 Prefix, Router Key)
Cache  -> Router: End of Data (with Serial Number + timing parameters)
```

### Typical Exchange (Incremental Update)

```
Cache  -> Router: Serial Notify (optional, with current serial)
Router -> Cache:  Serial Query (with last known serial + Session ID)
Cache  -> Router: Cache Response
Cache  -> Router: Payload PDUs (announcements and withdrawals since serial)
Cache  -> Router: End of Data (with new serial + timing parameters)
```

### No Incremental Update Available

```
Router -> Cache:  Serial Query
Cache  -> Router: Cache Reset
Router -> Cache:  Reset Query (after trying preferred caches)
Cache  -> Router: Cache Response + Payload PDUs + End of Data
```

### Cache Has No Data

```
Router -> Cache:  Serial Query or Reset Query
Cache  -> Router: Error Report (Code 2: No Data Available)
Router:           Retry periodically using Retry Interval
```

## Session ID and Serial Number Handling

| Concept | Rule |
|---------|------|
| Session ID scope | Specific to a particular protocol version; (version, session_id) pair identifies a session |
| Serial Number scope | Commensurate only within same (version, session_id) pair |
| Serial Number arithmetic | 32-bit unsigned, wraps from 2^32-1 to 0; comparison per RFC 1982 |
| Session ID generation | Pseudorandom, strictly increasing, or timestamp-based (e.g., POSIX time()) |
| Session ID mismatch | Terminate with Error Code 0, router flushes all data from that cache |
| Cross-version Session ID | Even if numerically same, sessions with different versions are separate |

### TLS Requirements

- Client routers MUST present client-side certificates
- Client certs MUST include subjectAltName with iPAddress identities
- Cache MUST check client IP against iPAddress identities
- Router MUST verify cache cert using subjectAltName dNSName per RFC 6125
- MUST NOT use Common Name (CN-ID) for authentication
- DNS names SHOULD NOT contain wildcard "*"

### SSH Requirements

- Cache MUST accept RSA, SHOULD accept ECDSA
- User authentication MUST be supported
- Client SHOULD verify cache public key

## Algorithms

### Router Startup

```
1. Choose most preferred cache from configured list
2. Establish transport connection
3. Send Reset Query (or Serial Query if resuming with known Session ID)
4. Version negotiation occurs implicitly via query/response versions
5. Receive Cache Response -> Payload PDUs -> End of Data
6. Store Session ID, Serial Number, and timing parameters
7. Set Refresh timer from End of Data
```

### Router Polling Loop

```
1. Wait for Refresh Interval (or Serial Notify receipt)
2. Send Serial Query with current (Session ID, Serial Number)
3. On success (Cache Response + data + End of Data):
   - Apply announcements and withdrawals
   - Update Serial Number from End of Data
   - Update timing parameters from End of Data
   - Restart Refresh timer
4. On Cache Reset:
   - Try more-preferred caches
   - If none: send Reset Query for full data
5. On Error (No Data Available):
   - Retry after Retry Interval
6. On connection failure:
   - Retry after Retry Interval
   - If Expire Interval exceeded: flush all data from this cache
```

### Cache Incremental Response

```
For Serial Query requesting serial S:
1. If S is too old or unknown: send Cache Reset
2. If S is current: send Cache Response + End of Data (empty update)
3. Otherwise, for each prefix/router key changed since S:
   - Merge all changes into minimal set
   - At most one withdrawal + one announcement per record
   - If changes cancel out: omit entirely
4. Send Cache Response
5. Send merged payload PDUs
6. Send End of Data with current serial + timing parameters
```

## Pitfalls

### Edge Cases

- **Serial Number wrap-around:** Comparison MUST use RFC 1982 arithmetic, not simple integer comparison
- **Session ID reuse by cache:** If cache erroneously reuses Session ID after restart, router may become confused; mitigate with persistent storage or clock-based Session IDs
- **Empty incremental update:** Cache Response followed immediately by End of Data is valid (no changes since requested serial)
- **Router Key SKI collision:** Multiple distinct public keys can hash to same SKI; compare full public key, not just SKI
- **Multiple caches:** Data from different caches is not commensurate; track data source separately per cache

### Interop

- **Version 0 caches:** Router must handle downgrade gracefully; v1 features (Router Key, timing parameters) unavailable in v0
- **End of Data length:** v0 End of Data is 12 bytes; v1 is 24 bytes (timing parameters added); implementations must check version before parsing
- **PDU Type 5 gap:** Type 5 is not assigned; implementations must handle gracefully
- **Error-on-error:** Never send Error Report in response to Error Report PDU; drop session instead

### Security

- **Cache spoofing:** Critical risk; cache feeds data directly to routing decisions; always use authenticated transport when possible
- **Last-hop trust:** Unlike inter-cache RPKI object security, cache-to-router link relies on transport security only
- **No Data Available is non-fatal:** Router should not flush existing data on this error; retry and use existing data until Expire Interval

## Compatibility

### With Version 0 (RFC 6810)

- All v0 PDU types valid in v1 (same type codes)
- Router Key (Type 9) reserved but unused in v0
- End of Data is longer in v1 (24 vs 12 bytes)
- Error Code 8 (Unexpected Protocol Version) new in v1
- Version negotiation allows interop between v0 and v1 implementations
