# RFC 8707 - Resource Indicators for OAuth 2.0

## Meta

| Field | Value |
|-------|-------|
| RFC | 8707 |
| Title | Resource Indicators for OAuth 2.0 |
| Status | Proposed Standard |
| Date | February 2020 |
| Authors | B. Campbell, J. Bradley, H. Tschofenig |
| Related | RFC 9728 (resource metadata), RFC 6749 (OAuth 2.0 core) |

**Purpose:** Adds a `resource` parameter to OAuth 2.0 authorization / token endpoints so the client can tell the authorization server which resource a requested access token is for. The AS encodes this as the `aud` claim on the issued JWT. Resource servers then REJECT any token whose `aud` does not match their own identity -- defeating the token-passthrough attack where a token stolen from one resource is replayed against another.

## Client Behavior (Section 2)

| Requirement | Item |
|-------------|------|
| `resource` parameter on authorization request | An absolute URI identifying the target resource. |
| MAY appear on token endpoint | When present, the AS uses it to select the audience. |
| Multiple `resource` values | MAY appear; AS policy decides. |

## Authorization Server Behavior (Section 3)

| Requirement | Item |
|-------------|------|
| Validate `resource` against registered metadata | AS decides which values it issues tokens for. |
| Set `aud` on the JWT | Access token's `aud` claim MUST be the canonical resource URL(s). |
| Reject unknown resources | AS MUST respond with `invalid_target` per RFC 8693 when it will not issue a token for the requested resource. |

## Resource Server Behavior (Section 5)

This is the rule the MCP resource server enforces:

| Requirement | Item |
|-------------|------|
| Verify `aud` on every token | MUST reject if `aud` does not match the resource's own canonical URL. |
| Canonicalisation | Compare after RFC 6749 Section 2.3.1 canonicalisation: scheme lowercase, default ports elided, no fragment, no query, no trailing slash. |
| Multiple audiences | If `aud` is a JSON array, at least one entry MUST match; array-shape decoding is required. |

## Canonical Resource URL (Section 2)

| Rule | Detail |
|------|--------|
| scheme | lowercase, typically `https` |
| host | lowercase, IDN normalised via IDNA / NFC |
| port | default for scheme elided (443 for https, 80 for http) |
| path | MAY be present; preserved as-is |
| fragment | MUST be absent |
| query | MUST be absent from the canonical form |
| trailing slash | operator decision; MUST be consistent on both sides |

## Ze Implementation

- Audience comparison: `internal/component/mcp/jwt.go` `audClaim.Matches` exact-match on string or array-form `aud`.
- Operator config: `environment.mcp.oauth.audience` is the canonical resource URL. MUST be set explicitly; the resource server never derives it from `r.Host`.
- Rejection path: `oauthAuthenticator.mapVerifyError` maps `errJWTAudienceMismatch` to `error="invalid_token", error_description="invalid audience"` per RFC 6750 vocabulary.

## Key Security Property

Without RFC 8707: a token issued for resource A can be replayed against resource B if both trust the same AS. B has no way to tell the token was not meant for it.

With RFC 8707: B validates `aud == B`. A token issued for A has `aud = A`. Replaying it against B fails closed.
