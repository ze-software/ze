# RFC 1334 - PPP Authentication Protocols

## Meta

| Field | Value |
|-------|-------|
| RFC | 1334 |
| Title | PPP Authentication Protocols |
| Status | Proposed Standard |
| Date | October 1992 |
| Author | B. Lloyd (L&A), W. Simpson (Daydreamer) |
| Obsoletes | - |
| Updates | - |
| Obsoleted-by (partial) | RFC 1994 for CHAP only -- PAP remains defined here |

**Purpose:** Defines two PPP authentication protocols carried over the
PPP link during the Authentication phase: PAP (Password Authentication
Protocol) and the original CHAP (Challenge Handshake Authentication
Protocol). CHAP has been obsoleted and replaced by RFC 1994; PAP is
still defined here and still deployed.

**Scope:** PAP runs during the PPP Authentication phase (between LCP
Opened and the first NCP). It is a simple two-way handshake where the
peer transmits an identity and cleartext password and the authenticator
replies Ack or Nak. The exchange happens exactly once after link
establishment unless a new LCP negotiation renegotiates the
Authentication-Protocol.

## Relationship to PPP

PAP is selected via the PPP LCP Authentication-Protocol Configuration
Option (RFC 1661 option type 3). The Authentication-Protocol value is
`0xC023` (no Algorithm field, unlike CHAP). LCP must reach Opened before
PAP begins. PPP frames carrying PAP packets use protocol number `0xC023`
in the PPP Information field. Two-way PAP (both sides authenticate)
requires the option in both directions; ze's LNS role authenticates the
peer, not vice versa.

## Wire Formats

### PAP Packet (Section 2.1)

```
+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
|     Code      |  Identifier   |            Length             |
+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
|    Data ...
+-+-+-+-+
```

| Field | Size | Meaning |
|-------|------|---------|
| Code | 1 octet | PAP packet type (see table below) |
| Identifier | 1 octet | Matches requests to replies |
| Length | 2 octets | Total packet length including Code/Identifier/Length/Data |
| Data | 0+ octets | Format depends on Code |

Octets outside the range of the Length field are Data Link Layer padding
and are ignored on reception.

### Code values (Section 2.1)

| Code | Meaning |
|------|---------|
| 1 | Authenticate-Request |
| 2 | Authenticate-Ack |
| 3 | Authenticate-Nak |

### Authenticate-Request Packet (Section 2.2)

```
+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
|     Code      |  Identifier   |            Length             |
+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
| Peer-ID Length|  Peer-ID ...
+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
| Passwd-Length |  Password ...
+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
```

| Field | Size | Meaning |
|-------|------|---------|
| Code | 1 octet | 1 Authenticate-Request |
| Identifier | 1 octet | Changed on each send; retransmissions reuse the Identifier |
| Length | 2 octets | Total packet length including header |
| Peer-ID Length | 1 octet | Length of Peer-ID in octets |
| Peer-ID | Peer-ID Length octets | Name of the peer being authenticated; no NUL termination |
| Passwd-Length | 1 octet | Length of Password in octets |
| Password | Passwd-Length octets | Cleartext password; no NUL termination |

Peer-ID and Password are transmitted in the clear; PAP has no
cryptographic protection. Confidentiality relies entirely on the lower
layer (e.g. IPsec or the physical link).

### Authenticate-Ack and Authenticate-Nak (Section 2.3)

```
+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
|     Code      |  Identifier   |            Length             |
+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
|  Msg-Length   |  Message ...
+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
```

| Field | Size | Meaning |
|-------|------|---------|
| Code | 1 octet | 2 Ack, 3 Nak |
| Identifier | 1 octet | Copied from the Authenticate-Request which caused this reply |
| Length | 2 octets | Total packet length including header |
| Msg-Length | 1 octet | Length of Message in octets |
| Message | Msg-Length octets | Implementation-defined, human-readable; may be empty |

Message is advisory; it MUST NOT affect operation of the protocol.
Recommended character range is ASCII 32-126.

## Encoding Rules

| Aspect | Rule |
|--------|------|
| Byte order | Network (big-endian) for Length |
| Length | Two octets, covers Code + Identifier + Length + Data |
| Identifier | One octet. Changed on every new Authenticate-Request; retransmissions reuse the same Identifier |
| Peer-ID Length, Passwd-Length, Msg-Length | One octet each, caps each field at 255 octets |
| Padding | Data beyond the Length field is Data Link Layer padding and ignored |

## Validation

| Check | Action on failure |
|-------|-------------------|
| Length < 4 or beyond frame | Silently discard |
| Unknown Code | Silently discard |
| Peer-ID Length + Peer-ID runs past Length | Silently discard |
| Passwd-Length + Password runs past Length | Silently discard |
| Ack/Nak received without outstanding Request | Silently discard |
| Authenticate-Request received outside Authentication phase | Silently discard |

## MUST Requirements

**Tx (peer -- the one being authenticated):**
- "After the Link Establishment phase is complete, the Peer repeatedly sends an id/password pair to the authenticator until authentication is acknowledged or the connection is terminated." (Section 2)
- "Authentication-Request is used to begin the PAP protocol" (Section 2.2)

**Tx (authenticator):**
- "If the id/password pair received in an Authenticate-Request is both recognizable and acceptable, then the authenticator MUST transmit a PAP packet with the Code field set to 2 (Authenticate-Ack)." (Section 2.3)
- "If the id/password pair received in an Authenticate-Request is not recognizable or acceptable, then the authenticator MUST transmit a PAP packet with the Code field set to 3 (Authenticate-Nak), and SHOULD take action to terminate the link." (Section 2.3)

**Identifier:**
- "The Identifier field MUST be changed each time an Authenticate-Request packet is issued." (Section 2.2)
- Ack/Nak "Identifier field MUST be copied from the Identifier field of the Authenticate-Request which caused this reply." (Section 2.3)

**Config:**
- "If authentication of the link is desired, an implementation MUST specify the Authentication-Protocol Configuration Option during Link Establishment phase." (Section 1)

## SHOULD / MAY

- "PAP is not a strong authentication method" -- Section 1 calls out that passwords are sent in clear and suggests CHAP (RFC 1994) when stronger protection is needed.
- "The authenticator SHOULD take action to terminate the link" on Nak (Section 2.3) -- the peer is expected to back off or reconfigure.
- Message field MAY be empty; if present, its content is implementation-defined and MUST NOT drive protocol decisions (Section 2.3).
- Implementations MAY retry Authenticate-Request at implementation-defined intervals; RFC 1334 does not define a retry counter (Section 2.2).

## Two-Way Handshake

1. After LCP reaches Opened with Authentication-Protocol = `0xC023`, the peer sends an Authenticate-Request (Code=1) containing Peer-ID and cleartext Password.
2. The authenticator verifies the credentials against its local store (or forwards them to an external service such as RADIUS). It replies with Authenticate-Ack (Code=2) on success or Authenticate-Nak (Code=3) on failure.
3. On Ack, PPP advances to the Network-Layer Protocol phase. On Nak, the authenticator SHOULD tear the link down via LCP Terminate-Request.

The peer is the sole initiator; the authenticator never sends Code=1. This is the opposite of CHAP, where the authenticator sends the first packet.

## Error Handling

| Situation | Action |
|-----------|--------|
| No Ack/Nak within the peer's timeout | Peer retransmits Authenticate-Request reusing the same Identifier; RFC 1334 defines no retry cap (Section 2.2) |
| Malformed Authenticate-Request | Silently discard; authenticator-side logging recommended |
| Ack/Nak received before any Request was sent | Silently discard |
| Authenticate-Request arrives outside Authentication phase | Silently discard |
| Nak received | Peer SHOULD abandon the link or renegotiate LCP authentication options |

## Timers and Counters

| Name | Value | Source |
|------|-------|--------|
| Authenticate-Request retransmit timer | Implementation-defined | RFC 1334 does not mandate a value |
| Max-Authenticate-Request (retry counter) | Implementation-defined | Not part of the RFC |
| Authentication-phase timeout | Set by the authenticator; RFC 1661 describes the phase but not a numeric value | LCP Restart precedent |

## Pitfalls

- Passwords are transmitted in cleartext. PAP requires an encrypted lower layer (IPsec, TLS-over-PPPoE, etc.) for any production use; otherwise any passive observer can capture credentials.
- The Peer-ID and Password fields are length-prefixed with a single octet each, so both fields are capped at 255 octets. Some legacy implementations silently truncate; ze MUST reject over-length values by refusing to encode them rather than truncating.
- The peer is the initiator. A server implementation that tries to send Authenticate-Request first will deadlock with any conformant client.
- PAP cannot begin unless LCP negotiated Authentication-Protocol = `0xC023`. A PAP frame arriving before that is a Protocol-Reject at LCP, not a PAP Nak.
- The Nak's Message field is advisory. Log it but tear the link down regardless; do not branch on its content.
- Section 2.1's silent-discard rule means malformed PAP frames do NOT get acknowledged in any form; a misbehaving peer will only learn it is broken via the auth-phase timeout.
- Some PPP stacks negotiate PAP even when CHAP is also available, simply because PAP negotiation requires one fewer packet. Explicit config is the only defense; ze SHOULD reject PAP on cleartext links when configured for CHAP-only.

## Security Considerations

Quoted from RFC 1334 Section 2:
- "PAP is not a strong authentication method. Passwords are sent over the circuit 'in the clear', and there is no protection from playback or repeated trial and error attacks."
- "The peer is in control of the frequency and timing of the attempts. Any implementation that includes a stronger authentication method (such as CHAP, described below) MUST offer to negotiate that method prior to PAP."
- "There is no mechanism within PAP itself to protect against playback attacks. Typically this protection is provided by the use of encrypted transmissions at lower layers."

Operational stance for ze:
- PAP is only acceptable when the link is protected at a lower layer (IPsec tunnel, physical security, etc.). The L2TP umbrella assumes IPsec when confidentiality is required.
- Ze never logs the Password field. The plaintext is passed to the auth handler (l2tp-auth plugin) and forgotten. Ack/Nak Message fields MAY be logged at info.
