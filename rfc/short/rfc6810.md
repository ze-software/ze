## Wire Formats

### Common PDU Header

All PDUs share an 8-byte header:

```
 0          8          16         24        31
.-------------------------------------------.
| Protocol |   PDU    |  Session ID /       |
| Version  |   Type   |  reserved / Error   |
|    0     |   0-10   |  Code               |
+-------------------------------------------+
|                                           |
|                  Length                   |
|                                           |
`-------------------------------------------'
```

| Field | Offset | Size | Type | Constraints |
|-------|--------|------|------|-------------|
| Protocol Version | 0 | 1 byte | uint8 | 0 for this RFC |
| PDU Type | 1 | 1 byte | uint8 | See PDU Types table |
| Session ID / Reserved / Error Code | 2 | 2 bytes | uint16 | Varies by PDU type |
| Length | 4 | 4 bytes | uint32 | Total PDU length including header |

### PDU Types

| Type | Name | Length | Direction | Header Byte 2-3 |
|------|------|--------|-----------|------------------|
| 0 | Serial Notify | 12 | Cache -> Router | Session ID |
| 1 | Serial Query | 12 | Router -> Cache | Session ID |
| 2 | Reset Query | 8 | Router -> Cache | reserved = zero |
| 3 | Cache Response | 8 | Cache -> Router | Session ID |
| 4 | IPv4 Prefix | 20 | Cache -> Router | reserved = zero |
| 6 | IPv6 Prefix | 32 | Cache -> Router | reserved = zero |
| 7 | End of Data | 12 | Cache -> Router | Session ID |
| 8 | Cache Reset | 8 | Cache -> Router | reserved = zero |
| 10 | Error Report | variable | Either | Error Code |

Note: PDU types 5 and 9 are unassigned in version 0.

## Encoding Rules

- All multi-byte fields: big-endian (network byte order)
- Protocol Version: always 0 for this RFC
- Reserved/zero fields: MUST be zero on transmission, MAY be ignored on receipt
- Flags lowest bit: 1 = announce, 0 = withdraw
- Max Length MUST NOT be less than Prefix Length
- Serial Number arithmetic per RFC 1982 (wraps from 2^32-1 to 0)

## Decoding Rules

1. Read 8-byte header: Version (1), Type (1), Session ID or reserved or Error Code (2), Length (4)
2. Validate Version == 0 (or send Error Code 4)
3. Validate PDU Type is known (or send Error Code 5)
4. Read remaining (Length - 8) bytes
5. For prefix PDUs: parse Flags, Prefix Length, Max Length, Prefix, ASN
6. For Error Report: parse encapsulated PDU length, PDU copy, error text length, error text

## Algorithms

### Router Startup / Session Establishment

```
1. Connect to most preferred cache (by configuration preference)
2. If router has unexpired data from previous session with same cache:
   a. Send Serial Query with previous Session ID and last known Serial Number
   b. If Cache Response received: process incremental updates
   c. If Cache Reset received: go to step 3
3. Otherwise (new session or cache reset):
   a. Send Reset Query
   b. Receive Cache Response
   c. Receive zero or more IPv4/IPv6 Prefix PDUs (all announce, Flags=1)
   d. Receive End of Data with new Session ID and Serial Number
   e. Replace all data from this cache with received dataset
```

### Incremental Update (Typical Exchange)

```
1. Cache sends Serial Notify (optional, unsolicited)
2. Router sends Serial Query with current Session ID and Serial Number
3. Cache responds:
   a. If incremental available:
      - Cache Response
      - Zero or more IPv4/IPv6 Prefix PDUs (announce and withdraw)
      - End of Data with new Serial Number
   b. If no changes: End of Data with same Serial Number
   c. If incremental unavailable: Cache Reset (router must Reset Query or switch cache)
   d. If no data at all: Error Report (Code 2: No Data Available)
4. Router updates its Serial Number to value in End of Data
```

### Session ID Handling

```
1. On Cache Response to Reset Query: adopt new Session ID
2. On Cache Response to Serial Query: verify Session ID matches
3. On mismatch: flush all data from this cache, treat as new session
4. Cache generates unique Session ID on startup (pseudo-random, clock-based, or persistent)
```

## Pitfalls

### Edge Cases

- **Serial Number wrap-around:** Serial Numbers wrap from 2^32-1 to 0. Comparison uses RFC 1982 serial number arithmetic, not simple integer comparison.
- **Session ID reuse after cache restart:** If cache erroneously reuses a Session ID, router may become confused. Cache SHOULD use persistent storage or clock-based generation to avoid collisions.
- **Duplicate announcements:** Cache MUST ensure at most one active PDU per unique {Prefix, Len, Max-Len, ASN}. Router SHOULD report Error 7 on duplicates.
- **Error Report for Error Report:** MUST NOT send. If received, SHOULD drop session (prevents infinite error loops).
- **Truncated erroneous PDU:** Allowed in Error Reports when the offending PDU is excessively long.

### Interop

- **Version negotiation:** This RFC defines version 0 only. RFC 8210 defines version 1 with additional PDUs. Unsupported version triggers Error Code 4.
- **Multiple caches:** Router MAY hold data from multiple caches but MUST track data by source. Data from different caches MUST NOT be distinguished during validation (RFC 6811).
- **Cache Reset recovery:** Router SHOULD try more preferred caches first before issuing Reset Query to current cache.

### Security

- **Transport trust:** Router cannot validate RPKI cryptography itself. The cache-to-router link is the weakest point. Transport authentication is critical.
- **Cache spoofing:** Without authentication, a rogue cache could inject invalid ROA data. Use SSH, TLS, TCP-AO, or restrict to trusted network.
- **DoS via routers:** Cache should authenticate connecting routers to prevent load from unauthorized clients.

## Compatibility

### With RFC 8210 (Version 1)

- RFC 8210 obsoletes this RFC and defines protocol version 1
- Version 1 adds Router Key PDU (Type 9) for BGPsec
- Version 1 adds Refresh, Retry, and Expire interval fields to End of Data PDU
- Version 0 implementations interoperate with version 1 caches that support both versions

### Feature Negotiation

- No capability negotiation within the protocol
- Version mismatch detected by Protocol Version field in PDU header
- Unsupported version triggers Error Code 4
