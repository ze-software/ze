# RFC 5575: Dissemination of Flow Specification Rules

## Meta

| Field | Value |
|-------|-------|
| RFC | 5575 |
| Title | Dissemination of Flow Specification Rules |
| Status | Standards Track |
| Date | August 2009 |
| Authors | P. Marques (Cisco), N. Sheth (Juniper), R. Raszuk (Cisco), B. Greene (Juniper), J. Mauch (NTT America), D. McPherson (Arbor Networks) |
| Obsoletes | - |
| Updates | - |
| Updated-by | RFC 7674 |
| Obsoleted-by | RFC 8955 |

**Purpose:** Defines a BGP NLRI encoding format for distributing traffic flow specifications, enabling routers to propagate granular traffic filtering rules (matching on src/dst prefix, protocol, ports, TCP flags, etc.) for DoS mitigation and VPN filtering.

**Scope:** AFI=1 (IPv4), SAFI=133 (Flow Specification) and SAFI=134 (VPNv4 Flow Specification)

**Errata:** RFC 5575 was obsoleted by RFC 8955, which incorporates all errata fixes and clarifications from RFC 7674 regarding redirect extended community encoding.

## Wire Formats

### Flow Specification NLRI

NLRI is encoded in MP_REACH_NLRI and MP_UNREACH_NLRI attributes per RFC 4760.

```
                     +------------------------------+
                     |    length (0xnn or 0xfn nn)  |
                     +------------------------------+
                     |    NLRI value  (variable)    |
                     +------------------------------+
```

| Field | Offset | Size | Type | Constraints |
|-------|--------|------|------|-------------|
| Length | 0 | 1-2 | variable | 1 byte if <240, 2 bytes otherwise |
| NLRI Value | 1 or 2 | variable | components | Ordered sequence of flow components |

**Length calculation:**
- If length < 240 (0xf0): encoded as single byte
- If length >= 240: encoded as 2 bytes with MSB nibble = 0xf, remaining 12 bits = length
- Example: 241 = 0xf0f1
- Maximum representable length: 4095

**Next-Hop:** For applications not requiring NEXT_HOP, encode as 0-octet length in MP_REACH_NLRI; ignore on receipt (Section 4).

### VPNv4 Flow Specification NLRI (SAFI 134)

```
+------------------+------------------------------+
|  Route Dist. (8) |    Flow Spec NLRI            |
+------------------+------------------------------+
```

| Field | Offset | Size | Type | Constraints |
|-------|--------|------|------|-------------|
| Route Distinguisher | 0 | 8 | RD | Per RFC 4364 encoding |
| Flow Specification | 8 | variable | components | Same as SAFI 133 |

**Length:** NLRI length includes both RD (8 bytes) and flow specification.

### Flow Component Types

Components MUST appear in strictly ascending numeric type order. Each component:

```
+------+------------------+
| Type |   Value          |
+------+------------------+
```

#### Type 1 - Destination Prefix

```
+------+--------+------------------+
| 0x01 | PfxLen |   Prefix         |
+------+--------+------------------+
```

| Field | Offset | Size | Type | Constraints |
|-------|--------|------|------|-------------|
| Type | 0 | 1 | uint8 | 0x01 |
| Prefix Length | 1 | 1 | uint8 | 0-32 bits |
| Prefix | 2 | 0-4 | bytes | ceil(PfxLen/8) bytes |

#### Type 2 - Source Prefix

```
+------+--------+------------------+
| 0x02 | PfxLen |   Prefix         |
+------+--------+------------------+
```

| Field | Offset | Size | Type | Constraints |
|-------|--------|------|------|-------------|
| Type | 0 | 1 | uint8 | 0x02 |
| Prefix Length | 1 | 1 | uint8 | 0-32 bits |
| Prefix | 2 | 0-4 | bytes | ceil(PfxLen/8) bytes |

#### Type 3 - IP Protocol

```
+------+---------------+
| 0x03 | [op, value]+  |
+------+---------------+
```

Uses numeric operator format. Values: 1 byte each.

#### Type 4 - Port (Source OR Destination)

```
+------+---------------+
| 0x04 | [op, value]+  |
+------+---------------+
```

Uses numeric operator format. Values: 1-2 bytes.

Evaluates to FALSE if:
- IP protocol is not TCP (6) or UDP (17)
- Packet is fragmented and not first fragment
- Transport header cannot be located

#### Type 5 - Destination Port

```
+------+---------------+
| 0x05 | [op, value]+  |
+------+---------------+
```

Uses numeric operator format. Values: 1-2 bytes.

#### Type 6 - Source Port

```
+------+---------------+
| 0x06 | [op, value]+  |
+------+---------------+
```

Uses numeric operator format. Values: 1-2 bytes.

#### Type 7 - ICMP Type

```
+------+---------------+
| 0x07 | [op, value]+  |
+------+---------------+
```

Uses numeric operator format. Values: 1 byte.
Evaluates to FALSE if IP protocol is not ICMP (1).

#### Type 8 - ICMP Code

```
+------+---------------+
| 0x08 | [op, value]+  |
+------+---------------+
```

Uses numeric operator format. Values: 1 byte.
Evaluates to FALSE if IP protocol is not ICMP (1).

#### Type 9 - TCP Flags

```
+------+------------------+
| 0x09 | [op, bitmask]+   |
+------+------------------+
```

Uses bitmask operator format. Values: 1-2 bytes.

- 1-byte value: matches byte 13 of TCP header (bits 8-15 of 4th word)
- 2-byte value: matches bytes 12-13, data offset field is "don't care"

Evaluates to FALSE if IP protocol is not TCP (6).

#### Type 10 - Packet Length

```
+------+---------------+
| 0x0a | [op, value]+  |
+------+---------------+
```

Uses numeric operator format. Values: 1-2 bytes.
Matches total IP packet length (excluding Layer 2, including IP header).

#### Type 11 - DSCP

```
+------+---------------+
| 0x0b | [op, value]+  |
+------+---------------+
```

Uses numeric operator format. Values: 1 byte.
Value encoding: 2 MSB bits zero, 6 LSB bits contain DSCP value.

#### Type 12 - Fragment

```
+------+------------------+
| 0x0c | [op, bitmask]+   |
+------+------------------+
```

Uses bitmask operator format.

Fragment bitmask bits:

```
                  0   1   2   3   4   5   6   7
                +---+---+---+---+---+---+---+---+
                |   Reserved    |LF |FF |IsF|DF |
                +---+---+---+---+---+---+---+---+
```

| Bit | Name | Meaning |
|-----|------|---------|
| 7 | DF | Don't Fragment |
| 6 | IsF | Is a Fragment |
| 5 | FF | First Fragment |
| 4 | LF | Last Fragment |
| 0-3 | Reserved | Must be zero |

### Numeric Operator Format

```
                  0   1   2   3   4   5   6   7
                +---+---+---+---+---+---+---+---+
                | e | a |  len  | 0 |lt |gt |eq |
                +---+---+---+---+---+---+---+---+
```

| Bit | Name | Meaning |
|-----|------|---------|
| 0 | e | End-of-list (set on last {op,value} pair) |
| 1 | a | AND (if set, AND with previous term; if unset, OR) |
| 2-3 | len | Value length = (1 << len) bytes |
| 4 | - | Reserved (must be 0) |
| 5 | lt | Less than |
| 6 | gt | Greater than |
| 7 | eq | Equal |

**Comparison combinations:**
- lt+eq: less than or equal
- gt+eq: greater than or equal
- lt+gt: not equal
- lt+gt+eq: always true (any value matches)

**Operator precedence:** AND has higher priority than OR.

**First operator:** AND bit should be unset in first operator byte of sequence.

### Bitmask Operator Format

```
                  0   1   2   3   4   5   6   7
                +---+---+---+---+---+---+---+---+
                | e | a |  len  | 0 | 0 |not| m |
                +---+---+---+---+---+---+---+---+
```

| Bit | Name | Meaning |
|-----|------|---------|
| 0 | e | End-of-list |
| 1 | a | AND bit |
| 2-3 | len | Value length = (1 << len) bytes |
| 4-5 | - | Reserved (must be 0) |
| 6 | not | NOT (logical negation of result) |
| 7 | m | Match mode |

**Match mode (m bit):**
- m=1: `(data & value) == value` (all bits must match)
- m=0: `(data & value) != 0` (any bit matches)

## Extended Community Actions

Default action: accept matching traffic.

```
        +--------+--------------------+--------------------------+
        | type   | extended community | encoding                 |
        +--------+--------------------+--------------------------+
        | 0x8006 | traffic-rate       | 2-byte AS#, 4-byte float |
        | 0x8007 | traffic-action     | bitmask                  |
        | 0x8008 | redirect           | 6-byte Route Target      |
        | 0x8009 | traffic-marking    | DSCP value               |
        +--------+--------------------+--------------------------+
```

All are non-transitive extended communities (type high bit set).

### Traffic-Rate (0x8006)

```
+--------+--------+-------------------+
| 0x8006 | AS (2) |   Rate (4)        |
+--------+--------+-------------------+
```

| Field | Offset | Size | Type | Constraints |
|-------|--------|------|------|-------------|
| Type | 0 | 2 | uint16 | 0x8006 |
| AS Number | 2 | 2 | uint16 | Informational; 2 LSB of 4-byte AS if applicable |
| Rate | 4 | 4 | IEEE 754 float | Bytes per second; 0 = discard all |

### Traffic-Action (0x8007)

```
+--------+-------------------+--------+
| 0x8007 |   Reserved (5)    | Flags  |
+--------+-------------------+--------+
```

Flags byte (byte 7, bits numbered from MSB):

```
                    40  41  42  43  44  45  46  47
                  +---+---+---+---+---+---+---+---+
                  |        reserved       | S | T |
                  +---+---+---+---+---+---+---+---+
```

| Bit | Name | Meaning |
|-----|------|---------|
| 47 | T | Terminal Action: if unset, stop evaluation; if set, continue to next rule |
| 46 | S | Sample: enable traffic sampling/logging |

### Redirect (0x8008)

```
+--------+--------------------------+
| 0x8008 |   Route Target (6)       |
+--------+--------------------------+
```

Same encoding as Route Target extended community (RFC 4360).
Traffic is redirected to VRF whose import policy lists this RT.

### Traffic-Marking (0x8009)

```
+--------+-------------------+--------+
| 0x8009 |   Zero (5)        |  DSCP  |
+--------+-------------------+--------+
```

| Field | Offset | Size | Type | Constraints |
|-------|--------|------|------|-------------|
| Type | 0 | 2 | uint16 | 0x8009 |
| Reserved | 2 | 5 | bytes | Must be zero |
| DSCP | 7 | 1 | uint8 | 6 LSB bits contain DSCP value |

## Encoding Rules

- Byte ordering: Network byte order (big-endian) throughout
- NLRI length: in octets (not bits like unicast NLRI)
- Flow components: concatenated with no separators, strict type ordering
- Operator sequences: last pair has end-of-list bit set
- Value lengths: determined by len field in operator (1, 2, 4, or 8 bytes)

## Decoding Rules

1. Parse NLRI length (1 or 2 bytes based on MSB nibble)
2. Parse flow components sequentially:
   - Read type byte
   - For prefix types (1,2): read length byte, then ceil(length/8) prefix bytes
   - For operator types: read {op, value} pairs until end-of-list bit
3. Validate component ordering (types must be ascending)
4. Unknown component types: cannot use for filtering, but propagate via BGP

## Validation

| Check | Valid | Invalid Action |
|-------|-------|----------------|
| NLRI length encoding | < 240 or 0xfnnn | Malformed NLRI |
| Component type ordering | Strictly ascending | Cannot apply, propagate anyway |
| Unknown component type | - | Cannot filter, propagate in BGP |
| Operator reserved bits | Zero | Undefined behavior |

### Flow Specification Feasibility Validation (Section 6)

A flow spec is feasible if and only if:

1. The originator of the flow spec matches the originator of the best-match unicast route for the embedded destination prefix
2. No more-specific unicast routes exist from a different neighboring AS than the best-match route

**Originator:** BGP ORIGINATOR_ID attribute (if present, for route reflection) or transport address of BGP peer.

## MUST Requirements

### Tx
- "Implementations wishing to exchange flow specification rules MUST use BGP's Capability Advertisement facility to exchange the Multiprotocol Extension Capability Code (Code 1)" (Section 4)
- Component types MUST appear in strictly ascending numeric order (Section 4)

### Rx
- "BGP implementations MUST also enforce that the AS_PATH attribute of a route received via the External Border Gateway Protocol (eBGP) contains the neighboring AS in the left-most position of the AS_PATH attribute" (Section 6)

### Validation
- Flow specification MUST be validated against unicast routing (Section 6)
- Originator matching MUST be performed for inter-AS flow specs

## SHOULD/MAY

- [SHOULD] "Standard BGP policy mechanisms, such as UPDATE filtering by NLRI prefix and community matching, SHOULD apply to the newly defined NLRI-type" (Section 3)
- [SHOULD] "Implementations SHOULD provide: A mechanism to log the packet header of filtered traffic" (Section 9)
- [SHOULD] "Implementations SHOULD provide: A mechanism to count the number of matches for a given flow specification rule" (Section 9)
- [MAY] User-defined community mappings MAY be used for platform-specific behaviors (Section 7)

| Condition | Detect How | Response | Code/Subcode |
|-----------|------------|----------|--------------|
| Unknown component type | Type byte not in [1-12] | Accept in BGP, cannot apply to filtering | N/A |
| Invalid component ordering | Type N follows Type M where N <= M | Cannot apply filter | N/A |
| Port component non-TCP/UDP | IP protocol != 6 or 17 | Component evaluates FALSE | N/A |
| ICMP component non-ICMP | IP protocol != 1 | Component evaluates FALSE | N/A |
| TCP flags non-TCP | IP protocol != 6 | Component evaluates FALSE | N/A |
| Fragment after first | Fragmented, not first fragment | Port components evaluate FALSE | N/A |

## Algorithms

### Flow Rule Ordering (Section 5.1)

Determines which rule has precedence when multiple rules match same traffic.

```
flow_rule_cmp (a, b)
{
    comp1 = next_component(a);
    comp2 = next_component(b);
    while (comp1 || comp2) {
        // component_type returns infinity on end-of-list
        if (component_type(comp1) < component_type(comp2)) {
            return A_HAS_PRECEDENCE;
        }
        if (component_type(comp1) > component_type(comp2)) {
            return B_HAS_PRECEDENCE;
        }

        if (component_type(comp1) == IP_DESTINATION || IP_SOURCE) {
            common = MIN(prefix_length(comp1), prefix_length(comp2));
            cmp = prefix_compare(comp1, comp2, common);
            // not equal, lowest value has precedence
            // equal, longest match has precedence
        } else {
            common =
               MIN(component_length(comp1), component_length(comp2));
            cmp = memcmp(data(comp1), data(comp2), common);
            // not equal, lowest value has precedence
            // equal, longest string has precedence
        }
    }

    return EQUAL;
}
```

**Key rules:**
1. Lower type number has precedence over absent type
2. For prefixes: lower IP value wins; if equal common prefix, more specific wins
3. For other types: memcmp() comparison; if equal prefix, longer string wins

### Flow Spec Feasibility Validation (Section 6)

```
validate_flowspec(flowspec, peer):
    dest_prefix = extract_destination(flowspec)  // may be empty/default

    best_unicast = lookup_best_unicast(dest_prefix)
    flowspec_origin = get_originator(flowspec, peer)
    unicast_origin = get_originator(best_unicast)

    if flowspec_origin != unicast_origin:
        return INFEASIBLE

    for each more_specific_route covering dest_prefix:
        if neighboring_as(more_specific_route) != neighboring_as(best_unicast):
            return INFEASIBLE

    return FEASIBLE
```

## Pitfalls

### Edge Cases

- **Empty destination prefix:** Flow spec may have no destination prefix (Type 1 absent), meaning it applies to all destinations - requires careful validation
- **Fragmented packets:** Port/TCP components evaluate FALSE for non-first fragments; may allow attack traffic through
- **IP options/ESP NULL:** Transport header location may fail; implementation-dependent behavior (Section 4)
- **Length encoding boundary:** 239 uses 1 byte, 240 uses 2 bytes; off-by-one errors common
- **VPNv4 length:** Must include 8-byte RD in NLRI length field
- **Component ordering:** Type 2 (source) cannot appear before Type 1 (destination) if both present

### Interop

- **RFC 8955 supersedes:** Production implementations should follow RFC 8955; this RFC has known issues
- **Redirect community encoding:** RFC 7674 clarified ambiguity in redirect extended community format
- **4-byte AS in traffic-rate:** Only 2 LSB bytes used; informational only
- **AS_PATH validation:** Enforcement of leftmost AS check is mandatory per Section 6, but optional in base BGP

### Security

- **DoS via filter rules:** Malicious peer could advertise rules dropping legitimate traffic
- **Validation bypass:** Without proper unicast route validation, attacker could inject rules for prefixes they don't own
- **Resource exhaustion:** Large number of flow specs can exhaust router ACL/filter resources
- **Fragmentation attacks:** Attackers may fragment to bypass port-based filters

## Compatibility

### Behavior with Non-Supporting Peers

- Flow spec requires MP-BGP capability negotiation (AFI=1, SAFI=133 or 134)
- Non-supporting peers will not receive flow spec updates
- No fallback mechanism; feature is negotiated or absent

### VPN Behavior Differences (Section 8)

| Aspect | SAFI 133 (IPv4) | SAFI 134 (VPNv4) |
|--------|-----------------|------------------|
| Validation | Required against unicast | Accepted by default from PE |
| Scope | Global table | Per-VRF |
| Propagation | Inter-AS validation | RT-based import/export |
| Default for remote PE | Requires validation | Accept by default |
