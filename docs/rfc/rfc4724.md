# RFC 4724: Graceful Restart Mechanism for BGP

## Meta

| Field | Value |
|-------|-------|
| RFC | 4724 |
| Title | Graceful Restart Mechanism for BGP |
| Status | Standards Track |
| Date | January 2007 |
| Updates | RFC 4271 (verified errata 7915) |
| Purpose | Minimize routing disruption during BGP restart by preserving forwarding state and deferring route deletion |

## Wire Formats

### Graceful Restart Capability

Type code: 64 (Capability)

```
+--------------------------------------------------+
| Restart Flags (4 bits)                           |
+--------------------------------------------------+
| Restart Time in seconds (12 bits)                |
+--------------------------------------------------+
| Address Family Identifier (16 bits)              |
+--------------------------------------------------+
| Subsequent Address Family Identifier (8 bits)    |
+--------------------------------------------------+
| Flags for Address Family (8 bits)                |
+--------------------------------------------------+
| ...                                              |
+--------------------------------------------------+
| Address Family Identifier (16 bits)              |
+--------------------------------------------------+
| Subsequent Address Family Identifier (8 bits)    |
+--------------------------------------------------+
| Flags for Address Family (8 bits)                |
+--------------------------------------------------+
```

| Field | Offset | Size | Type | Constraints |
|-------|--------|------|------|-------------|
| Restart Flags | 0 | 4 bits | bitfield | See Restart Flags below |
| Restart Time | 4 bits | 12 bits | uint | 0-4095 seconds |
| AFI | 2 + (n*4) | 2 bytes | uint16 | Per IANA AFI registry |
| SAFI | 4 + (n*4) | 1 byte | uint8 | Per IANA SAFI registry |
| Address Family Flags | 5 + (n*4) | 1 byte | bitfield | See AF Flags below |

Length calculation: `2 + (4 * num_afi_safi_tuples)` bytes

**Restart Flags (4 bits):**
```
 0 1 2 3
+-+-+-+-+
|R|Resv.|
+-+-+-+-+
```

| Bit | Name | Meaning |
|-----|------|---------|
| 0 (MSB) | R | Restart State - speaker has restarted |
| 1-3 | Reserved | MUST be zero on send, ignored on receive |

**Address Family Flags (8 bits):**
```
 0 1 2 3 4 5 6 7
+-+-+-+-+-+-+-+-+
|F|   Reserved  |
+-+-+-+-+-+-+-+-+
```

| Bit | Name | Meaning |
|-----|------|---------|
| 0 (MSB) | F | Forwarding State preserved for this AFI/SAFI |
| 1-7 | Reserved | MUST be zero on send, ignored on receive |

### End-of-RIB Marker

**IPv4 Unicast:**
UPDATE message with minimum length per RFC 4271:
- Withdrawn Routes Length: 0
- Total Path Attribute Length: 0
- No NLRI

**Other AFI/SAFI:**
UPDATE message containing only MP_UNREACH_NLRI (type 15) with:
- AFI: target address family
- SAFI: target sub-address family
- Withdrawn Routes: empty (length 0)

## Encoding Rules

- Restart Flags and Restart Time share first 2 bytes (big-endian)
- AFI/SAFI tuples: 0 to 63 tuples allowed
- Empty capability (no AFI/SAFI tuples): indicates support for receiving speaker procedures only, no forwarding state preservation

## Decoding Rules

1. Parse Restart Flags (4 bits) and Restart Time (12 bits) from first 2 bytes
2. Remaining bytes / 4 = number of AFI/SAFI tuples
3. For each tuple: parse AFI (2), SAFI (1), Flags (1)
4. If capability length < 2: malformed
5. If (capability length - 2) % 4 != 0: malformed

## Validation

| Check | Valid | Invalid Action |
|-------|-------|----------------|
| Capability length | >= 2, (len-2) % 4 == 0 | Treat as malformed capability |
| Reserved bits in Restart Flags | Any value | Ignore (Section 3) |
| Reserved bits in AF Flags | Any value | Ignore (Section 3) |
| Multiple GR capabilities | N/A | Use last instance only (Section 3) |
| Restart Time | 0-4095 | Always valid |
| AFI/SAFI tuple count | 0-63 | > 63 would exceed capability max length |

## State Machine Changes

RFC 4724 modifies the BGP FSM defined in RFC 4271 Section 8.2.2.

### Idle State Modification

**Original:**
> initializes all BGP resources for the peer connection

**Modified:**
> initializes all BGP resources for the peer connection, other than those resources required in order to retain routes according to section "Procedures for the Receiving Speaker"

### Established State - Collision Handling (Event 16/17)

**Without GR Capability:**
Track second connection until it sends OPEN (original behavior)

**With GR Capability (one or more AFI/SAFI received):**

| Step | Action |
|------|--------|
| 1 | Retain all routes per "Procedures for the Receiving Speaker" |
| 2 | Release all other BGP resources |
| 3 | Drop TCP connection associated with ESTABLISHED session |
| 4 | Initialize BGP resources (except route retention resources) |
| 5 | Set ConnectRetryCounter to zero |
| 6 | Start ConnectRetryTimer with initial value |
| 7 | Change state to Connect |

### Established State - NOTIFICATION/TcpConnectionFails (Event 18/24/25)

**On NOTIFICATION (Event 24/25) OR TcpConnectionFails without GR Capability:**

| Step | Action |
|------|--------|
| 1 | Set ConnectRetryTimer to zero |
| 2 | Delete all routes associated with connection |
| 3 | Release all BGP resources |
| 4 | Drop TCP connection |
| 5 | Increment ConnectRetryCounter by 1 |
| 6 | Change state to Idle |

**On TcpConnectionFails (Event 18) with GR Capability:**

| Step | Action |
|------|--------|
| 1 | Set ConnectRetryTimer to zero |
| 2 | Retain routes per "Procedures for the Receiving Speaker" |
| 3 | Release all other BGP resources |
| 4 | Drop TCP connection |
| 5 | Increment ConnectRetryCounter by 1 |
| 6 | Change state to Idle |

### State Machine Summary Table

| State | Event | GR Received? | Route Action | Next State |
|-------|-------|--------------|--------------|------------|
| Established | Event 16/17 (TCP up) | No | Track 2nd conn | Established |
| Established | Event 16/17 (TCP up) | Yes | Retain, mark stale | Connect |
| Established | Event 24/25 (NOTIF) | Any | Delete all | Idle |
| Established | Event 18 (TCP fail) | No | Delete all | Idle |
| Established | Event 18 (TCP fail) | Yes | Retain, mark stale | Idle |

## Algorithms

### Restarting Speaker Procedure

```
1. On restart:
   a. Retain forwarding state for routes in Loc-RIB
   b. Mark all retained routes as stale
   c. Continue forwarding (no stale differentiation)

2. Re-establish session:
   a. Set Restart State (R) bit in GR Capability
   b. Set Forwarding State (F) bit per AFI/SAFI if state preserved

3. After session established:
   a. Receive and process BGP messages
   b. Defer route selection per AFI/SAFI until:
      - End-of-RIB received from all peers (excluding R-bit peers, non-GR peers)
      - OR Selection_Deferral_Timer expires

4. After route selection:
   a. Update forwarding state
   b. Remove stale routes
   c. Advertise Adj-RIB-Out
   d. Send End-of-RIB for each AFI/SAFI
```

### Receiving Speaker Procedure

```
1. On TCP session termination (peer has GR Capability):
   a. Retain routes for all AFI/SAFI in peer's GR Capability
   b. Mark retained routes as stale
   c. Delete any previously stale routes from this peer
   d. Start Restart Time timer

2. On new connection while existing session (peer has GR Capability):
   a. Close previous TCP session (no NOTIFICATION)
   b. Accept new connection
   c. Retain routes, mark as stale

3. On session re-establishment:
   a. Check new GR Capability:
      - If no GR Capability: delete all stale routes
      - If AFI/SAFI missing: delete stale routes for that AFI/SAFI
      - If F bit not set: delete stale routes for that AFI/SAFI
   b. Do NOT set R bit unless self has restarted

4. During session:
   a. Replace stale routes with new updates
   b. On End-of-RIB for AFI/SAFI:
      - Delete remaining stale routes for that AFI/SAFI

5. On Restart Time expiry (session not re-established):
   a. Delete all stale routes from peer
```

### End-of-RIB Detection

```
For IPv4 Unicast:
  UPDATE where:
    - Withdrawn Routes Length == 0
    - Total Path Attribute Length == 0
    - No NLRI

For other AFI/SAFI:
  UPDATE containing:
    - MP_UNREACH_NLRI attribute only
    - Withdrawn Routes in MP_UNREACH_NLRI is empty
    - Correct AFI/SAFI for target address family
```

## Pitfalls

### Edge Cases

- **Consecutive restarts:** Must delete previously stale routes before marking current routes as stale (prevents stale route accumulation)
- **R-bit deadlock avoidance:** When R bit is set, peer MUST NOT wait for End-of-RIB before sending routes (prevents mutual waiting)
- **Empty GR Capability:** Valid - indicates support for receiving speaker procedures only, Restart Time is irrelevant
- **NOTIFICATION terminates GR:** Normal BGP procedures apply - no route retention on NOTIFICATION

### Interop

- **Non-GR peers:** Exclude from End-of-RIB wait list during selection deferral
- **R-bit peers:** Exclude from End-of-RIB wait list (they're also restarting)
- **Capability duplication:** Use only last instance if multiple GR capabilities received

### Security

- **DoS via spurious connections:** New connections can terminate existing sessions; mitigate with TCP MD5 (RFC 2385)
- **Route injection window:** Attacker connection replaced by legitimate peer reconnection (brief window)
- **Errata 6217 (held):** Potential capability downgrade via NOTIFICATION for permanent hijacking - consider in security-sensitive deployments

## Compatibility

### With Non-GR Peers

- No route retention on session failure
- Standard BGP behavior applies
- Can still send End-of-RIB marker (recommended even without GR)

### Feature Negotiation

- GR Capability exchanged in OPEN
- Per-AFI/SAFI granularity for forwarding state
- Absence of AFI/SAFI tuple = no forwarding preservation for that family
- Empty capability = receiving speaker support only

### Fallback Behavior

| Scenario | Behavior |
|----------|----------|
| Peer lacks GR Capability | Standard BGP (delete routes on session down) |
| Peer's GR lacks AFI/SAFI | Delete stale routes for that AFI/SAFI immediately |
| Peer's F bit = 0 | Delete stale routes for that AFI/SAFI immediately |
| NOTIFICATION received | Standard BGP (delete all routes) |
