## Task

EBGP AS-PATH prepending in the engine's `ForwardUpdate` path.

BGP mandates different treatment for routes forwarded to peers in the same AS (IBGP) vs.
different ASes (EBGP):

- **IBGP peers** (same AS): Forward raw bytes unchanged. AS-PATH MUST NOT be modified.
- **EBGP peers** (different AS): Prepend local ASN to AS-PATH (RFC 4271 Section 9.1.2).

The current `ForwardUpdate` in `reactor.go` forwards raw wire bytes unchanged to ALL matching
peers regardless of peer type. This violates RFC 4271 for EBGP sessions.

### Solution: Engine-Smart ForwardUpdate with Lazy EBGP Patch

The engine already knows each peer's `LocalAS` and `PeerAS` via `PeerSettings`, and already has
`IsEBGP()` / `IsIBGP()` methods. `ForwardUpdate` already makes per-peer decisions (zero-copy vs
re-encode, split vs no-split). Adding EBGP awareness is a natural extension.

Each `ReceivedUpdate` stores up to three wire versions:
1. **IBGP version**: Original raw payload (already `WireUpdate.Payload()`), backed by `poolBuf`.
2. **EBGP ASN4 version**: Lazily generated by rewriting AS-PATH with 4-byte ASN encoding into a pool buffer. Cached as `ebgpWireASN4`.
3. **EBGP ASN2 version**: Lazily generated with 2-byte ASN encoding (AS_TRANS for large ASNs). Cached as `ebgpWireASN2`.

Memory management: `RewriteASPath` takes a destination buffer (from `readBufPool4K`/`readBufPool64K`)
and writes the patched payload into it. `ReceivedUpdate` stores pool buffers as `ebgpPoolBuf4` and
`ebgpPoolBuf2`. `evictLocked` returns all pool buffers (`poolBuf`, `ebgpPoolBuf4`, `ebgpPoolBuf2`)
to the pool. Zero heap allocation on the hot path — same pool discipline as the original read path.

In `ForwardUpdate`:
- Scan matching peers to check if any are EBGP (tracking ASN4 vs ASN2 separately).
- If EBGP-ASN4 peers exist, generate `ebgpWireASN4` once. If EBGP-ASN2 peers exist, generate `ebgpWireASN2` once.
- Distribute: IBGP peers get original wire, EBGP peers get the version matching their ASN4 negotiation.

No new commands. No RR plugin changes. No interface changes. The `forward` command works
exactly as before — the engine is just smarter about what bytes it sends per-peer.

### AS-PATH Rewrite Using Existing Encoding

`RewriteASPath(dst, payload []byte, localASN uint32, srcAsn4, dstAsn4 bool) (int, error)` in new `wireu/aspath_rewrite.go`.

Uses the existing `attribute.ASPathIterator` to parse and `attribute.ASPath.WriteToWithASN4` to re-encode. No custom byte-level ASN encoding — all ASN encoding logic reuses the existing infrastructure that already handles segment splitting, AS_TRANS, and Extended Length.

**Algorithm:**

1. Parse UPDATE body layout to find AS-PATH attribute:
   ```
   [0:2]          wdLen (2 bytes)
   [2+wdLen:4+wdLen]  attrLen (2 bytes)  ← UPDATE after rewrite
   [4+wdLen:4+wdLen+attrLen]  path attributes
   [4+wdLen+attrLen:]  NLRI
   ```

2. Scan attrs to locate AS-PATH (type=2). Record: attribute start offset, header size (3 or 4 bytes depending on ExtLength flag), value offset, old value size.

3. Parse existing AS-PATH value using `attribute.ASPathIterator(value, srcAsn4)` → collect segments into `attribute.ASPath` struct.

4. Prepend `localASN` to the parsed ASPath:
   - If first segment is AS_SEQUENCE with count < 255: insert localASN at front of that segment's ASNs.
   - Otherwise: prepend new AS_SEQUENCE{localASN} segment.

5. Compute sizes:
   - `oldAttrWireSize` = attribute header + old value size (from step 2).
   - `newValueSize` = `newASPath.LenWithASN4(dstAsn4)` (uses existing method — handles segment splitting, ASN2/ASN4).
   - `newAttrWireSize` = attribute header size + `newValueSize`. Header may change size if Extended Length threshold crossed (value > 255 bytes).
   - `shift` = `newAttrWireSize - oldAttrWireSize`. Can be positive, zero, or negative (ASN4→ASN2 transcoding).

6. Write patched payload into `dst` (pool buffer):
   - Copy bytes before AS-PATH attribute.
   - Write new AS-PATH attribute header (flags + type + length).
   - Call `newASPath.WriteToWithASN4(dst, off, dstAsn4)` — reuses existing encoding.
   - Copy bytes after old AS-PATH attribute.
   - Update global `attrLen` at `dst[2+wdLen:4+wdLen]` by adding `shift`.
   - Return total bytes written.

7. If no AS_PATH exists in payload (Case D): insert a complete AS_PATH attribute before NLRI, using `attribute.WriteAttrTo` with the new ASPath. Shift = full attribute size.

**Why this approach:**
- Reuses `ASPath.WriteToWithASN4` which already handles AS_TRANS (23456), segment splitting at 255, and Extended Length headers.
- Naturally supports ASN4→ASN2 transcoding: the iterator parses with `srcAsn4`, the writer encodes with `dstAsn4`. The shift accounts for the encoding change.
- Two cached versions per ReceivedUpdate: `ebgpWireASN4` and `ebgpWireASN2`, since the shift differs.

**Buffer sizing:** worst case grows by ~9 bytes (new attribute, ASN4). A pool buffer from `readBufPool4K`/`readBufPool64K` always has room since the original payload fit in the same pool tier. ASN4→ASN2 transcoding can even shrink the output.

## Design Insights

- **Engine-smart vs plugin-smart**: The engine already has all per-peer information (LocalAS, PeerAS, negotiated caps) and already makes per-peer decisions in ForwardUpdate. Adding EBGP awareness there is a single if-branch, not a new command+interface+dispatch chain. Prefer enriching existing decision points over adding new commands.
- **Pool-backed EBGP wire**: `RewriteASPath` writes into a pool buffer (`readBufPool4K`/`readBufPool64K`), same pools as the original read path. Pool buffers are stored in `ReceivedUpdate.ebgpPoolBuf4`/`ebgpPoolBuf2` and returned via `ReturnReadBuffer` on cache eviction — same lifecycle as `poolBuf`. Zero heap allocation on the forwarding hot path.
- **RewriteASPath is idempotent from the cache's perspective**: The patched payload lives in a separate pool buffer; the original `WireUpdate.Payload()` is never modified. Zero-copy guarantees for the IBGP path are preserved.
- **EBGP wire preserves SourceCtxID**: The patched bytes have the same encoding context (same ASN4, ADD-PATH) as the original — only AS-PATH content changed. This means zero-copy works for EBGP peers with matching context too.
- **AS_TRANS (RFC 6793)**: When `asn4=false` and `localASN > 65535`, the engine should encode AS_TRANS=23456 in the 2-byte field. This is standard BGP4 behavior for ASN4-capable senders advertising to non-ASN4 peers.

## RFC Documentation

- RFC 4271 Section 9.1.2: "When a BGP speaker propagates a route which it has learned from another BGP speaker's UPDATE message, it shall modify the route's AS_PATH attribute based on the location of the BGP speaker to which the route will be sent: [...] if the BGP speaker is an EBGP speaker [...] it MUST prepend its own AS number as the last element of the sequence."
- RFC 4271 Section 5.1.2: AS_PATH construction rules: IBGP must NOT modify AS_PATH; EBGP prepends.
- RFC 6793 Section 4 (AS4): When advertising to non-AS4-capable peers, ASNs > 65535 map to AS_TRANS=23456 in the 2-octet AS_PATH.

## Implementation Summary

All six implementation steps completed. `RewriteASPath` in `wireu/aspath_rewrite.go` handles byte-level AS-PATH patching using existing `ASPathIterator`/`ASPath.WriteToWithASN4` infrastructure. `ReceivedUpdate.EBGPWire()` provides lazy, mutex-protected dual-cache (ASN4/ASN2) with pool-backed buffers. `ForwardUpdate` in `reactor.go` now scans matching peers for EBGP variants, pre-generates patched wires once, then selects the correct wire version per peer (IBGP=original, EBGP=patched) before the existing zero-copy/re-encode/split logic.

Key deviation: `ebgpLocalAS` is sourced from `peer.Settings().LocalAS` during the EBGP scan (not `a.r.config.LocalAS`) to avoid nil-config dependency in tests and better support per-peer LocalAS.
