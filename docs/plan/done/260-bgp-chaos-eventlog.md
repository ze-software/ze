## Task

Add a structured, replayable event log to `ze-bgp-chaos` that records every action, BGP message, state transition, and validation result with sequence numbers and simulated timestamps.

The event log serves two purposes:
1. **Post-mortem analysis** — understand exactly what happened in a chaos run
2. **Replay artifact** — `ze-bgp-chaos --replay events.json` re-executes the same scenario against the validation model without needing a running Ze instance (enables unit testing the validation model against recorded runs)

**Scope:**
- NDJSON event log format with global sequence numbers
- Event types covering: lifecycle, BGP messages, chaos actions, validation results, convergence
- `--replay <file>` mode that feeds recorded events into the validation model
- Determinism check: same seed produces identical event log (byte-for-byte)
- Log diffing tool: compare two event logs to find divergence point

**Relationship to DST:**
This is the external (black-box) version of DST Section 8's `EventLog`. When Ze later gains internal instrumentation (clock/network abstraction), internal and external logs can be correlated by timestamp and sequence to give a full picture of what happened on both sides of the TCP connection.

## Data Flow (MANDATORY)

### Entry Point
- Events generated by peer simulators, chaos scheduler, and validation engine
- Each event gets a global sequence number and timestamp at the reporter

### Transformation Path
1. Event generated (peer/chaos/validation goroutine)
2. Sent on event channel to reporter
3. Reporter assigns sequence number (atomic counter) and timestamp (relative to start)
4. Event serialized to NDJSON line and written to log file
5. On replay: log file read line by line, events fed to validation model

### Integration Points
- Phase 5 reporter (extends or replaces JSON log component)

- Scenario generator (replay verifies determinism of scenario, not of TCP)

### Architectural Verification
- [ ] Event log is append-only (no seeking, no rewriting)
- [ ] Sequence numbers assigned by single goroutine (no races)
- [ ] Replay mode shares validation model code with live mode (no duplication)
- [ ] Log format is self-describing (includes version, seed, CLI flags in header)

## Implementation Summary

### What Was Implemented

1. **Enhanced JSONLog format** (`report/jsonlog.go`): Added `JSONLogConfig` struct, NDJSON header line with `record-type: "header"`, version, seed, peers, chaos-rate, start-time. Events now carry `record-type: "event"`, `seq` (monotonic), `time-offset-ms` (relative to start).

2. **Replay engine** (`replay/replay.go`): `Run(r, w) int` reads NDJSON, parses header, creates validation Model/Tracker/Convergence, routes events through same switch logic as live `EventProcessor`, runs `Check()`, prints summary via `report.Summary.Write()`.

3. **Diff tool** (`replay/diff.go`): `Diff(r1, r2, w) int` compares two logs line-by-line on event-type/peer-index/prefix (skips time-offset-ms). Reports first divergence with 5 lines of context. Returns 0=identical, 1=divergence, 2=error.

4. **CLI wiring** (`main.go`): Renamed `--event-file` → `--event-log`. Added `--replay` and `--diff`/`--diff2` flags. Updated usage text with Replay section.

### Design Decisions

- **Inlined EventProcessor logic in replay**: `EventProcessor` is in `package main` and can't be imported by `replay` package. The validation primitives (Model, Tracker, Convergence, Check) are fully reused; only the ~25-line switch dispatch is inlined.
- **`writeErr()` helper**: Gosec requires checking `fmt.Fprintf` returns, but on error paths there's nothing useful to do. Created a helper that checks but discards the error (best-effort reporting).
- **Time-offset-ms instead of absolute timestamps**: Enables deterministic log comparison — same seed produces same relative offsets regardless of wall-clock start time.
