# `make([]byte, N)` Pool-Migration Audit -- 2026-04-16

Companion to `make-pool-2026-04-16.csv` (249 classified hits).

Driven by the two design rules added this session to
`.claude/rules/design-principles.md`:

1. **No `make` where pools exist** -- any variable-size `make([]byte, N)`
   on a wire-facing path must come from a bounded pool.
2. **Pool strategy by goroutine shape** -- sanctions two shapes:
   single-backing ring vs `sync.Pool` seeded for peak.

TACACS+ migration in this session is the worked example, but the audit
shows residual `make` on its own send path too (see below).

## Key Findings

### 1. TACACS+ pool migration is incomplete

The session's work pooled the wire send/recv buffer (`client.go::trySend`)
but the body construction that feeds it still allocates on every call.

| Site | Issue |
|------|-------|
| `authen.go:61` | `body := make([]byte, 8+userLen+portLen+remLen+dataLen)` -- every `Authenticate()` |
| `authen.go:116` | `replyData := make([]byte, dataLen)` -- every reply unmarshal |
| `author.go:62` | `body := make([]byte, 8+argCount+varLen)` -- every authz request |
| `author.go:136` | `respData := make([]byte, dataLen)` -- every authz response |
| `acct.go:63` | `body := make([]byte, 9+argCount+varLen)` -- every accounting request |
| `acct.go:120` | `replyData := make([]byte, dataLen)` -- every accounting reply |
| `packet.go:105` | `input := make([]byte, 4+len(key)+2)` in `Encrypt()` -- every packet MD5 pad |
| `packet.go:175` | `Marshal()` -- test-only (confirmed: only test callers) |
| `packet.go:197` | `UnmarshalPacket()` -- test-only (live path uses `UnmarshalPacketHeader`) |

The session's `client.go:199` claim "no wire path ever calls
`make([]byte, N)` for variable-size N" is overstated: the body marshal
paths called before `sendToServers` still do.

Fix shape: add `MarshalBinaryInto(buf []byte) (int, error)` to
`AuthenStart`, `AuthorRequest`, `AcctRequest` writing into a sub-slice of
the already-pooled buf. Encrypt input can be stack-allocated (`[256]byte`
scratch since key ~= shared secret ~= 16-64 bytes).

### 2. Plugin RPC is unpooled on every message

`pkg/plugin/rpc/message.go` and `framing.go` allocate a fresh buffer for
every emitted RPC. This is the hot path: plugins stream UPDATE events
continuously via this encoder.

| Site | Allocates |
|------|-----------|
| `message.go:102,117,127,137,143` | one `make([]byte, 0, ...)` per outbound envelope |
| `framing.go:41,76` | `len(msg)` or `len(msg)+1` copy for newline-framing |

Fix shape: `sync.Pool` seeded for peak (matches TACACS+). Shared by
engine-side and plugin-side so both sides of the pipe allocate at most
once at startup. Envelope max is bounded by the 10MB `MaxMessageSize`
ceiling -- pick a pool slot size between 4KiB and 64KiB and overflow to
a one-shot `make` for rare large messages.

### 3. BGP forward_build.go allocates per modified payload

`reactor/forward_build.go` constructs the modified-egress payload when an
outbound filter rewrites attributes. Per the design principle "copy only
on modify," this buffer MUST come from the destination peer's Outgoing
Peer Pool, not `make`.

| Site | Allocates |
|------|-----------|
| `forward_build.go:122` | `buf = make([]byte, needSize)` -- grow case |
| `forward_build.go:303,343,403,411,417,425` | `result`, `unreachVal`, `unreachAttr` slices |

`reactor/forward_pool.go` already exists with the single-backing-ring
shape. These call sites should acquire from the outgoing pool, not
allocate. This is the highest-leverage single migration in BGP.

### 4. BFD Verify() allocates twice per received packet

`bfd/auth/sha1.go:144,146` allocate two scratch slices per Verify() call.
BFD timers are sub-second, so this is hundreds of allocations per session
per second.

Fix shape: per-session pre-allocated scratch (BFD already has
`packet/pool.go`). A two-slot slice on the session struct is cheaper than
`sync.Pool` given the one-goroutine-per-session topology.

### 5. L2TP reliable send allocates per control message

`l2tp/reliable.go:311` allocates `bytes := make([]byte, length)` on every
outbound control message. L2TP control rate is low (session setup,
hellos) but still not one-shot.

Fix shape: use the existing `l2tp/pool.go` (already sync.Pool-seeded).
Thread the pool buf through `send()`.

### 6. BGP `update_build*.go` slips through the encoding hook

`.claude/hooks/block-encoding-alloc.sh` scopes to file names matching
`update_build`, `message/pack`, `reactor_wire` AND excludes `result :=
make` from the check (line 61). `update_build.go:496` `result :=
make([]byte, totalSize)` is accepted by the hook but is one of the
highest-frequency allocations in the tree -- one per outbound UPDATE.

Recommendation (outside this audit's pool scope): tighten the hook's
`result :=` exemption or add a comment on each surviving call explaining
why it is not from a pool.

## Pool Shape Decisions

For the subsystems that need migration, shape assignment per the rule
table in `design-principles.md`:

| Subsystem | Goroutine shape | Pool shape |
|-----------|-----------------|-----------|
| TACACS+ | multiple goroutines share client (auth + authz + acct) | `sync.Pool` seeded for peak -- already exists, thread deeper |
| plugin-rpc (engine + plugin sides) | multi-goroutine (event loop + RPC requests) | `sync.Pool` seeded for peak |
| BFD (per-session Verify) | one goroutine per session | per-session scratch on the session struct |
| L2TP reliable | single tunnel goroutine | existing `l2tp/pool.go` (sync.Pool already) |
| PPP session_run | single session goroutine | per-session scratch OR existing pool |
| BGP forward_build | reactor goroutine + forward workers | existing single-backing ring (`forward_pool.go`) |
| BGP update_build result allocs | reactor goroutine | `buildBufPool` (already exists, needs wiring) |
| BGP BMP sender | per-peer monitoring goroutine | `sync.Pool` or per-peer scratch (low rate, either works) |

## Shared `bufpool` Package Decision (Phase 3)

Three subsystems want the same `sync.Pool`-seeded-for-peak shape:
TACACS+, plugin-rpc, potentially BMP. The TACACS+ constructor is ~50
lines.

**Recommendation: promote `internal/component/tacacs/pool.go` to
`internal/core/bufpool/` with signature**

```go
package bufpool
type Pool struct { ... }
func New(peakN, size int, name string) *Pool   // name for metrics
func (p *Pool) Get() []byte
func (p *Pool) Put(b []byte)
```

BGP's single-backing ring stays local (`reactor/forward_pool.go`) because
it is coupled to peer-pool index semantics and cannot generalise without
losing the `backing[i*size:(i+1)*size:(i+1)*size]` guarantee.
