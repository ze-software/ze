## Task

Implement a REST/HTTP transport over the shared API engine defined in `spec-api-0-umbrella.md`.
The transport exposes all engine commands as RESTful endpoints, generates an OpenAPI 3.1
specification at startup, and serves interactive API docs (Swagger UI or Redoc).

This is a thin adapter: all logic lives in the engine. The REST layer handles HTTP routing,
JSON marshaling, SSE streaming, CORS, and OpenAPI doc serving.

## Design: REST Endpoints

### Generic Command Execution

| Method | Path | Engine call | Description |
|--------|------|-------------|-------------|
| GET | /api/v1/commands | ListCommands | List all available commands with metadata |
| GET | /api/v1/commands/{path} | DescribeCommand | Metadata for one command |
| POST | /api/v1/execute | Execute | Execute any command |
| GET | /api/v1/execute/stream | Stream (SSE) | Stream command output via Server-Sent Events |
| GET | /api/v1/complete | Complete | Tab-completion for partial commands |

POST /api/v1/execute request body: JSON object with "command" (string) and "params" (object).
Response body: JSON object with "status", "data", "error".

GET /api/v1/execute/stream: query params "command" and optional params. Response is SSE
(Content-Type: text/event-stream), each event carries a JSON Response.

### RESTful Convenience Routes

These map to generic Execute internally -- no separate engine calls.

| Method | Path | Maps to command | Description |
|--------|------|-----------------|-------------|
| GET | /api/v1/peers | bgp summary | List all peers with state |
| GET | /api/v1/peers/{name} | peer {name} detail | Peer detail |
| DELETE | /api/v1/peers/{name} | peer {name} teardown | Terminate peer session |
| POST | /api/v1/peers/{name}/refresh | peer {name} refresh | Route refresh |
| GET | /api/v1/rib/{family} | rib routes {family} | Routes by family |
| GET | /api/v1/rib/{family}/best | rib best {family} | Best routes by family |
| GET | /api/v1/system/version | show version | Version info |
| GET | /api/v1/system/status | daemon status | Daemon status |
| POST | /api/v1/system/reload | daemon reload | Config reload |

### Config Session Routes

| Method | Path | Engine call | Description |
|--------|------|-------------|-------------|
| GET | /api/v1/config/running | ConfigGetRunning | Running config |
| POST | /api/v1/config/sessions | ConfigEnter | Start candidate session, returns session ID |
| PUT | /api/v1/config/sessions/{id} | ConfigSet | Set config path (body: path + value) |
| DELETE | /api/v1/config/sessions/{id}/{path} | ConfigDelete | Delete config path |
| GET | /api/v1/config/sessions/{id}/diff | ConfigDiff | Preview pending changes |
| POST | /api/v1/config/sessions/{id}/commit | ConfigCommit | Apply changes (body: optional message) |
| DELETE | /api/v1/config/sessions/{id} | ConfigDiscard | Discard candidate |

### Documentation Routes

| Method | Path | Description |
|--------|------|-------------|
| GET | /api/v1/openapi.json | OpenAPI 3.1 specification (auto-generated) |
| GET | /api/v1/docs | Interactive API documentation (Swagger UI or Redoc) |

### HTTP Details

| Concern | Approach |
|---------|----------|
| Content-Type | application/json for all responses (except SSE and docs) |
| Error responses | JSON with "status": "error" and "error": "message", appropriate HTTP status code |
| Auth | Bearer token in Authorization header; 401 on missing/invalid, 403 on insufficient permissions |
| CORS | Configurable via YANG (api.rest.cors-origin); preflight handled |
| Versioning | /api/v1/ prefix; future versions add /api/v2/ |
| Streaming | Server-Sent Events (text/event-stream) for Stream endpoint |
| Pagination | Query params "limit" and "offset" for list endpoints (future, not in v1) |

## Data Flow (MANDATORY)

### Entry Point
- HTTP request arrives at REST listener (configured port, default 8080)

### Transformation Path
1. HTTP router matches path to handler
2. Handler extracts: command/params from URL/body, auth from Authorization header
3. Handler calls engine method (Execute, Stream, ConfigSet, etc.)
4. Engine returns Response (ze-native types)
5. Handler marshals Response to JSON
6. HTTP response sent with appropriate status code

## Design Decisions

### Convenience routes map to generic Execute, not separate engine calls

GET /api/v1/peers internally calls engine.Execute("bgp summary"). This ensures the
response is always identical to what Execute returns. No REST-specific data shaping.

### SSE for streaming, not WebSocket

Server-Sent Events are simpler (HTTP GET, no upgrade), work through proxies, and
sufficient for server-to-client event delivery. WebSocket would be needed only for
bidirectional streaming (not a current requirement).

### OpenAPI 3.1, not 3.0

OpenAPI 3.1 aligns with JSON Schema 2020-12, which the engine already generates for
command parameters from YANG metadata. No translation needed.

### No pagination in v1

Commands return their full output. If specific commands return large datasets (rib routes
with full table), pagination can be added later as query params without breaking changes.
