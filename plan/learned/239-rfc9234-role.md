# 239 — RFC 9234 BGP Role

## Objective

Implement RFC 9234 BGP Role capability: capability advertisement, OPEN validation, and role-based filtering as a plugin.

## Decisions

- Phase 1: engine-side validation in `handleOpen()` — chosen over async plugin OPEN gate (simpler, no new mechanism needed).
- Phase 2: replaced engine-side logic with `validate-open` synchronous RPC (engine→plugin via Socket B); fail-fast on first rejection.
- `Strict` field removed from `CapabilityDecl` — plugin handles strict enforcement internally via validate-open callback.
- `CodeRole` kept in `capability.go` despite plan to remove it — used for logging in `Code.String()`.

## Patterns

- `validate-open` RPC: synchronous engine→plugin callback for OPEN validation; plugin returns accept/reject.
- Plugin declares capability codes it handles; engine calls validate-open before session establishment.

## Gotchas

- validate-open is synchronous on Socket B (command socket), not the async event socket.
- Strict role enforcement must remain in the plugin, not the engine — engine is policy-free.

## Files

- `internal/component/bgp/plugins/role/` — RFC 9234 role plugin
- `internal/component/bgp/capability/` — capability code registry
